# 3강 — 처음부터 끝까지 — 도구 1개 손에 쥐기

> 외부 비개발자 청중 워크숍. **5단계 워크플로우 라이브 빌드**.
> 1·2강에서 본 단어들이 합쳐져서 작은 도구 1개가 *완성*되는 과정을 청중과 같이 본다.

**시간 배분(총 130분)**: 도입 15 / 1부 35 / 휴식 10 / 2부 30 / 3부 30 / 4부 15 / 마무리 15
**5단계**: 뼈대 → 기획 → 하니스 설계 → 자동 실행 → 리뷰
**라이브 빌드 주제**: `[강의자가 결정]` (이 script는 *주제 무관 워크플로우 골격*)
**슬라이드 마커**: `[데모: NN-xxx.html]`

> **운영 메모**: 3강은 1·2강에서 본 단어가 *다 같이 쓰이는 자리*. 청중도 자기 PC에 *빈 폴더 1개*를 미리 준비해두고, 강의자를 따라 같은 단계를 자기 폴더에서도 시도. 단, 청중이 *완전히 따라하기 어려운 단계*(예: claude -p 헤드리스 시연)는 강의자만 시연하고 청중은 결과를 *읽으면서* 본다.

## 0. 도입 — 오늘 만들 도구 + 5단계 흐름 (15분)

[데모: 01-cover.html]

여러분, 마지막 강의입니다. 1강에서 *AI의 한계*를 봤고, 2강에서 *그 한계를 보완하는 도구 5개*를 봤어요. 오늘은 그걸 **다 합쳐서** *작은 도구 1개*를 만듭니다.

오늘 만들 도구는 — *[강의자가 결정한 빌드 주제 소개]*. 끝까지 가면 *여러분 부서에서 바로 쓸 수 있는 수준*까지 갑니다.

---

### 5단계 워크플로우 한 장

도구를 만드는 표준 흐름입니다. 외워두세요.

| 단계 | 누가 | 무엇 | 1·2강 연결 |
|---|---|---|---|
| **1. 뼈대** | 사용자 + AI | 빈 폴더 → 기본 구조 | 1강 라이브 라운드 |
| **2. 기획** | 사용자 + AI | PRD·ARCHITECTURE·ADR·UI_GUIDE | 2강 컨텍스트 |
| **3. 하니스 설계** | 사용자 + AI | CLAUDE.md + Skills + Hooks | 2강 도구 5개 |
| **4. 자동 실행** | AI 혼자 (`claude -p` 헤드리스) | 위 1~3을 읽고 끝까지 진행 | 2강 자동화 |
| **5. 리뷰** | 사용자 + AI | 결과 검수 + 환각·시코펀시 잡아내기 | 1강 한계 단어 |

핵심 원칙: **사람은 절대 생각을 멈추지 않는다.** AI가 자동화한다고 해서 *판단을 위임하지 않습니다*. 1·3·5단계는 사람이 주도, 4단계는 AI가 주도, 2단계는 같이.

### 단계 1 — 뼈대 (10분)

빈 폴더에서 출발합니다. *Claude Code 진입* → *뭘 만들지 의도 대화*.

```text
[강의자 시연]
빈 폴더에서:
$ claude

> 나는 [빌드 주제]를 만들고 싶어.
  핵심 기능 3개: [A], [B], [C].
  사용자는 [대상 사용자]야.

  먼저 폴더 구조부터 같이 잡자.
  뭐부터 만들까?
```

Claude가 *기본 폴더 구조*를 제안합니다. *"src/, docs/, tests/, …"* 식으로요. 강의자는 *질문을 주고받으면서* 어떤 구조로 갈지 합의합니다.

**청중 활동**: 자기 PC 빈 폴더에서도 같은 단계 따라하기. 빌드 주제는 *각자 정해도 됨* (예: 가계부, 회의록 요약기, 명함 OCR).

---

### 단계 2 — 기획: 4종 문서 (25분)

뼈대가 잡혔으면 *기획 문서 4종*을 만듭니다. 이게 2강에서 짚은 *컨텍스트 엔지니어링*의 실전 형태예요. 4종 = **PRD · ARCHITECTURE · ADR · UI_GUIDE**.

| 문서 | 한 줄 정리 | 일상 비유 |
|---|---|---|
| **PRD** (Product Requirements Document) | *무엇을 만들지* — 기능·사용자·시나리오 | 영화 시나리오 |
| **ARCHITECTURE** | *어떻게 만들지* — 폴더·모듈·데이터 흐름 | 건축 설계도 |
| **ADR** (Architecture Decision Record) | *왜 이렇게 정했는지* — 결정 1건씩 기록 | 회의록 |
| **UI_GUIDE** | *어떻게 보일지* — 색·타이포·간격 | 의상 코디북 |

```text
[강의자 시연]
> docs/PRD.md 초안 만들어줘.
  대상 사용자, 핵심 기능 3개, 성공 기준 3개 포함해서.

> docs/ARCHITECTURE.md 만들어줘.
  PRD를 바탕으로 폴더 구조와 데이터 흐름.

> docs/ADR/0001-기술스택-선정.md 만들어줘.
  왜 이 스택을 골랐는지 결정 기록 형식으로.

> docs/UI_GUIDE.md 만들어줘.
  다크 테마, 미니멀, 한국어 우선.
```

각 문서가 만들어지면 *강의자가 한 번씩 읽고 손봅니다*. 2강에서 본 *시코펀시* 떠올리세요 — *"이거 틀린 부분 없나?"* 한 번씩 던집니다.

**왜 4종 문서인가**: 단계 4 *자동 실행*에서 AI 혼자 일할 때, *이 4종 문서가 컨텍스트의 핵심*이 됩니다. 잘 깔리면 AI가 헤매지 않아요.

### CLAUDE.md 작성 (15분)

```text
[강의자 시연]
> /init

(자동으로 CLAUDE.md 초안 생성)
```

초안에 *프로젝트 헌법*을 추가합니다. **Karpathy 4원칙**을 한국어 강의톤으로 옮긴 형태가 표준입니다.

```markdown
# [프로젝트 이름]

## CRITICAL — 반드시 지킬 것
- TDD: 모든 새 기능은 테스트 먼저 작성
- 응답은 한국어로
- 환각 방지: 본 적 없는 라이브러리·함수는 *반드시 검색·확인*

## Karpathy 4원칙
1. **Think** — 코드 짜기 전 의도부터 적기
2. **Simple** — 추상화 금지, 직선적으로
3. **Surgical** — 요청한 것만, 옆 코드 건드리지 말기
4. **Goal-Driven** — PRD 기준으로 항상 자기 검열

## 코드 스타일
- 함수 한 개 = 한 가지 일
- 주석은 *왜*만 적기 (*무엇*은 코드가 말함)
```

**청중 활동**: 자기 폴더에도 같은 CLAUDE.md를 깐다.

### Skills + Hooks (15분)

```text
[강의자 시연]
> /harness

(이 슬래시 명령은 강의 자료에 포함된 커스텀 skill.
 docs/PRD·ARCHITECTURE·ADR·UI_GUIDE를 읽고
 표준 하니스 골격(테스트·린트·빌드 hook)을 자동 설치)
```

설치되는 것:
- **Hook 1**: 모든 응답 후 자동 테스트 실행
- **Hook 2**: 한국어 출력 보정 (caveman-kor 스타일)
- **Hook 3**: 커밋 전 ADR 갱신 알림

**왜 자동화하나**: 1강에서 *비결정성* 봤죠. 사람이 매번 *"테스트 돌려봐, 한국어 맞춰, ADR 적어"* 시키면 빠뜨립니다. **결정론적 기계가 빠뜨리지 않습니다**.

### claude -p 헤드리스 모드

`claude -p` = Claude Code의 *헤드리스(headless) 모드*. 사람이 옆에 없어도 *Phase 1 → Phase 5*를 *각각 새 Claude 세션으로* 흘리는 기능.

**왜 새 세션인가** (1강 *Lost in the Middle* 떠올리기):
한 세션에서 길게 일하면 컨텍스트가 쌓여서 *가운데를 무시*하는 현상이 생깁니다. 그래서 *각 Phase마다 새 Claude*가 *깨끗한 메모리*로 들어가요. 그 새 Claude는 *문서로 컨텍스트를 채웁니다* — 우리가 단계 2에서 깐 *PRD·ARCHITECTURE·ADR·UI_GUIDE* + 단계 3에서 깐 *CLAUDE.md*가 그 자료입니다.

**시연**:

```bash
[강의자 시연 — 청중은 화면을 본다]
$ claude -p "프로젝트 PRD를 읽고
  Phase 1~5를 순서대로 진행해.
  각 Phase 종료 시 진행 docs를 반드시 갱신해."

[Phase 1 — 뼈대 코드 생성]
[Phase 2 — 핵심 모듈 구현]
[Phase 3 — 테스트 작성]
[Phase 4 — UI 적용]
[Phase 5 — 통합 검증]
```

화면이 *흘러갑니다*. 강의자는 화면을 보면서 *"지금 AI가 무슨 docs를 읽고 있는지", "왜 저 결정을 내렸는지"* 청중에게 짚어줍니다.

**예상 결과**: 17~25분 정도에 끝남 (rhwp 사례 기준 5/5 Phase, 54 테스트 통과 케이스).

**라이브 위험관리**: 헤드리스 진행 중 *기다리는 시간*이 길어질 수 있음. 강의자는 *왜 이렇게 자동화가 되는지* — 1·2강 단어를 다시 짚으면서 청중 흥미 유지.

- *"지금 새 Claude가 들어왔어요. 1·2강에서 본 *지식 컷오프* 떠올리세요. 새 Claude는 우리 프로젝트를 *모릅니다*. 그래서 docs를 읽는 거예요."*
- *"Phase 3 테스트 단계입니다. *비결정성* 막으려고 모든 결과를 자동 테스트로 잡아내요."*

## 4부 — 단계 5: 리뷰 (15분)

[데모: 10-part4-cover.html]

자동 실행이 끝났습니다. *AI 혼자 만든 결과물*이 폴더에 가득 차 있어요. **그대로 믿으면 안 됩니다.**

```text
[강의자 시연]
> /review

(전체 결과를 사람 검수 모드로 훑음)
```

**리뷰 체크리스트** — 1강 단어로 정리:

| 1강 단어 | 무엇을 확인하나 |
|---|---|
| **환각** | AI가 *없는 라이브러리·함수*를 import하지 않았나 |
| **시코펀시** | *"좋아요 다 됐어요"* 라고 자신만만하게 답한 부분이 *진짜* 다 됐나 |
| **비결정성** | 같은 입력에 *같은 출력*이 나오나 (테스트 2~3회 반복) |
| **지식 컷오프** | 옛 라이브러리 버전을 쓰지 않았나 (보안 문제 가능) |

**문제 발견 시 — 사람의 역할**:
1. *질문을 거꾸로 던진다* — *"이 부분 틀린 거 같은데 다시 봐줘"* (시코펀시 해소)
2. *컨텍스트를 더 준다* — 관련 docs 1장 추가 첨부
3. *부분만 다시 시킨다* — 전체 재실행 X, *문제 부분만* 부분 재생성

**일상 비유**: *AI는 야간 근무 중인 신입사원*. 아침에 출근한 선임이 *밤새 한 일을 다 검수*해야 합니다. 그게 사람의 일.

### 오늘 본 것 정리

5단계가 한 사이클입니다.

```text
[1. 뼈대] → [2. 기획] → [3. 하니스] → [4. 자동 실행] → [5. 리뷰]
   사람        같이         같이           AI               사람
```

이 한 사이클이 *작은 도구 1개*를 만듭니다. 큰 프로젝트는 이 사이클을 *여러 번* 돕니다 — 그게 *하이퍼-워터폴* 방법론입니다 (rhwp 50일).

---

### AI Adoption Spectrum — 자기 진단

마지막으로 *여러분이 지금 어디 있는지* 가늠하는 4단계입니다.

| 단계 | 한 줄 설명 | 여러분은 어디? |
|---|---|---|
| **L1 — AI-aware** | 존재만 안다. 가끔 ChatGPT 시도 | 강의 시작 전의 다수 |
| **L2 — AI-enabled** | 기존 업무에 AI를 *얹어* 효율화 | 2026년 *대다수 기업* 이 머무는 단계 |
| **L3 — AI-maximalist** | "AI 어디에나" 신념 단계 — *번아웃 위험* | 신중히 가야 |
| **L4 — AI-native** | AI 중심으로 *균형·설계* 들어간 단계 | 목표 |

**핵심 메시지**: maximalist > native가 아닙니다. *"AI에 다 맡기자"* 는 신념 단계를 거쳐 *"사람이 어디서 판단하고 어디서 위임할지 설계"* 하는 native가 더 성숙한 단계예요.

여러분이 오늘 *5단계 워크플로우*를 본 게 정확히 native 단계의 *훈련*입니다.
