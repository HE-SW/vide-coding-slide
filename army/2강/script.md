# 2강 — 같은 일을 잘 시키는 법

> 외부 비개발자 청중 워크숍. **3겹 동심원 + 도구 5개 + 회사 도입 사례**.
> 1강에서 본 "나쁜 결과"를 "좋은 결과"로 바꾼다. 끝나면 *자기 업무용 CLAUDE.md 1장*을 손에 들고 간다.

**핵심**: 프롬프트 · 컨텍스트 · 하니스 (3겹 동심원) + CLAUDE.md · Skills · Hooks · MCP · 서브에이전트 (도구 5개)
**함께 입력 마커**: `[함께 입력 #N]`

## 0. 도입 — Before/After 충격 (15분)

[데모: 01-cover.html]

여러분, 다시 만났습니다. 1강 끝나고 *불만 있었던 분 있죠*?

- *"왜 매번 다른 결과가 나오지"*
- *"왜 진짜 같은데 다 가짜지"*
- *"이걸 우리 부서에 도입하라고? 너무 들쭉날쭉하잖아"*

그 불만이 오늘 강의의 *입구*입니다. 같은 prompt를 *훨씬 잘 시키는 법*을 봅니다.

### 1강 Pomodoro vs 오늘 Pomodoro

자, 1강에서 다 같이 만든 Pomodoro 타이머 기억나시죠. 결과 다 달랐어요. 어떤 사람은 색깔도 이상했고, 어떤 사람은 동작 자체가 안 됐고요.

오늘 같은 prompt를 다시 써볼 거예요. **단, 이번엔 폴더 안에 *몇 가지 장치*가 깔려 있습니다.** 같이 입력해 봅시다.

```text
[함께 입력 #1 — Before]
빈 폴더에서:
> Pomodoro 타이머 만들어줘. 다크 테마, 미니멀.
```

(**5분**) 결과 보세요. 1강과 비슷하게 *제각각*이죠.

```text
[함께 입력 #1 — After]
같은 prompt를, 단 CLAUDE.md + UI_GUIDE.md가 깔린 폴더에서:
> Pomodoro 타이머 만들어줘. 다크 테마, 미니멀.

(CLAUDE.md / UI_GUIDE.md는 강의 자료로 미리 배포)
```

(**5분**) 자, 이번엔요? **거의 다 비슷한 결과**가 나왔을 겁니다. 색깔도, 폰트도, 폴더 구조도. 같은 AI에 같은 prompt를 넣었는데 *결과가 모이는* 이유 — 오늘 강의의 핵심입니다.

### 1겹 — 프롬프트 엔지니어링

**프롬프트(prompt)** = AI에게 *지금 이 순간* 던지는 한 줄. 같은 AI에 같은 한 줄을 넣어도, *어떻게 적느냐*에 따라 결과 품질이 달라집니다.

같이 비교해 봅시다.

```text
[함께 입력 #2 — Zero-shot]
> 우리 회사 슬로건 5개 만들어줘.
  업종은 의료 AI 솔루션이야.
```

```text
[함께 입력 #3 — Few-shot]
> 우리 회사 슬로건 5개 만들어줘.
  업종은 의료 AI 솔루션이야.

  좋은 슬로건 예시 (다른 회사):
  - Nike: "Just Do It"
  - Apple: "Think Different"
  - Tesla: "Accelerate the World's Transition to Sustainable Energy"

  위 예시처럼 짧고 강한 동사형 슬로건으로 부탁해.
```

(**결과 비교**) Few-shot 쪽이 *훨씬* 잘 나오죠. 이게 *프롬프트 엔지니어링*입니다. 핵심 패턴 5가지만 알아 두면 충분해요.

| 패턴 | 한 줄 정리 | 일상 비유 |
|---|---|---|
| **Zero-shot** | 그냥 시킴 | "알아서 해" |
| **Few-shot** | 예시 2~5개 같이 줌 | "이렇게 만든 것처럼" |
| **Chain-of-Thought** | *"단계별로 생각하고 답해"* 한 줄 추가 | "풀이 과정 보여주고 답해" |
| **Role Prompting** | *"너는 시니어 카피라이터야"* 역할 부여 | "전문가 모자 쓰고 답해" |
| **Self-Consistency** | 같은 질문 여러 번 → 다수결 | "여러 의견 듣고 종합" |

**Chain-of-Thought 시연**:

```text
[함께 입력 #4]
> 우리 부서 9명이 회식 가는데
  3명은 해산물 못 먹고, 2명은 술 안 마시고, 1명은 비건이야.
  강남역 근처 추천 식당 3곳. 단계별로 생각하고 답해.
```

*"단계별로 생각하고 답해"* 한 줄이 답 품질을 *눈에 띄게* 끌어올립니다. 일상 비유로 — *수학 시험에서 풀이 과정을 쓰는* 것과 같아요. 답만 적으면 틀리기 쉬운데, 과정을 쓰면 정확해집니다.

[데모: 05-prompt-engineering.html]

### 2겹 — 컨텍스트 엔지니어링

프롬프트가 *지금 던지는 한 줄*이라면, **컨텍스트**는 그 한 줄과 *함께 주는 모든 자료*입니다.

**일상 비유 — 시험장에 책 들고 가기**:
- 기억력만으로 푸는 시험 → 환각 + 부정확
- 교과서·노트·계산기 들고 가는 시험 → 정확
- 컨텍스트 엔지니어링 = *AI에게 시험 자료를 넉넉히 챙겨주는 일*

같이 비교해 봅시다.

```text
[함께 입력 #5 — 컨텍스트 없음]
> 우리 회사 환불 정책에 따라
  이 고객 메일에 답장 써줘.

  [고객 메일 내용 붙여넣기]
```

```text
[함께 입력 #6 — 컨텍스트 첨부]
> 우리 회사 환불 정책은 다음과 같아:
  [환불 정책 PDF/문서 1장 첨부 또는 본문 붙여넣기]

  이 정책에 따라 아래 고객 메일에 답장 써줘.

  [고객 메일 내용]
```

(**결과 비교**) 첨부한 쪽은 *정책에 정확히 맞는 답장*이 나옵니다. 안 한 쪽은 *환각*해서 *없는 정책*을 만들어 답할 수도 있어요.

**컨텍스트의 4가지 층**:
1. **System Context** — 역할·규칙·톤 (*"너는 정중한 고객지원 담당자야"*)
2. **Conversation Context** — 사용자 입력 + 대화 + 이전 기억
3. **Knowledge Context** — RAG·외부 문서·웹 검색 (← 위에서 PDF 첨부한 것이 여기)
4. **Tool & Environment Context** — 도구 결과·OS·디렉토리

**RAG (Retrieval-Augmented Generation)** = *Knowledge Context*의 표준 구현. 한국어로 풀면 *"검색해서 찾아온 자료를 함께 주는 생성"*. 사내 챗봇이 *우리 회사 위키만 보고 답하는* 게 RAG입니다.

**실제 사례 — 사내 챗봇 도입 시 RAG 효과**:
일반 ChatGPT에 *"우리 회사 휴가 규정 알려줘"* 물으면 → 환각.
사내 위키를 RAG로 연결하면 → *우리 회사 진짜 규정*이 나옵니다.

[데모: 06-context-engineering.html]

### 3겹 — 하니스 엔지니어링

가장 바깥 동심원입니다. **하니스(harness)** = 영어로 *마구*. 마차에 말을 묶는 도구. 말의 힘은 그대로 두되, *방향과 속도를 통제*하는 장치.

**왜 필요하나**: 1강에서 본 *비결정성·환각·시코펀시*는 prompt만으론 절대 못 막아요. *결과를 검증·교정하는 자동 절차*가 별도로 필요합니다. 그게 하니스입니다.

**1층 + 2층 구조**:

| 층 | 누가 깔았나 | 무엇 |
|---|---|---|
| **1층 — 내장 하니스** | Claude Code (Anthropic) | 시스템 프롬프트, 권한 체크, 도구 경계, JSON 스키마 |
| **2층 — 내가 까는 하니스** | 강의자(여러분) | CLAUDE.md, docs/, Skills, Hooks |

여러분이 1강부터 쓰고 있던 Claude Code는 *이미 1층이 깔려 있어서* 그 정도라도 동작했던 겁니다. 오늘 2부에서 *2층 도구 5개*를 라이브로 깔아 봅니다.

### 도구 1 — CLAUDE.md (프로젝트 헌법)

**CLAUDE.md**는 Claude Code가 폴더에 들어올 때 *자동으로 읽는* 파일입니다. *"이 프로젝트는 이런 규칙으로 일해라"* 라고 적어두는 *프로젝트 헌법*이에요.

같이 만들어 봅시다.

```text
[함께 입력 #7 — CLAUDE.md 생성]
빈 폴더에서:
> /init

(Claude Code의 기본 슬래시 명령. 폴더 내용을 읽고 CLAUDE.md 초안 자동 생성)
```

이제 그 CLAUDE.md 파일에 *여러분 업무 규칙*을 한두 줄 적어 보세요. 예를 들어:

```markdown
# 우리 부서 작업 규칙

## CRITICAL (반드시 지킬 것)
- 모든 응답은 한국어로
- 외부 고객에게 보내는 메일은 정중체("드립니다", "되시겠습니다")
- 회의록 요약은 Action Item을 가장 위에

## 톤
- 실무자 대상은 간결하게, 임원 대상은 결론부터
```

같은 폴더에서 다시 prompt를 던져 보세요.

```text
[함께 입력 #8]
> 다음 회의록을 우리 규칙대로 요약해줘.

  [회의록 텍스트]
```

(**결과 확인**) Action Item이 위에 올라와 있죠. 정중체로 변환돼 있고요. **CLAUDE.md 안 깐 폴더에서 같은 걸 시키면 매번 다른 결과**가 나옵니다.

**일상 비유**: *신입사원에게 매번 회사 규칙을 다시 설명하는 대신, 입사 첫날 한 장짜리 행동지침을 주는 것*. 한 번 받으면 매번 그걸 보고 일합니다.

[데모: 10-claude-md.html]

### 도구 2 — Skills (`/슬래시 명령어`)

**Skill**은 자주 쓰는 절차를 *한 줄짜리 슬래시 명령*으로 묶어둔 것입니다. 매번 긴 prompt를 다시 적는 대신, `/이름` 한 줄로 호출해요.

같이 시연해 봅시다.

```text
[함께 입력 #9]
> /init
> /review
```

`/init`은 *프로젝트 초기화*, `/review`는 *코드 리뷰*. 둘 다 *Claude Code 기본 내장* 스킬이에요. 직접 만든 스킬도 같은 방식으로 작동합니다.

**Skill의 정체**: 사실 *프롬프트 한 덩어리를 미리 적어둔 파일* + 자동 호출 규약(YAML frontmatter). 그 폴더 구조는 `~/.claude/skills/이름/SKILL.md`.

**실제 사례 — Andrej Karpathy의 65줄 스킬**:
Karpathy가 자기 LLM 사용에서 *4대 실패 패턴*을 65줄짜리 Skill 1개로 압축한 사례가 있어요 (오픈소스). LLM이 같은 실수를 반복하지 않게, 매 prompt마다 그 65줄이 자동 주입됩니다.

[데모: 11-skills.html]

### 도구 3 — Hooks (자동 후처리)

**Hook**은 AI가 답을 *내놓기 전·후*에 자동으로 실행되는 *후처리 스크립트*입니다. AI 답에 *기계적인 보정*을 더하는 장치예요.

**유명한 사례 — `caveman-kor` Hook**:
한국어 답변에 *영어 원문이 너무 많이 섞여 나오는 문제*를 한국어 권장으로 강제 변환하는 hook. 이걸 적용한 팀에서 *출력 토큰 30~50% 절감* 효과를 봤다는 보고가 있습니다 (오픈소스 사례).

같이 시연해 봅시다.

```text
[함께 입력 #10 — Hook 끄기]
설정에서 hooks 비활성:
> 다음 영어 문서를 한국어로 요약해줘.
  [영어 문서 1장]
```

```text
[함께 입력 #10 — Hook 켜기]
같은 prompt를 hook 활성 상태에서:
```

(**결과 비교**) Hook을 켠 쪽이 *영어 원문이 덜 섞이고, 자연스러운 한국어*로 나옵니다. 토큰도 덜 씁니다.

**일상 비유**: *맞춤법 검사기*. 글을 다 쓴 다음 자동으로 한 번 훑고 보정.

[데모: 12-hooks.html]

### 도구 4 — MCP (Model Context Protocol)

**MCP**는 한국어로 *모델 컨텍스트 프로토콜*. AI에게 *외부 도구를 연결하는 표준 규약*입니다.

**일상 비유**: *AI 도구계의 USB-C*. 옛날엔 노트북마다, 휴대폰마다 충전 단자가 다 달랐죠. USB-C가 표준이 되니까 *어떤 기기든 같은 케이블*로 충전됩니다. MCP도 마찬가지 — *어떤 도구든 같은 규약*으로 AI에 꽂으면 작동.

**실제 사례**:
- **Figma MCP** — Claude가 Figma 디자인을 직접 읽고, 디자인 코드로 변환
- **GitHub MCP** — Claude가 PR 만들고, 이슈 생성
- **Google Drive MCP** — Claude가 Drive 파일을 직접 검색·요약
- **2025-12 Linux Foundation 기증** — MCP가 *오픈 표준*으로 정식 등록

같이 시연해 봅시다.

```text
[함께 입력 #11]
(Figma MCP 연결된 상태에서)
> 이 Figma 디자인 (URL 첨부) 을 React 코드로 만들어줘.
```

(**결과 확인**) Claude가 *Figma 파일을 직접 열어 읽고*, 디자인 토큰·컴포넌트 매핑을 따져서 코드를 짭니다. *디자이너 → 개발자* 사이의 다리를 비개발자가 직접 건널 수 있는 모먼트예요.

[데모: 13-mcp.html]

### 도구 5 — 서브에이전트 (Subagent)

**서브에이전트**는 큰 작업을 *전문 분야 작은 AI 동료*에게 위임하는 도구. *사장이 부장에게 부장이 과장에게* 일을 나누듯, 큰 prompt를 작은 prompt 여러 개로 *분업*시킵니다.

**왜 필요하나**: 1강에서 *컨텍스트가 길어지면 가운데를 못 본다(Lost in the Middle)* 같은 현상이 있다고 짚었죠. 큰 작업을 한 AI에 다 맡기면 *맥락이 흐려집니다*. 서브에이전트로 작게 쪼개면 각자 *작은 맥락*에서 일하니까 정확도가 올라갑니다.

같이 시연해 봅시다.

```text
[함께 입력 #12]
> 우리 회사 1년치 보고서 50개를 다음 3가지 관점에서 분석해줘:
  1. 매출 트렌드
  2. 고객 만족도 변화
  3. 신규 사업 진척도

  각 관점은 별도 서브에이전트에게 위임해서 병렬로 진행해.
```

Claude Code가 자동으로 *3개 서브에이전트*를 띄워서 동시에 분석하고, *최종 결과*를 합쳐 냅니다. 한 AI가 끙끙대며 50개를 순차로 읽는 것보다 *훨씬 빠르고 정확*해요.

**실제 사례 — Anthropic 자사 워크플로우**:
Anthropic 내부에서 코드 리뷰를 *3개 서브에이전트*로 병렬 진행. 보안·성능·가독성 각자 따로 보고, 최종 합본. *2~3배 시간 단축* 보고.
