# 블록 콘텐츠 → 레이아웃 추천

각 슬라이드 블록의 raw 콘텐츠 시그니처에서 `presentation_slides`의 8가지 레이아웃 중 최적을 추천하는 규칙. 모호하면 최종 결정은 사용자 확인을 받음.

---

## 추천 알고리즘 (우선순위 순)

블록을 위에서부터 아래 패턴과 매칭하고, **가장 먼저 매칭되는 패턴**의 레이아웃을 추천한다.

### 1. summary-grid — 마무리 / 총정리

**시그니처:**
- 블록 위치가 마지막 또는 마지막에서 두 번째
- 키워드: "정리", "마무리", "총정리", "오늘 살펴본", "결론", "요약"
- 6개 내외 항목 + 결론 한 줄

**예시 입력:**
```md
## 마무리

오늘 살펴본 6가지 핵심:
- 시스템 프롬프트
- 안전장치
- 코드 생성
- 리뷰 흐름
- 협업 패턴
- 다음 영상 예고

한마디로: Claude Code는 도구가 아니라 동료다.
```
→ `summary-grid`

---

### 2. roadmap — 개요 / 진행률

**시그니처:**
- 블록 위치가 첫 번째 또는 두 번째 (인트로 직후)
- 키워드: "오늘 다룰", "이번 영상에서", "전체 시리즈", "N편 중 X편"
- 3~6개 짧은 라벨

**예시 입력:**
```md
## 시리즈 개요
- 1강: 시스템 프롬프트
- 2강: 코드 생성
- 3강: 리뷰
- 4강: 협업
오늘은 1강.
```
→ `roadmap`

---

### 3. comparison-2col — 대조 / before-after

**시그니처:**
- "기존" / "예전" / "옛날" / "before" + "이제" / "지금" / "after" 같은 시간 대조
- "X 대신 Y", "X vs Y"
- `→` 또는 화살표 기호
- 1~3개 비교 행

**예시 입력:**
```md
## 변화

기존엔 직접 PR 코멘트 달았는데, 이제는 claude review 명령어 하나로 끝.
기존: 수동 git 명령 → 이제: 자연어 지시.
```
→ `comparison-2col`

---

### 4. step-flow — 순차 단계

**시그니처:**
- 번호 매긴 리스트 (1, 2, 3, ...) — 3~5개
- 또는 "먼저", "그다음", "마지막으로" 같은 시간 부사
- 각 단계가 제목 + 짧은 설명 형태

**예시 입력:**
```md
## 설치 과정
1. 노드 설치
2. claude 글로벌 설치
3. 인증
4. 워크스페이스 등록
```
→ `step-flow`

---

### 5. three-stage-flow — 정확히 3단계 발전

**시그니처:**
- 정확히 3개 단계 (4 이상이면 step-flow)
- "1단계 → 2단계 → 3단계" 또는 "초급 → 중급 → 고급" 같은 발전/레벨

**예시 입력:**
```md
## 사용자 레벨
- 입문: 코드 자동완성
- 활용: 자연어 → 코드
- 마스터: 멀티 에이전트 코워크
```
→ `three-stage-flow`

---

### 6. diagram-box — 중심 + 분기 구조

**시그니처:**
- "중심에는", "핵심은 X이고, 여기서 Y, Z, W로"
- 1개 핵심 개념 + 3~6개 하위 노드
- 시스템 구조 / 아키텍처 / 보안 모델

**예시 입력:**
```md
## 하네스 구조
중심에는 Claude Code 하네스가 있고, 거기서 갈라지는 것들:
- Tool 권한 시스템
- 시스템 프롬프트
- 컨텍스트 압축
- Skill 시스템
```
→ `diagram-box`

---

### 7. grid-2x2 — 기능 / 도구 나열

**시그니처:**
- 4~6개 항목 나열, 각 항목이 이모지 + 짧은 제목 + 1~2줄 설명
- 동급 항목들 (계층 없음, 순서 의미 없음)
- "이런 기능들이", "지원하는 도구"

**예시 입력:**
```md
## 주요 기능
- 코드 생성
- 코드 리뷰
- 테스트 실행
- 깃 워크플로우
```
→ `grid-2x2`

---

### 8. hero-cards — 임팩트 / 핵심 강조 (default)

**시그니처:**
- 위 패턴 어디에도 강하게 매칭되지 않을 때 default
- 코드 블록 1~3개 + 임팩트 멘트
- 2~3개 강한 메시지를 카드로 보여주고 싶을 때
- 인트로 훅(hook)에 자주 사용

**예시 입력:**
```md
## 인트로

```system prompt
Never run force push to main/master
Never use git commands with -i flag
Always create NEW commits rather than amending
```

여러분이 Claude Code를 실행하는 순간, 이 하네스가 자동으로 작동하고 있었습니다.
```
→ `hero-cards` (3개 룰을 카드로, 마지막 멘트는 banner로)

## 콘텐츠 보존 규칙

레이아웃이 결정되면 다음을 그대로 보존하여 `presentation_slides`로 넘김:

### 코드 블록
- fenced code(` ``` `)는 라벨/언어 힌트 포함하여 그대로 유지
- 라벨이 `system prompt`, `bash`, `js` 등이면 슬라이드에서 monospace 카드(`.card-code`)로 렌더링됨
- 인라인 백틱 코드는 슬라이드의 `<code>` 또는 코드 카드로 렌더링됨

### 강조 키워드
- `**굵게**`는 슬라이드의 `.accent`, `<strong>` 으로 렌더링됨
- `Never`, `Always`, `절대로` 같은 부정/단정 강조어는 `.kw-never`/`.kw-always` 컬러 토큰이 될 가능성이 높으므로 그대로 살림

### 이모지
- 사용자가 메모에 이모지를 적었으면 보존 (각 카드/단계의 아이콘으로 사용됨)
- 메모에 없으면 강제 추가하지 않음 (presentation_slides가 알아서 보강)

### 라벨 / 태그
- "라벨:", "태그:", "TYPE:" 같은 prefix가 있으면 카드 태그(`.card-tag`)로 변환되도록 유지
- 예: "SYSTEM PROMPT" 라벨이 있는 코드 → hero-cards의 카드 태그로 그대로

---

## 모호한 경우의 처리

1. **두 패턴 모두 가능** — 사용자 1차 확인 표에 둘 다 표시:
   ```
   추천 레이아웃: step-flow (또는 grid-2x2 가능)
   ```
2. **항목 수가 경계** — 4개 항목: step-flow와 grid-2x2 둘 다 가능. 항목에 순서가 있으면 step-flow, 동급이면 grid-2x2.
3. **콘텐츠가 너무 적음** — 1~2 문장만 있고 항목 없음 → hero-cards 단일 카드 또는 배너 위주
4. **콘텐츠가 너무 많음** — 7개 이상 항목 → 두 슬라이드로 분할 제안 (사용자 확인)

---

## 동일 레이아웃 연속 방지

`presentation_slides`의 K 섹션 규칙: 같은 레이아웃이 3번 이상 연속되면 시각적으로 단조로움.

이 스킬에서 레이아웃을 추천할 때:
- 직전 두 슬라이드가 같은 레이아웃이면 차선책으로 변경 제안
- 예: 슬라이드 3, 4가 step-flow → 슬라이드 5는 grid-2x2 또는 diagram-box 등으로
- 변경 사유를 사용자에게 알려줌: "(연속 방지를 위해 step-flow 대신 grid-2x2 추천)"
