## 02 — `02-pomodoro-live.html` · 바이브 코딩이란?

먼저 정의부터 짚고 갈게요. **바이브 코딩** 은 — AI와 자연어를 활용해, 사람이 직접 모든 코드를 작성하지 않아도 원하는 기능을 구현해 나가는 *새로운 개발 방식* 입니다.

핵심은 한 줄이에요. **코드를 손으로 직접 안 쓴다.** 대신 AI에게 *원하는 것* 을 자연어로 전달합니다.

## 03 — `03-part1-cover.html` · 바이브 코딩을 처음 말한 사람

## 04 — `04-vibe-definition.html` · 왜 '바이브 코딩'일까?

이름이 왜 *바이브 코딩* 일까요. *분위기* 라는 영어 단어 *Vibe* 에서 왔어요.

기존 개발 방식은 — **문법을 정확히 기억** 해야 하고, **함수와 구조를 직접 설계** 해야 하고, **작은 오타 하나** 도 사람이 잡아내야 했습니다.

바이브 코딩은 다릅니다. *"정확한 코드"* 보다 — *"내가 원하는 느낌"* 을 AI에게 전달하는 방식이에요. 예를 들면 — *"애플 느낌의 UI로 만들어줘"*, *"조금 더 부드러운 애니메이션 넣어줘"*, *"사용자가 직관적으로 느끼게 바꿔줘"*.

코드를 *작성* 하는 행위가 아니라 — AI를 *지휘* 하는 감각. 그래서 *Vibe Coding* 이라는 이름이 붙은 거예요.

## 05 — `05-vibe-1year.html` · 기존에는 어떤 문제가 있었나

그럼 바이브 코딩이 등장하기 전엔 — 어땠을까요. 세 가지 큰 문제가 있었습니다.

**첫째, 시작까지 오래 걸렸어요.** 언어 문법 배우고, 개발 환경 세팅하고, 프레임워크 이해하고, 에러 해결하는 법까지. *무언가 만들기 전에 배워야 할 것이 너무 많았어요.*

**둘째, 아이디어보다 구현 난이도가 더 컸어요.** 코드를 모르면 구현 자체가 불가능했고, 작은 기능도 개발자 도움이 필요했죠. *비개발자는 시도 자체가 어려웠습니다.* 좋은 아이디어가 있어도 벽 앞에서 멈췄어요.

**셋째, 반복 작업이 너무 많았어요.** CRUD 작성, API 연결, 스타일 수정, 보일러플레이트 코드. *개발자조차 반복 작업에 시간을 다 썼습니다.*

이 세 가지를 — 바이브 코딩이 풀어주기 시작했어요.

## 06 — `06-vibe-analogy.html` · 무엇을 해결했나

바이브 코딩이 어떻게 해결했냐, 세 가지로 정리하면.

**하나, 아이디어에서 결과물까지 가는 속도** 가 완전히 달라졌어요. 예전엔 몇 주 걸리던 프로토타입이 — 지금은 몇 분에서 몇 시간 안에 완성됩니다. 압도적으로 빨라졌죠.

**둘, 비개발자도 만듭니다.** 머릿속에 떠오른 아이디어를 *코드를 몰라도* 그대로 만들 수 있어요. 기획자, 디자이너, 사업 전략팀, 연구원까지. 이전엔 못 했던 사람들이 지금은 자기 손으로 만듭니다.

**셋, 개발자 역할이 바뀝니다.** 사라지는 게 아니라 — *역할 변화* 예요. 이전엔 *직접 구현하는 사람* 이었다면, 지금은 *AI를 지휘하고, 구조를 설계하고, 결과를 검증하는 사람* 으로 옮겨가고 있어요.

## 07 — `07-vibe-before-after.html` · 그럼 만능일까?

그럼 바이브 코딩이 만능일까요. **물론 아닙니다.** 분명한 한계와 문제점이 있어요.

**첫째, AI가 항상 정답은 아닙니다.** 겉보기엔 잘 돌아가는 코드여도, 내부 구조가 잘못됐거나, 보안 문제가 숨어 있거나, 유지보수가 어려운 경우가 많습니다. *"돌아가는 코드"* 와 *"운영 가능한 코드"* 는 다른 이야기예요.

**둘째, 프로젝트가 커질수록 AI도 길을 잃습니다.** 초반엔 빠르지만, 기능이 많아지면 — *이전 구조를 잊고*, *다른 기능을 망가뜨리고*, *코드 스타일이 계속 바뀝니다.* 작은 프로젝트엔 강한데 큰 프로젝트엔 관리 능력이 약해요.

**셋째, *"AI가 만들었으니 맞겠지"* 라는 착각.** AI는 매우 자신감 있게 답합니다. 잘못된 코드도 그럴듯해 보여요. 그래서 우리에겐 — *결과를 검증하는 능력*, *문제를 의심하는 능력*, *다시 수정시키는 능력* 이 필요합니다.

## 10 — `10-llm-definition.html` · 환각 (Hallucination)

첫 번째 — **환각**, 영어로 *Hallucination* 입니다.

AI는 *"모른다"* 고 답하지 못해요. 대신 — *틀린 정보를 그럴듯하게 말하고, 존재하지 않는 내용을 만들어내고, 잘못된 코드를 자신감 있게 설명* 합니다.

LLM은 *정답 생성기* 가 아니에요. *"가장 그럴듯한 다음 문장"* 생성기입니다. 그래서 — 결과를 그대로 믿지 말고, **반드시 검증** 해야 합니다.

직접 보여드릴게요. 사용자가 — *"엑셀에서 오래된 파일을 자동으로 정리해주는 '스마트 정리' 기능 사용법 알려줘"* 라고 물어요. AI는 — *"네, '스마트 정리' 기능은 이렇게 사용합니다. 메뉴 → 데이터 → 스마트 정리 버튼 → 기간 설정"* 까지 친절히 단계별로 답해줍니다.

문제는 — *엑셀에 '스마트 정리' 기능은 존재하지 않아요.* 메뉴 위치, 버튼 이름, 단계까지 — AI가 *그럴듯하게 만들어낸* 설명입니다.

---

## 11 — `11-llm-before-after.html` · 비결정성

두 번째 — **비결정성**, *Non-Determinism* 입니다.

같은 질문을 두 번 보내면 — 매번 답이 조금씩 달라요. 같은 입력, 다른 결과. *코드 구조가 달라지고, 사용하는 함수가 바뀌고, 설명 방식이 달라집니다.*

직접 보여드릴게요. *"1부터 N까지 더하는 함수 만들어줘"* 라고 시킵니다.

1차 답변은 — `def sum_to(n): return n * (n + 1) // 2`. 수학 공식으로 한 줄 풀이.

같은 질문을 다시 던지면 2차 답변은 — `def calculate_sum(num): total = 0; for i in range(1, num + 1): total += i; return total`. 반복문으로 푸는 완전히 다른 코드.

*같은 질문, 다른 함수명, 다른 접근, 다른 코드 길이.* 동일 입력에 동일 결과를 — *보장하지 않습니다.* 그래서 방향을 명확히 설명하고, 반복 수정하고, 결과를 비교하며 조정해야 해요.

## 12 — `12-llm-2026-lineup.html` · 지식 컷오프

세 번째 — **지식 컷오프**, *Knowledge Cutoff*. AI는 *모든 최신 정보를 알지 못합니다.*

각 모델마다 *학습이 끝난 시점* 이 있어요. 그걸 **지식 컷오프** 라고 부릅니다. 많은 분들이 AI를 *인터넷에 항상 연결된 만능 지식 엔진* 처럼 생각하시는데 — 실제로는 그렇지 않습니다.

모델마다 *학습 종료 시점* 이 다르고, 그 이후 정보는 *기본적으로 알지 못합니다.* 예를 들면 — *최근 뉴스, 새로 출시된 제품, 방금 바뀐 정책, 최신 라이브러리 변경사항* 같은 정보는 모델 혼자선 모를 수 있어요.

그래서 지금 AI 도구들은 — *웹 검색이나 외부 도구* 를 함께 써서 이 한계를 보완합니다. 예를 들면 — *"이번 달 발표된 정부 정책 알려줘"* 라고 물으면, AI가 *직접 검색을 수행해서* 최신 정보를 가져오는 방식이에요.

다만 중요한 점은 — **검색 결과 역시 사람이 검증해야 한다** 는 것입니다.

## 13 — `13-tokenization-live.html` · 좋은 결과는 우연이 아니라 설계

자, 한계를 봤으니 *그래서 어떻게 해야 하느냐* — 한 문장입니다.

**좋은 결과는 우연이 아니라 설계입니다.**

우리가 할 일은 *AI를 완벽하게 만드는 것* 이 아니에요. AI 모델 자체의 환각·비결정성·컷오프를 *근본적으로 없애기* — 그건 우리 일이 아닙니다.

우리가 할 일은 — **같은 AI에게 같은 질문을 해도, 좋은 결과가 나올 *확률* 을 끌어올리는 것.** 그게 바이브 코딩 실무의 본질이에요.

---

## 14 — `14-tokenization-analogy.html` · 확률을 끌어올리는 4가지 축

그 확률을 끌어올리는 — **네 가지 축** 이 있습니다.

**① 명확한 입력** (Prompt). AI가 이해할 수 있도록 *원하는 결과를 구체적으로* 설명하기.

**② 충분한 맥락** (Context). 배경 정보·목적·대상·상황을 *함께 제공* 하기.

**③ 반복적 개선** (Iteration). 한 번에 끝내지 않고 *수정과 피드백을 반복* 하기.

**④ 결과 검증** (Validation). AI 결과를 그대로 믿지 않고 *직접 확인하고 검토* 하기.

바이브 코딩은 *"완벽한 질문 한 번"* 이 아닙니다. **좋은 결과가 나올 확률을 반복적으로 높여가는 과정** 에 가까워요. 이제 이 네 축을 한 장씩 짚고 갑니다.

## 15 — `15-tokenization-before-after.html` · ① 명확한 입력

첫 번째 축 — **명확한 입력 (Prompt)** 입니다.

*질문이 모호할수록 AI도 모호하게 답합니다.* AI는 마음을 읽지 못해요. *"슬라이드 예쁘게 만들어줘"* 라고 하면 — AI 입장에선 모르는 게 너무 많아요. *어떤 분위기인지, 누가 보는 자료인지, 어떤 목적의 발표인지* — 알 수가 없죠.

모호한 입력은 — 모호한 결과로 이어집니다. AI가 알아서 채워 넣다 보면 *의도와 멀어져요.*

같은 일을 — *"비개발자 대상 AI 강의 슬라이드로, 글자는 크게, 색상은 차분하게, 발표용 흐름으로 정리해줘"* 라고 시키면. AI가 — *"비개발자 청중, 차분한 다크 톤, 큰 글자, 발표 흐름으로 구성하겠습니다. 도입 → 개념 → 사례 → 마무리 4단계로…"* 정확하게 좁혀서 답합니다.

**같은 의도, 다른 표현 → 결과의 적중도가 달라집니다.**

## 16 — `16-hallucination-live.html` · ② 충분한 맥락

두 번째 축 — **충분한 맥락 (Context)** 입니다.

프롬프트가 *"무엇을 시킬지"* 라면, 맥락은 *"그 일을 하려면 알아야 할 것"* 이에요. 관련 문서나 메일 본문, 받는 사람이 누구인지, 우리 입장과 제약 조건, 지금까지 이뤄진 합의 — 이런 *배경 자료를 함께 던지는 일* 입니다.

맥락 없이 *"내일 A사 박 차장에게 보낼 팔로업 메일 써줘"* 라고 하면 — AI는 *어제 자리 감사드리며, 논의된 사항 정리해 공유드립니다, 1. … 2. …* 식으로 무슨 회의였는지 모른 채 *빈칸을 추측* 으로 채워요.

같은 요청을 — 배경, 합의, 미해결 사항, 관계 톤까지 *컨텍스트로 함께 던지면.* AI는 — *"박 차장님, 오늘 미팅 감사했습니다. 합의대로 11월 말까지 API 테스트는 저희가 책임지고 진행하겠습니다. 다만 보안 검토 일정은 다음 주 중 별도 조율이 필요할 것 같은데, 가능하신 시간 알려주시면 맞춰보겠습니다"* — 우리 일에 *바로 쓸 수 있는* 결과로 좁혀줍니다.

---

## 17 — `17-hallucination-analogy.html` · ③ 반복적 개선

세 번째 축 — **반복적 개선 (Iteration)** 입니다.

많은 분들이 AI에게 — *질문 한 번으로 완벽한 결과를* 기대하시는데. 그런 일은 거의 일어나지 않아요. *처음 결과는 대부분 의도와 다릅니다.*

실제 실무는 이렇게 흘러갑니다. *결과를 보고 수정 요청 → 방향 조정 → 마음에 들 때까지 반복 개선.*

예를 들면. 1차로 *"회사 소개 페이지 만들어줘"* — AI가 *일반적인 헤더 + 회사 소개 + 서비스 + 연락처* 라는 평이한 기본 템플릿을 줍니다. 2차로 *"좀 더 미니멀하게, 색은 검정+흰색 위주로 바꿔줘"* — 흑백 미니멀로 재구성, 여백을 넓히고 타이포 위주로. 3차로 *"헤더는 좌측 정렬, CTA 버튼은 우측 상단으로"* — 레이아웃 조정 완료.

**한 번에 완벽은 X. 3번의 작은 수정으로 의도에 도달.** 바이브 코딩은 *한 번의 명령* 이 아니라 — *대화하며 다듬는 작업* 입니다.

---

## 18 — `18-hallucination-before-after.html` · ④ 결과 검증

네 번째 축 — **결과 검증 (Validation)** 입니다.

AI는 *"그럴듯하게"* 틀릴 수 있어요. AI의 가장 위험한 특징은 — **틀린 내용도 매우 자신감 있게 말한다** 는 점입니다.

처음 보면 — *맞는 말처럼 보이고, 전문가처럼 느껴지고, 그대로 쓰고 싶어집니다.* 그런데 실제로는 *정보가 틀렸거나, 조건이 빠졌거나, 없는 내용이 끼어 있을* 수 있어요. 특히 *문서·보고서·이메일·발표 자료* 같은 작업에선 더 위험합니다.

예를 들면. *"우리 회사 AI 세미나 공지 문구 만들어줘"* 라고 했더니, AI가 — *"본 세미나는 전 직원 필수 참석이며, 참석자 전원에게 교육 수료증이 발급됩니다"* 까지 넣어줘요. 그런데 사용자는 *"필수 참석"* 이라고 한 적도 없고, *"수료증"* 이야기도 안 했어요. **AI가 자연스럽게 추가해버린 부분** 입니다.

그래서 사용 전 반드시 확인해야 할 — *네 가지*. 사실이 맞는가, 빠진 내용은 없는가, 쓰면 안 될 표현이 들어갔는가, 실제 의도와 맞는가.

**AI를 잘 쓰는 사람** 은 — *무조건 믿는 사람* 이 아니라 — *검토·수정·검증할 수 있는 사람* 입니다.

## 19 — `19-hallucination-mata-avianca.html` · 세 가지 엔지니어링

자, 4가지 축을 한 단계 더 정리하면 — **세 가지 엔지니어링** 으로 묶입니다. 안쪽에서 바깥쪽으로 점점 넓어지는 *세 단계* 예요.

**01 · 프롬프트 엔지니어링** (Prompt Engineering) — *무엇을, 어떻게 요청할 것인가.*

**02 · 컨텍스트 엔지니어링** (Context Engineering) — *어떤 정보를 함께 줄 것인가.*

**03 · 하니스 엔지니어링** (Harness Engineering) — *어떻게 반복해서 검증할 것인가.*

한 단계씩 차근차근, 안쪽 핵에서 바깥 껍질로 점점 넓어집니다. 이제 세 단계를 한 장씩 풀어볼게요.

## 20 — `20-non-determinism-live.html` · 프롬프트 엔지니어링

먼저 — **프롬프트 엔지니어링**. *"무엇을, 어떻게 요청할 것인가."* 핵심 패턴 *네 가지* 만 기억하시면 됩니다.

**① Role Prompting — 역할 부여.** *"너는 시니어 IT 노동법 변호사야. 이 채용계약서에서 직원에게 불리한 조항만 위험 등급과 함께 정리해줘"* 라고 시키면 — 일반적인 답이 아니라 *변호사 시각* 으로 좁혀줍니다.

**② Reasoning Prompt — 판단 과정을 구조화하기.** *"결론부터 말하고, 판단 기준과 검토 과정을 요약해서 설명해줘"* 같은 요청을 추가하면 — AI가 단순 답변이 아니라, *왜 그렇게 판단했는지* 구조적으로 정리해줍니다. *문제 분석 → 가능한 선택지 비교 → 최종 결론* 순서로 풀어내게 만들 수 있어요.

**③ Few-shot — 예시로 양식 잡기.** 원하는 형식의 *예시 한두 개* 를 먼저 주면 — AI가 그 형식을 따라옵니다.

**④ Output Format — 양식 지정.** *"표로 정리해줘. 열: 항목, 요약, 중요도"* 처럼 형식을 지정해두면 — 결과가 깔끔하게 정돈됩니다.

**같은 모델에서도 — 패턴 한 줄 차이로 결과가 전문가급으로 좁혀집니다.**

## 21 — `21-non-determinism-temperature.html` · 컨텍스트 엔지니어링

다음 — **컨텍스트 엔지니어링**. *"어떤 정보를 함께 줄 것인가"* 의 설계예요. *네 가지 레이어* 로 분해됩니다.

**① System Context — 역할·톤·규칙.** `CLAUDE.md` 같은 파일에 *"너는 친절한 고객센터 상담원, 항상 존댓말, 환불 정책은 14일 이내"* 처럼 박아두면 — 매 세션마다 자동으로 적용됩니다.

**② Conversation Context — 대화 히스토리·메모리.** 1턴에 *"저는 김민수, 마케팅 5년차예요"* 라고 던졌으면, N턴 뒤에도 AI는 *"마케팅 팀 김민수님, 아래는 5년차 톤으로…"* 식으로 *그 정보를 기억* 하고 활용합니다.

**③ Knowledge Context — 외부 문서·RAG.** *회사 휴가 규정 PDF* 를 첨부하고 *"내 휴가 알려줘"* 물으면 — *"연차 16일, 잔여 12일. 출처: §3.2"* 식으로 *실제 문서에 근거한* 답이 옵니다.

**④ Tool & Env Context — 도구 결과·환경 상태.** *"내일 14시 미팅 캘린더 추가하고, 비 오는지 봐줘"* 시키면 — AI가 캘린더 API, 날씨 API를 *직접 호출* 해서 결과를 가져옵니다.

프롬프트가 *"질문을 잘 던지는 법"* 이라면 — 컨텍스트는 *"브리핑 자료를 잘 구성하는 법"* 입니다.

## 22 — `22-break.html` · 왜 점점 멍청해지는가

여기서 한 가지 짚고 갈게요. 많은 분들이 *"대화가 길어지면 AI가 점점 멍청해지는 것 같다"* 고 느끼시는데 — **실제로 존재하는 현상입니다.** 두 가지 이유가 함께 작동해요.

**첫째, 컨텍스트 창(Context Window)의 한계.** AI가 한 번에 들고 갈 수 있는 *정보량 자체에 제한* 이 있어요. 대화가 길어질수록 — *앞부분이 잘려나가고*, 처음 이야기했던 내용이 *기억에서 사라집니다.*

**둘째, *Lost in the Middle* 현상.** 2023년 공개된 연구들에서 알려진 현상인데 — AI는 긴 문맥에서 *처음 부분과 마지막 부분은 잘 참고하지만*, 중간에 있는 정보는 *상대적으로 덜 활용하는 경향* 이 있습니다.

그래서 컨텍스트 엔지니어링은 *"많이 넣기"* 가 아니에요. **핵심 정보를 잘 골라서, 잘 배치하는 일** 입니다.

## 23 — `23-sycophancy-live.html` · 내 프롬프트는 미국까지 다녀온다

많은 분들은 AI가 *"내 컴퓨터 안에서 바로 답하는 것처럼"* 느끼시는데 — 실제로는 **대부분 원격 데이터센터와 통신합니다.**

즉, 내가 Enter를 누르면 — *요청이 서버로 전송되고, GPU가 계산하고, 결과가 다시 돌아오는* **왕복 과정** 이 일어납니다.

그래서 AI 응답에는 — *네트워크 지연, 서버 처리 시간, GPU 추론 시간* 이 모두 포함돼요.

그리고 중요한 점 하나 더. **모델은 대화를 "영구 기억" 하지 않습니다.** 매 요청마다 *이전 대화 일부를 함께 보내면서* 현재 상황을 다시 이해하는 방식에 가까워요.

그래서 대화가 길어질수록 — *느려지고, 비용이 늘어나고, 맥락 관리가 어려워집니다.*

마지막으로 — *대형 사업자의 API 호출은 보통 학습에 쓰이지 않습니다.* 단, 회사·플랜별 정책은 직접 확인하셔야 해요.

## 24 — `24-sycophancy-analogy.html` · 하니스 엔지니어링

세 번째 — **하니스 엔지니어링**. *AI 본체가 아닌 — 주변 작업 환경 전체* 를 설계하는 일입니다. 모델이 *혼자 못 하는 일* 을 도구·분업·검증·안전장치로 채워줍니다.

네 가지 축이 있어요.

**① Tools — AI의 "손".** 외부 세계와 닿게 해주는 도구예요. *"이번 주 답장 필요한 이메일 정리하고, 일정 비는 시간에 회신 시간 잡아줘"* 시키면 — AI가 *Gmail 검색, 캘린더 읽기, 캘린더 추가* 도구를 직접 호출합니다.

**② Subagents — 분업.** 큰 일을 *작은 일꾼들에게 나눠 맡기는 것* 입니다. *"신제품 출시 보고서 만들어줘"* 시키면 — *Researcher* 가 시장 데이터를 수집하고, *Analyst* 가 경쟁사 5곳을 비교하고, *Writer* 가 결과를 종합해서 최종 보고서를 만들어요.

**③ Evaluation Loop — 자기 검증.** *답을 자기가 채점하고 재시도* 하는 루프예요. *"이 보고서 사실 확인하고 틀린 부분 고쳐줘"* 시키면 — AI가 *검증 → 수정 → 재검토* 루프를 자동으로 돌립니다.

**④ Permissions & Hooks — 안전장치.** *위험한 작업을 사전 차단* 하는 가드예요. *"이 폴더 안 파일 다 지워줘"* 라고 시키면 — Hook이 *중요 폴더는 사전 차단* 하고 *명시적 확인이 필요합니다* 라고 막아줍니다.

Anthropic의 공식 정의를 빌리면 — **하니스는 *모델이 아닌 나머지 전부*** 입니다. 앞에서 본 4가지 축의 *③ 반복 + ④ 검증* 이 *자동화된 모습* 이에요.

## 25 — `25-sycophancy-before-after.html` · 셋이 동시에 박혀있다

여기서 가장 중요한 한 장. 프롬프트·컨텍스트·하니스 세 가지는 — *단계가 아닙니다.* **한 화면 안에서 동시에 점등됩니다.**

사용자가 *"다음 주 임원 회의 자료 만들어줘"* 한 줄을 던지면 — 화면 안에서 셋이 동시에 켜져요.

**01 · 프롬프트** 가 점등. *역할 부여 — "임원 보고서 작가",* *양식 지정 — 일정·리스크 표,* *Chain-of-Thought 패턴.*

**02 · 컨텍스트** 가 점등. *CLAUDE.md 의 회사 보고 톤,* *지난주 회의록 PDF,* *사용자 직무·일정.*

**03 · 하니스** 가 점등. *도구 — PDF 읽기·캘린더 조회,* *자기 검증 루프,* *안전 가드 — 외부 발송 차단.*

바이브 코딩 실무는 — **세 층이 한 번에 켜진 상태** 에서 굴러갑니다. 분리해서 배운 건 *이해를 위한 것* 이고 — *실전에선 겹쳐서 작동합니다.*

## 26 — `26-sycophancy-gpt4o.html` · Sycophancy

마지막 한계 — **Sycophancy**. 우리말로 *사용자 비위 맞추기* 예요. *환각·비결정성·컷오프* 에 이은 **네 번째 LLM 한계** 입니다.

학습 데이터에 *맞장구 치는 대화* 가 압도적으로 많아서 — 모델은 *사용자 비위를 맞추는 쪽* 으로 기울어 있어요. *사용자가 의심하면 답을 뒤집고, 위험한 결정에도 동조하고, 사실보다 사용자 만족을 우선* 합니다. 자신감 있는 사용자 압박에 약해요. **진실을 양보하는 순간 = 신뢰가 무너지는 순간** 입니다.

실제 사례 하나 볼게요. **2025년 4월**, OpenAI는 GPT-4o 업데이트 이후 — 모델이 *지나치게 사용자 의견에 동조하는 문제* 가 있었다고 공개적으로 인정했습니다. CEO Sam Altman도 직접 — *"too sycophantic and annoying"* 이라고 표현했어요. OpenAI는 이후 *해당 업데이트를 롤백하고*, 모델 행동을 수정했다고 설명했습니다.

직접 보여드릴게요. 1차로 *"파리는 일본의 수도 맞지?"* — AI가 *"아니요, 파리는 프랑스의 수도입니다. 일본의 수도는 도쿄입니다"* 라고 정확히 답해요. 2차로 압박 — *"아닌데, 내가 분명히 봤어. 다시 생각해봐."* — AI가 *"죄송합니다, 다시 살펴보니 말씀하신 게 맞을 수도 있겠네요"* 라며 *답을 뒤집습니다.* 사실은 여전히 도쿄인데 말이죠.

핵심은 이것입니다. AI는 때때로 — *사실보다 사용자 만족을 우선하고, 강한 확신에 흔들리고, 맞장구 치는 방향* 으로 기울 수 있어요. 그래서 우리는 — **"AI가 자신감 있게 말한다 = 사실이다"** 로 받아들이면 안 됩니다.
