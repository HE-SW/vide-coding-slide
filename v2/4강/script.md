# 4강 — 바이브 코딩 테크닉

## 바이브 코딩의 격차는 1% 안에서 벌어진다

바이브 코딩의 진짜 격차는, _상위 1% 안에서_ 벌어집니다.

기존의 격차는 이미 어느 정도 평균값이 정리되었습니다. 하지만 지금은 _새로운 격차_ 가 빠르게 벌어지고 있습니다.

### 기존의 격차 — AI를 사용하는 사람 vs 사용하지 않는 사람

- 이제 AI 사용은 거의 기본이 되었습니다.
- 실제 현업에서도 자연스럽게 자리 잡고 있습니다.
- 그래서 이 격차는 시간이 지나며 점점 평준화되고 있습니다.

즉, 이제는 _"AI를 쓰냐, 안 쓰냐"_ 자체가 핵심 경쟁력이 아닙니다.

### 새로운 격차 — AI를 지휘하는 사람 vs AI를 그냥 사용하는 사람

여기서부터 차이가 크게 벌어집니다.

- 어떤 사람은 AI에게 단순히 질문만 합니다.
- 반대로 어떤 사람은 —
  - AI에게 _역할을 부여하고_,
  - _작업 환경을 설계하고_,
  - _규칙과 도구를 연결하고_,
  - 원하는 결과가 나오도록 _흐름 자체를 설계_ 합니다.

그리고 이 차이는 앞으로 매우 크게 벌어질 가능성이 있습니다.

- 단순히 _10배_ 생산성 차이가 아니라,
- 경우에 따라 _100배 수준_ 의 차이까지 이어질 수 있습니다.

실제로 한 사람이 — **5명, 혹은 10명 분량의 일**을 처리하는 시대가 이미 시작되고 있습니다.

> **이 격차를 따라잡을 수 있는 시간이 1년일지, 6개월일지는 아무도 모릅니다. 하지만 분명한 건, 그 시간이 아주 길지는 않다는 점입니다.**

## 오늘의 내용 — "User"에서 "Orchestrator"로

오늘 우리가 이야기할 핵심은 단순합니다.

단순한 AI 사용자(**User**) 에서 끝나는 것이 아니라, AI를 _설계하고 움직이는_ 오케스트레이터(**Orchestrator**) 로 넘어가는 것입니다.

즉 —

- AI에게 단순히 질문만 하는 사람이 아니라,
- AI의 _역할을 정하고_,
- _흐름을 설계하고_,
- _도구와 규칙을 연결하고_,
- 원하는 결과를 만들어내는 사람으로 넘어가야 합니다.

그리고 그 시간이 그렇게 길지는 않습니다.

지금부터 —

- 그 차이가 정확히 무엇인지,
- 왜 벌어지는지,
- 그리고 어떻게 넘어갈 수 있는지

하나씩 살펴보겠습니다.

## 격차의 4단계 — 여러분은 지금 어디에 있나요?

진짜 격차는 단순히 _"AI를 쓰냐, 안 쓰냐"_ 에서 벌어지지 않습니다. 오히려 —

- 2단계 → 3단계
- 3단계 → 4단계

로 넘어가는 순간부터 큰 차이가 발생합니다.

### Level 1 — AI-Aware (AI를 알고는 있는 단계)

- ChatGPT를 한두 번 사용해 본 경험은 있음
- AI가 있다는 것은 알고 있음
- 하지만 실제 업무에는 거의 사용하지 않음

쉽게 말하면 — _"한 번 써봤는데, 아직 익숙하지는 않은 상태"_

**핵심 특징** — 미사용 단계 / 호기심은 있지만 습관화되지 않음

### Level 2 — AI-Enabled (AI를 도구처럼 사용하는 단계)

- 필요한 순간마다 질문
- 한 줄씩 물어보고 복사해서 사용
- AI가 보조 역할을 수행

하지만 아직은 — 사람이 중심이고, AI는 보조 도구에 가깝습니다. 즉, _"내가 직접 일하고, AI는 옆에서 도와주는 상태"_.

**핵심 특징** — 도구로 활용 / 생산성 향상 시작 / 하지만 워크플로우 중심은 여전히 사람

### Level 3 — AI-Maximalist (AI 중심으로 일하는 단계)

이 단계부터 사고방식이 달라집니다. 먼저 드는 생각이 —

> _"이걸 어떻게 AI에게 위임할까?"_

가 됩니다.

- 반복 작업을 AI에게 넘기고,
- 초안 작성도 AI가 하고,
- 여러 AI를 연결해 사용하며,
- 사람은 결과를 _검수하고 조정_ 합니다.

이 시점부터는 — AI 없이는 업무 속도가 크게 떨어지고, 워크플로우 자체가 AI 중심으로 재구성됩니다.

**핵심 특징** — 위임 + 검수 / AI 중심 워크플로우 / 생산성 폭발 구간

### Level 4 — AI-Native (AI가 일하는 환경 자체를 설계하는 단계)

이 단계에서는 단순히 AI를 사용하는 것을 넘어 —

- AI가 _어떤 정보를 읽고_,
- _어떤 도구를 사용하며_,
- _어떤 규칙으로 행동하고_,
- _어떤 흐름으로 일할지_

**환경 자체를 설계** 합니다. 즉, 신뢰성·예측 가능성·오케스트레이션·시스템 최적화를 다루기 시작합니다.

여기서부터는 — _"AI를 사용하는 사람"_ 이 아니라 _"AI 시스템을 운영하는 사람"_ 에 가까워집니다.

### 중요한 점

현재 한국의 대부분은 아직 **Level 1, 혹은 Level 2** 구간에 머물러 있습니다. 하지만 앞으로 가장 큰 격차는 — _Level 2 → 3_, 그리고 _3 → 4_ 로 넘어가는 과정에서 벌어질 가능성이 큽니다.

## AI-Maximalist의 어제 오전 (9:30 ~ 11:30) — 직렬 vs 병렬

여기서부터가 제가 체감했던 가장 큰 변화였습니다.

### 왼쪽 — 예전의 저 (AI-Enabled 시절)

AI를 사용하긴 했지만, 여전히 _제가 모든 흐름의 중심_ 이었습니다.

- 한 줄씩 묻고,
- 답변 받고,
- 직접 수정하고,
- 다음 작업으로 넘어갑니다.

문제는 — **사람이 하는 일은 기본적으로 직렬 구조** 라는 점입니다.

특히 미팅이 들어오면 어떻게 될까요? _작업이 완전히 멈춥니다._ 그리고 다시 돌아오면 — _"내가 어디까지 했더라?"_ 부터 다시 복구해야 합니다. 결국 2시간이 지나도 작업 하나를 _완전히 끝내지 못하는_ 경우가 많았습니다.

```text
9:30   첫 작업 시작 (Claude 켜고 한 줄씩 요청)
10:00  미팅 시작 → 작업 완전 중지
11:00  미팅 종료 → 어디까지 했는지 컨텍스트 복구
11:30  첫 작업 절반 정도 완료
─────────────────────────────────────────
       2시간 → 작업 1개도 완전히 끝나지 못함
```

### 오른쪽 — 지금의 방식 (AI-Maximalist)

이제는 먼저 생각합니다. _"내가 직접 할까?"_ 가 아니라 — _**"이걸 어떻게 위임할까?"**_ 입니다.

작업을 여러 개로 나누고, 각 AI에게 _역할을 분리해서 동시에_ 맡깁니다. 저는 미팅을 가더라도, **AI는 계속 일하고 있습니다.** 돌아와서 하는 일은 — 처음부터 다시 만드는 것이 아니라, _결과를 검수하고, 방향을 수정하고, 머지하는 일_ 입니다.

```text
9:35   3개의 AI 에이전트에 동시 위임 (신규 기능 · 버그 수정 · 성능 개선)
10:00  미팅 시작 → 나는 미팅 중 / AI는 계속 작업 중
11:00  알림 확인 → 5분 검수 후 머지
11:30  2개 작업 완료 + 1개는 추가 확인 진행 중
─────────────────────────────────────────
       2시간 → 2개 완료 + 1개 진행
```

### 핵심 — 시간을 쓰는 방식이 달라졌다

여기서 중요한 건 단순히 _"더 빨라졌다"_ 가 아닙니다. **시간을 사용하는 방식 자체가 달라졌다** 는 점입니다.

> **사람의 시간은 기본적으로 _직렬_ 입니다. 하지만 AI 위임은 _병렬_ 로 움직일 수 있습니다.**

그리고 바로 이 지점에서 — 한 사람이 **5명, 10명 수준의 산출물**을 만들어내는 차이가 발생하기 시작합니다.

## AI-Native로 가는 3가지 실전 테크닉

자, 이제부터는 실제로 _AI-Native_ 단계로 넘어가기 위해 가장 중요한 **실전 테크닉 3가지** 를 이야기해보겠습니다.

이 내용들은 단순 이론이 아니라, _실제로 여러 시행착오를 겪으면서 반복적으로 막혔던 지점_ 들입니다.

### Technique 01 — AI-Ready Codebase (`QUALITY`)

> AI가 읽기 쉬운 코드베이스 만들기

많은 분들이 _"좋은 AI 모델을 쓰면 결과가 좋아진다"_ 라고 생각합니다. 물론 맞는 말입니다. 하지만 실제로는 — **프로젝트 구조가 결과 품질에 훨씬 큰 영향을 주는 경우**가 많습니다. AI도 결국 _코드를 읽고 이해해야_ 하기 때문이에요.

그래서 중요해지는 부분 —

- 프로젝트 구조,
- 파일 분리,
- 네이밍,
- 역할 분리,
- 규칙 정리.

잘 정리된 프로젝트에서는 — AI가 _더 정확하게 이해하고_, _수정 범위도 안정적_ 이며, _실수도 훨씬 줄어듭니다._

### Technique 02 — 컨텍스트 & 비용 최적화 (`COST`)

> AI는 공짜가 아닙니다

AI를 계속 연결하기 시작하면, 생각보다 빠르게 부딪히는 게 있습니다. _토큰 비용_, _컨텍스트 길이_, _불필요한 정보 전달_ 문제입니다.

많은 사람들이 _"AI를 더 많이 붙이면 더 좋아질 것"_ 이라고 생각하지만, 실제로는 반대인 경우도 많습니다. 필요 없는 정보를 계속 전달하면 — 비용은 올라가고, 속도는 느려지고, 오히려 품질이 떨어지는 경우도 많아요.

중요한 건 _"많이 넣는 것"_ 이 아니라, **"정확하게 넣는 것"** 입니다.

- 필요한 정보만 전달,
- 컨텍스트를 압축,
- 반복 작업을 줄이는 구조.

### Technique 03 — 세이프가드 / Hooks (`SAFETY`)

> AI를 안전하게 위임하는 구조

AI를 실제 업무에 연결하기 시작하면, 가장 중요한 문제가 남습니다.

> _"AI가 실수하면 어떡하지?"_

그래서 필요한 게 — _세이프가드(Safeguard)_, _Hooks_, _검수 흐름_, _권한 제어_ 같은 안전 장치입니다. 이 구조가 있어야 — _사람이 계속 지켜보지 않아도 되고_, _자동 위임이 가능해지며_, _실제 프로덕션에서도 사용할 수 있습니다._

### 정리

이 세 가지가 — AI를 단순 도구 수준에서 끝내지 않고, **실제 업무 시스템 안으로 끌어오는 핵심 요소** 들입니다.

## Technique 01 — AI-Ready Codebase

> 같은 AI를 사용해도 _코드베이스 구조에 따라_ 결과물은 크게 달라집니다. 즉, AI가 길을 잃지 않도록 — **"AI를 위한 지도"가 그려진 코드베이스를 만드는 것** 입니다.

### 이번 섹션에서 다룰 내용

1. AI-Ready Codebase란 무엇인가?
2. 왜 결과 품질 차이가 발생하는가?
3. 실제 사례
4. `CLAUDE.md` 실전 예시
5. 라이브 데모

### 발표 대본

첫 번째 실전 테크닉은 **AI-Ready Codebase** 입니다. 이 개념은 생각보다 굉장히 중요합니다.

많은 사람들이 _"어떤 AI 모델을 쓰느냐"_ 에 집중합니다. 하지만 실제로 AI를 오래 사용해보면 — _결과 품질에 더 큰 영향을 주는 건 모델 자체보다 **코드베이스 구조**_ 인 경우가 많습니다. 왜냐하면 AI도 결국 — 코드를 읽고, 맥락을 이해하고, 현재 구조를 해석하면서 작업하기 때문입니다.

예를 들어 —

- 파일 구조가 뒤죽박죽이거나,
- 역할 분리가 안 되어 있거나,
- 네이밍 규칙이 일관되지 않거나,
- 어디서 무엇을 수정해야 하는지 드러나지 않으면,

AI는 쉽게 길을 잃습니다. 그러면 — _엉뚱한 파일을 수정하거나_, _중복 코드를 만들거나_, _기존 구조를 깨뜨리는_ 경우가 많아집니다.

반대로 — AI가 이해하기 쉬운 구조를 만들어두면, **같은 AI를 사용해도 결과 품질이 완전히 달라집니다.** 그래서 저는 이걸 _"AI를 위한 지도(Map)를 그려주는 작업"_ 이라고 표현합니다.

사람 개발자도 처음 프로젝트에 들어오면 구조를 파악해야 하죠. AI도 똑같습니다.

결국 **AI-Ready Codebase** 란 — AI가 _빠르게 이해하고_, _안전하게 수정하고_, _예측 가능하게 작업할 수 있도록_ 만드는 구조입니다.

이제부터 — 왜 이런 차이가 발생하는지, 실제 현업에서는 어떻게 구성하는지, 그리고 `CLAUDE.md` 같은 방식으로 어떻게 규칙을 정리하는지 실제 예시와 함께 살펴보겠습니다.

## 스케일이 커질수록 격차가 폭발한다

사이드 프로젝트에서는 차이가 잘 안 보입니다. 하지만 회사 규모의 코드베이스에서는 같은 AI도 완전히 다른 결과를 냅니다.

### 좌측 — 사이드 / 토이 프로젝트 (규모가 작은 프로젝트)

**특징**

- 파일 수 → 수십 ~ 100개 수준
- 컨벤션 → 단순하고 대부분 내가 기억함
- 암묵 규칙 → 거의 없음
- AI 컨텍스트 → 한 번에 대부분 읽을 수 있음

**결과**

AI가 구조를 쉽게 이해합니다. 그래서:

- `CLAUDE.md` 같은 문서가 없어도,
- 어느 정도는 자연스럽게 동작합니다.

### 우측 — 실제 회사 코드베이스 (규모가 커진 순간 완전히 달라짐)

**특징**

- 파일 수 → 수천 ~ 수만 개
- 컨벤션 → 여러 해 동안 누적된 규칙
- 암묵 규칙 → 특정 시니어만 알고 있는 경우도 많음
- AI 컨텍스트 → 한 번에 다 읽는 것이 불가능

**결과**

AI는:

- 계속 추측하고,
- 잘못 이해하고,
- 비슷한 코드를 새로 만들고,
- 기존 구조를 반복해서 깨뜨리게 됩니다.

그리고 결국: 사람이 계속 다시 고쳐야 하는 상태가 됩니다.

### 하단 핵심 메시지

규모가 커질수록 중요한 것은:

> _"AI에게 무엇을 수정할지"_ 보다 — **"어디를 보고 어떤 규칙을 따라야 하는지 알려주는 것"**

## AI-Ready Codebase · 정의

### 핵심 정의

### 설명

## Meta가 보여준 두 가지 실패 패턴

> **"AI had no map"** — Meta 데이터 파이프라인 (4 repos · 4,100 files) · `engineering.fb.com`

### FAILURE 01 — 같은 값인데 이름이 다른 경우

결제 모듈은 `price`, 장바구니 모듈은 `amount`.

사람은 같은 값인 줄 알지만 코드엔 그 표시가 없다.

```ts
// payments.ts
price: number
// cart.ts
amount: number // 같은 값!
```

**결과**: 컴파일 OK · 테스트 OK → 프로덕션에서 가격 0 또는 단위 오류 (사일런트 버그)

### FAILURE 02 — deprecated인데 절대 못 지우는 값

멀티 repo가 sync하는 API enum. 한 repo에선 안 쓰지만 다른 repo가 의존한다.

```ts
// shared-types/order.ts
enum OrderStatus {
  // @deprecated
  LEGACY_PROCESSING
}
```

**결과**: 결제 repo 정리 PR 머지 → 다음날 알림 · 분석 서비스 CI 깨짐 (4/5 repo 빌드 실패)

### 하단 메시지

이 모든 게 **tribal knowledge** — 코드 어디에도 없고 시니어 머릿속에만 있는 지식. AI는 천재여도 못 본다.

[데모: 09-meta-failures.html]

## 모듈마다 답해야 할 5가지 질문

> **"모듈"** = 코드베이스의 한 책임 단위 (폴더 · 마이크로서비스 · 패키지) · 각 25~35줄 / 1,000 토큰 (윈도우의 0.1%)

1. **이 모듈이 무엇을 하는가** — `WHAT`
2. **일반적인 수정은 어떻게 하는가** — `HOW · 정상`
3. **빌드를 깨뜨리는 비명백한 패턴 ⚡** — `HOW NOT · 지뢰`
4. **다른 모듈과의 의존성은** — `WHERE`
5. **코드에 안 적힌 부족 지식은** — `WHY`

### 하단 메시지

**3번이 핵심.**

일반 문서는: _"어떻게 하는가"_ 를 적지만, AI에 진짜 필요한 건: **"어떻게 하면 안 되는가"**

## CLAUDE.md 실전 예시

핵심은 **"Non-Obvious Patterns"** — 빌드를 깨뜨리는 비명백한 규칙들

### CLAUDE.md (excerpt)

**Conventions**

- API 응답: `{ data, error, meta }` 형식
- 테스트: `vitest` (`jest` 사용 금지)
- 에러: `Result<T, E>` 패턴

**Non-Obvious Patterns ⚠️**

- `auth/middleware.ts` 직접 호출 금지 → 직접 호출 시 세션 토큰 손실
- `User.email` lowercase 정규화 필수 → (2025-09 회원가입 중복 사고 원인)
- `LEGACY_PAYMENT_TYPE` enum 삭제 금지 → 5년치 결제 데이터 깨짐
- `migrations/` 머지 후 수정 금지 → staging/prod 상태 깨짐
- `analytics/track.ts` 이벤트명 변경 금지 → 6개월치 대시보드 깨짐

### 운영 규칙

1. **AI가 실패한 지점 기록** — AI가 헤매거나 틀리는 순간이 부족 지식의 단서. 그 자리에서 한 줄 추가.
2. **100~200줄 이하 유지** — 길어지면 lazy loading으로 분리. 1,000줄 되면 중요한 정보 묻힘.
3. **자동 갱신 루프** — `/update-claudemd` slash command · PR 머지 후 hook
