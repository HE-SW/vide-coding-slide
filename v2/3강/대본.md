## 02 — `02-system-prompt-leak.html` · Claude Code 시스템 프롬프트

## 09 — `09-docs.html` · docs/ — 프로젝트의 뇌

두 번째 레이어. **docs/.** *마크다운 3개* 로 AI한테 프로젝트 전체를 알려줍니다. 각 문서가 *한 가지 질문* 에 답해요.

**PRD.md — "뭘".** 뭘 만드는지. 목표, 핵심 기능, MVP에서 *제외할 것들*. *안 만들 것* 까지 명시하는 게 핵심이에요.

**ARCHITECTURE.md — "어떻게".** 어떻게 만드는지. 디렉토리 구조, 패턴, 데이터 흐름.

**ADR.md — "왜".** *왜 이렇게 만드는지.* 결정 사항 · 이유 · 트레이드오프. *뭘 포기했는지까지 적는다* 는 게 ADR의 핵심입니다.

이 세 문서가 합쳐져서 — *AI가 "뭘·어떻게·왜" 를 동시에 이해한 상태로 답을 만들어요.* 비유하자면 — **새 멤버가 입사했을 때 *신입 교육 자료 한 묶음을 건네주는 것* 과 같은 일** 입니다.

## 09b — `09b-coding-guidelines.html` · Karpathy 4원칙

`docs/` 안에 *한 폴더* 가 더 있어요. **`docs/rules/`.** *행동 규율* 을 깔아두는 곳이에요. 처음부터 직접 쓸 필요 *전혀 없습니다.* **Andrej Karpathy 의 4원칙** 을 그대로 가져다 쓰면 돼요.

**1. Think Before Coding.** 짐작하지 말고 *드러내라*. 가정을 명시하고, 해석이 둘이면 *둘 다 제시*, 모호하면 *멈추고 묻는다*.

**2. Simplicity First.** *100줄로 될 일을 1,000줄로 짜지 마라.* 시키지 않은 기능 · 추상화 · 유연성 *금지*.

**3. Surgical Changes.** *부탁한 곳만 만져라.* 인접한 코드, 주석, 포맷 *"개선" 금지.* 망가지지 않은 걸 리팩터링 금지.

**4. Goal-Driven Execution.** *검증 가능한 목표* 로 바꿔라. *"버그 고쳐줘"* 가 아니라 — *"버그를 재현하는 테스트를 먼저 짜고, 통과시켜라."*

설치는 *한 줄* 이에요. `curl` 한 줄로 `docs/rules/coding-guidelines.md` 받고, `CLAUDE.md` 마지막에 *참조 한 줄만* 남기면 끝. 다음 세션부터 AI가 이 4원칙을 따라갑니다. *마크다운 한 장으로 행동 규율이 깔리는 거죠.*

Karpathy 본인의 말로 마무리할게요. *"Don't tell it what to do, give it success criteria and watch it go."*

## 11 — `11-harness-flow.html` · /harness 실행 흐름

이게 오늘의 *흐름 슬라이드* 입니다. **`docs/` 채우고, `/harness` 치면 끝.**

5단계로 갑니다.

**1단계 — *자동*.** `docs/` 문서 전부 읽기. PRD, ARCHITECTURE, ADR, UI_GUIDE.

**2단계 — *같이*.** 사용자와 논의. 구체화할 게 있으면 *물어봅니다.* 사람이 끼어드는 *유일한 단계* 예요.

**3단계 — *자동*.** 구현 계획을 *Phase 로 쪼갭니다.* 사용자 피드백 반영.

**4단계 — *자동*.** Phase 파일 생성. `phases/{task-name}/` 폴더에 저장.

**5단계 — *자동*.** `execute.py` 실행. Phase 순차 자동 실행 + 상태 관리 + 자동 커밋.

**다섯 단계 중 네 단계는 자동.** 사람이 끼어드는 건 *2번 한 번뿐* — 방향을 같이 정하기 위해서. 이게 핵심이에요.

## 13 — `13-claude-headless.html` · claude -p 헤드리스 모드

여기서 *진짜 매커니즘* 을 짚고 갈게요. *어떻게 사람 없이 5 Phase 가 순서대로 도는 거냐.*

평소 우리가 쓰는 건 *대화형 모드* — `claude` 명령. 사람이 채팅하듯 주고받죠. 그거 말고 — **`claude -p` 라는 헤드리스 모드** 가 있어요. *프롬프트만 넣으면 알아서 실행* 합니다. *사람이 앞에 앉아 있을 필요가 없어요.*

`execute.py` 가 하는 일은 사실 단순합니다. **Phase 마다 새로운 `claude -p` 를 호출** 해요. Phase 1, Phase 2, Phase 3, Phase 4, Phase 5. *전부 따로따로 새로운 Claude 인스턴스*.

**핵심은 여기에 있어요. 각 Phase 는 *완전히 새로운 Claude* — *이전 기억이 없습니다.*** Phase 1 의 Claude 가 Phase 2 의 Claude 한테 *직접 정보를 못 넘겨요*. 그래서 — *`docs/` 에 전부 적어두는 것이 핵심* 인 거예요.

**하니스의 진짜 역할 = 매번 잊어버리는 만큼 다시 알려줄 *기록 시스템*. ** 우리가 docs를 비싸게 짜는 이유가 바로 이것 때문입니다.

## 14 — `14-hooks.html` · hooks/ — caveman-kor

네 번째 레이어. **hooks/.** *Stop 이벤트에 자동 후처리* 를 거는 거예요. 예시로 보여드릴 건 **caveman-kor**.

**문제** — 한국어 응답은 *길어지기 쉽습니다.* 친절하게 길게 답할수록 — 같은 작업인데 *토큰이 1.5~2배.* 반복 작업에서 비용이 *무섭게 쌓입니다.*

**해결** — `caveman-kor` 훅. 응답이 끝날 때마다 자동으로 — *"군더더기 빼고, 핵심만"*. 출력 톤이 *원시인 화자처럼 짧아져요.*

비교해볼게요.

**BEFORE — 일반 응답.** *"네, 알겠습니다! 말씀하신 부분에 대해서 자세히 설명드리면, 먼저 첫 번째로... 이런 흐름으로 진행하시면 됩니다."*

**AFTER — caveman-kor 적용.** *"Phase 1 완료. 다음. Phase 2 시작."*

**효과 — 출력 토큰 30~50% 절감.** 같은 작업, 같은 품질, *출력만 반토막.* 설정은 `.claude/settings.json` 한 줄이에요. *매번 같은 지시를 다시 시키지 않아도 되는 — 자동화의 가장 작은 단위* 가 hooks 입니다.
