## 2026년 3월 — Claude Code 유출 사건, 그리고 하니스 구조의 발견

2026년 3월, Anthropic의 Claude Code 소스코드 일부가 유출되는 사건이 있었습니다.

당시 많은 개발자들은 _"도대체 AI 코딩 도구는 내부적으로 어떻게 동작할까?"_ 라는 궁금증으로 그 구조를 분석하기 시작했습니다.

그 과정에서 발견된 핵심 중 하나가 바로 **하니스(Harness) 구조** 였습니다.

쉽게 말하면, AI가 단순히 혼자 동작하는 것이 아니라 —

- 어떤 정보를 먼저 읽고,
- 어떤 규칙을 참고하고,
- 어떤 도구를 사용할 수 있으며,
- 어떤 순서로 행동할지

를 미리 정리해 둔 **작업 환경** 이 존재했던 것입니다.

그리고 Claude Code 내부 역시 이런 하니스 구조를 기반으로 동작하고 있었습니다.

### 예시 — 유출 소스에서 실제로 발견된 하니스 (비개발자용 비유)

말이 어려우니, 유출된 소스 안에서 진짜로 나온 것 세 개만 일상 비유로 보여드릴게요.

```text
1️⃣  "절대 하면 안 되는 일" 목록  ──  어떤 규칙을 참고하나
    유출 소스: "메인 브랜치를 강제로 덮어쓰지 마",
              "되돌릴 수 없는 명령은 실행 전에 사용자에게 한 번 더 물어봐",
              "AI가 멋대로 저장(commit)·전송(push)하지 마"
    👉  신입 사원 매뉴얼 첫 장의 "절대 하지 말 것" 목록

2️⃣  "명령 실행 전 23개 체크리스트"  ──  어떤 순서로 행동하나
    유출 소스: 터미널 명령 하나를 실행하기 전에 23개 항목을 차례로 점검
              ("이 명령이 시스템을 망가뜨리나 / 민감한 파일을 건드리나 …")
    👉  비행기 이륙 전 조종사가 짚어 내려가는 점검표

3️⃣  "사용자가 짜증나 있는지" 감지  ──  어떤 정보를 먼저 읽나
    유출 소스: 욕설·짜증 표현 패턴이 통째로 박혀 있어서, 사용자가
              "이거 왜 안 돼" 같은 말을 하면 AI가 톤을 바꾼다
    👉  콜센터 상담 매뉴얼의 "고객이 화났을 때 응대법"
```

세 개 다 — _AI를 혼자 두지 않고, "이건 읽어라 / 이 규칙 지켜라 / 이 순서로 해라" 를 미리 정리해 둔_ 작업 환경입니다. 여러분이 Claude Code를 실행하는 그 순간, 이미 이 하니스는 작동하고 있었던 거죠.

## 시스템 프롬프트 외에도 — 이미 깔려 있는 3가지

시스템 프롬프트는 _내장 하니스의 한 가지 사례_ 일 뿐이에요. Claude Code 안에는 _여러분이 따로 손대지 않아도 작동하는_ 안전장치가 세 가지 더 있습니다.

```text
🛡️  권한 체크          위험한 명령 직전, 사용자에게 한 번 더 묻는 시스템
                      (rm, force push 같은 명령은 실행 전 확인 창)

🚧  도구 경계          AI가 쓸 수 있는 도구 / 못 쓰는 도구의 명확한 울타리
                      (Read·Write·Edit·Bash 등 정해진 목록 안에서만 동작)

📋  JSON 스키마         도구 호출이 자유 텍스트가 아닌 *정해진 양식* 안에서만
                      (양식에 안 맞는 호출은 시스템이 거부 — 환각 차단)
```

비유하자면 — **시스템 프롬프트** 가 _"하면 안 되는 일 목록"_ 이고, **권한 체크** 가 _"중요한 일 전에 한 번 더 묻기"_, **도구 경계** 가 _"갈 수 있는 길의 울타리"_, **JSON 스키마** 가 _"신청서 양식"_ 입니다. 세 가지 다 — 여러분이 깔지 않았는데 이미 깔려 있던 거예요.

## 한 층만 더 올리면 됩니다

Claude Code 자체에도 이미 **하니스(Harness)** 가 적용되어 있습니다. 예를 들면 —

- 시스템 프롬프트
- 권한 체크
- 사용할 수 있는 도구
- JSON 스키마
- 실행 규칙

같은 구조들이 미리 세팅되어 있죠. 2층짜리 건물로 그리면, 이게 _1층_ 입니다.

```
┌─────────────────────────────────────────────────────────────┐
│ 2층 — 내 프로젝트 하니스 (여러분이 올리는 층)                 │
│                                                             │
│   [CLAUDE.md]   [docs/]   [skills/]   [hooks/]              │
├─────────────────────────────────────────────────────────────┤
│ 1층 — 내장 하니스 (이미 있음, Claude Code 자체)              │
│                                                             │
│   [시스템프롬프트] [권한체크] [도구] [JSON스키마] [실행규칙]    │
└─────────────────────────────────────────────────────────────┘
```

하지만 이런 기본 하니스에는 _한계_ 가 있습니다. Claude Code는 많은 사람이 쓸 수 있도록 _범용적으로_ 설계된 도구이기 때문이에요. 그렇다 보니 실제 프로젝트에서는 이런 질문이 생깁니다.

- "우리 프로젝트에서는 어떤 패턴을 써야 하지?"
- "어떤 API를 호출해야 하지?"
- "AI에게 어디까지 접근 권한을 줘야 하지?"
- "우리 팀의 규칙은 어떻게 반영하지?"

결국 중요한 건 — **우리 프로젝트에 맞는 세팅** 입니다. 그래서 이번 시간에는 Claude Code를 그대로 쓰는 것을 넘어서, _우리 프로젝트에 맞게 AI가 일할 수 있도록_ — **커스텀 하니스(2층)를 직접 올려보겠습니다.**

## 잠깐, 한 층을 _끝까지_ 올리면 — oh-my-claudecode

본격적인 실습 전에, _진짜로 한 층을 끝까지 다 올리면_ 어떻게 보이는지 한 가지 사례를 미리 보여드릴게요. 한국 개발자 **Yeachan-Heo** 가 만든 오픈소스, **oh-my-claudecode** 입니다.

```text
oh-my-claudecode  —  "학습 곡선 제로. 최대 파워."

🤖  전문 에이전트  19개      설계 / 코딩 / 테스트 / 리서치 등 역할별로 분리된 AI 팀
                            예시: architect.md · executor.md · code-reviewer.md · test-engineer.md
📦  스킬           36개      반복 작업을 명령어 하나로 실행
                            예시: /autopilot · /ultrawork · /team · /skillify · /hud
🔁  자동 파이프라인 5단계    단계별 자동 진행, 순서 건너뛰기 불가
                            plan → prd → exec → verify → fix
📊  모니터링 + 비용 추적     실시간 대시보드 · 토큰 비용 · 스마트 모델 라우팅 (토큰 30~50% ↓)
                            예시: src/hud/ · src/feature/model-routing/
```

**처음부터 다 직접 짤 필요는 없습니다.** _이미 잘 만들어진 한 층을 통째로 가져다 쓸 수도_ 있어요. 다만 — _내 프로젝트에 딱 맞는 한 층을 직접 짜는 능력_ 은 별도로 있어야 합니다. 그래서 다음 한 장으로 — _직접 짤 때의 뼈대_ 를 먼저 보여드릴게요.

## 범용 하니스 vs 커스텀 하니스 — 한 층 올리는 방법은 자유

oh-my-claudecode를 보면서 _"우와, 나도 저거 깔아야겠다"_ 싶을 수도 있고, _"내 프로젝트엔 너무 복잡한데"_ 싶을 수도 있어요. 둘 다 정상적인 반응입니다. 한 층을 올리는 _방식이 두 가지_ 라서 그래요.

```text
┌─────────────────────────────────────┬─────────────────────────────────────┐
│  범용 하니스 (오픈소스)              │  커스텀 하니스 (직접 구성)           │
│  oh-my-claudecode 같은 프로젝트       │  내 프로젝트에 맞게 직접 구성         │
├─────────────────────────────────────┼─────────────────────────────────────┤
│  에이전트 분리, 파이프라인, 모니터링  │  CLAUDE.md 에 프로젝트 규칙          │
│  한 방에 풀세팅                      │  docs/ 에 기획문서 + 아키텍처        │
│  내 프로젝트 규칙은 *모름*            │  hooks 으로 자동 검증                │
│  초기 세팅이 복잡할 수 있음           │  가볍고 빠르게 시작 가능             │
│  수정이 어려움                       │  수정이 쉬움                         │
└─────────────────────────────────────┴─────────────────────────────────────┘
```

범용 하니스는 _남이 이미 잘 만들어둔 한 층을 통째로 가져오는_ 방식이에요. 빠르고 강력하지만 — _내 프로젝트의 특수한 사정_ 은 모릅니다. 커스텀 하니스는 _내 프로젝트에 딱 맞게 직접 짜는_ 방식이고요. 가볍게 시작하고 자유롭게 고칠 수 있습니다.

**중요한 건 한 층을 올린다는 것. 방법은 자유입니다.** 이번 강의에서는 _직접 한 층을 올리는 법_ — 즉 커스텀 쪽 — 을 자세히 보겠습니다.

## 하니스 프레임워크 — 4개의 레이어로 구성된 프로젝트 뼈대

자, 그러면 _직접 한 층 올릴 때_ 의 뼈대를 한 장으로 보여드릴게요. 프로젝트 폴더에 이런 파일·폴더들을 두면 됩니다.

```text
project/
├── CLAUDE.md                       ← 프로젝트 헌법
├── docs/
│   ├── rules/
│   │   └── coding-guidelines.md    ← 코딩 가이드라인
│   ├── PRD.md                      ← 뭘 만드는지
│   ├── ARCHITECTURE.md             ← 어떻게 만드는지
│   ├── ADR.md                      ← 왜 이렇게 만드는지
│   └── UI_GUIDE.md                 ← 어떻게 보여야 하는지 (선택)
├── .claude/
│   ├── skills/
│   │   ├── harness.md              ← /harness — 원스톱 실행
│   │   └── review.md               ← /review — 규칙 기반 리뷰
│   ├── hooks/
│   │   └── caveman-kor             ← 한국어용 caveman (output 토큰 절약)
│   └── settings.json               ← hooks 설정
├── scripts/
│   └── execute.py                  ← Phase 순차 실행 + 상태 관리
└── phases/                         ← Phase 파일 + 실행 상태
```

복잡해 보이지만 — 핵심 레이어는 **네 가지** 입니다.

```text
1. CLAUDE.md   →  프로젝트 헌법 (가장 위에 한 장)
2. docs/       →  뭘·어떻게·왜·어떻게 보여야 하는지 (4종 문서)
3. skills/     →  슬래시 한 줄로 호출하는 작업 절차
4. hooks/      →  특정 사건마다 자동으로 일어나는 행동
```

지금부터 — _이 네 레이어를 차례대로 한 슬라이드씩_ 보겠습니다.

## 2층 #1 — CLAUDE.md, 프로젝트의 헌법

먼저 가장 위 — **CLAUDE.md** 입니다.

프로젝트 루트에 `CLAUDE.md` 라는 마크다운 파일을 두면 — Claude Code가 _세션을 새로 열 때마다 자동으로 읽어서_ 컨텍스트에 넣어줍니다. 매번 같은 지시를 다시 입력하지 않아도 되는 거죠.

**왜 "헌법"이라고 부르냐면** — _프로젝트의 가장 높은 수준의 규칙_ 을 담기 때문이에요. 실제 템플릿은 이렇게 생겼습니다.

```markdown
## 프로젝트: {프로젝트명}

목표: {목표 작성}

## 기술 스택

- {프레임워크} {언어} {주요 라이브러리}

## 아키텍처 규칙

- CRITICAL: {절대 지켜야 할 규칙 1
  (예: 모든 API 로직은 app/api/ 라우트 핸들러에서만 처리)}

## 개발 프로세스

- CRITICAL: 새 기능 구현 시 반드시 테스트를 먼저 작성하고,
  테스트가 통과하는 구현을 작성할 것 (TDD)

## 클로드 코드 사용시 주의사항

docs/rules/coding-guidelines.md
```

여기 두 가지 패턴이 핵심이에요. 첫째 — **`CRITICAL:` 접두어**. AI에게 _"이건 절대 어기면 안 된다"_ 고 강조하는 표시입니다. 영어 대문자가 들어가면 모델이 더 무겁게 받아들여요. 둘째 — **`docs/rules/coding-guidelines.md` 참조**. CLAUDE.md는 _높은 수준의 헌법_ 만, 세부 규칙은 _docs로 넘기는_ 연결 구조입니다.

한 장만 잘 적어두면 — 같은 AI에게 같은 질문을 해도 답의 질이 _눈에 띄게_ 달라집니다.

[데모: 08-claudemd.html]

## 2층 #2 — docs/, 프로젝트의 뇌

CLAUDE.md가 *한 장짜리 헌법*이라면, **docs/** 는 _프로젝트의 뇌_ 입니다. _마크다운 3개_ 만 잘 적어두면 — AI에게 _프로젝트 전체_ 를 알려줄 수 있어요. 각 문서는 _한 가지 질문_ 에 답합니다.

### 📘 PRD.md — _뭘_ 만드는지

```markdown
# PRD: {프로젝트명}

## 목표

{한 줄 요약}

## 핵심 기능

1. {기능 1}
2. {기능 2}

## MVP 제외 사항

- {안 만들 것 1}
```

### 🏗️ ARCHITECTURE.md — _어떻게_ 만드는지

```markdown
# 아키텍처

## 디렉토리 구조

{폴더 트리}

## 패턴

{디자인 패턴}

## 데이터 흐름

{데이터 흐름도}
```

### 🧭 ADR.md — _왜_ 이렇게 만드는지

```markdown
### ADR-001: {결정 사항 (예: Next.js App Router 선택)}

**결정**: {뭘 선택했는지}
**이유**: {왜 선택했는지}
**트레이드오프**: {뭘 포기했는지}

### ADR-002: {결정 사항}

...
```

세 문서가 합쳐져서 — AI가 _"뭘·어떻게·왜"_ 를 동시에 이해한 상태에서 답을 만듭니다. 새 멤버가 입사할 때 신입 교육 자료 한 묶음을 건네주는 것과 같은 일이에요. **한 번 잘 적어두면 강의 1편이 4편으로 늘어도, 새 기능이 추가돼도, 같은 품질로 확장됩니다.**

## 2층 #2 보강 — docs/rules/, 행동 규율을 한 장으로 깔기

방금 본 _PRD·ARCHITECTURE·ADR_ 세 문서가 _"우리 프로젝트가 어떤 모양인지"_ 를 알려준다면, 한 가지가 더 필요합니다. **AI가 코드를 짜는 _매 순간_ 따라야 할 행동 규율** — 이걸 `docs/rules/coding-guidelines.md` 한 장에 깔아둡니다.

근데 직접 쓸 필요는 _전혀_ 없어요. **이미 검증된 65줄짜리 가이드** 가 GitHub에 깔려 있습니다.

```text
andrej-karpathy-skills        10만 ★, MIT
   원전: Andrej Karpathy 트윗  (2026-01-26)
   정리: Forrest Chang
   → LLM 코딩의 4대 실패 패턴을 4원칙으로 압축한 단 한 장
```

### Karpathy의 4대 실패 패턴 → 4원칙

```text
🤔  Think Before Coding     →  짐작 금지. 혼란을 드러내라. 트레이드오프를 말해라
✂️  Simplicity First        →  100줄로 될 일을 1,000줄로 짜지 마라
🔪  Surgical Changes        →  부탁한 곳만 만져라. 무관한 코드 손대지 마라
🎯  Goal-Driven Execution   →  검증 가능한 목표로 변환. 통과할 때까지 루프해라
```

특히 **4번 — 검증 가능한 목표** 가 핵심이에요. _"버그 고쳐줘"_ 대신 _"버그를 재현하는 테스트를 먼저 짜고, 그걸 통과시켜라"_. 이렇게 _목표_ 를 주면 AI가 _스스로 루프_ 합니다. _명령_ 을 주면 매번 사람의 명료화가 필요해요.

### 어떻게 깔까 — 한 줄

```bash
curl -o docs/rules/coding-guidelines.md \
  https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

CLAUDE.md 마지막 줄에 _"규칙 세부는 docs/rules/coding-guidelines.md 참조"_ 만 남기면 끝. **다음 세션부터 AI가 코드를 짤 때마다 이 4원칙을 따라갑니다.** 매번 _"불필요한 리팩터링 하지 마"_ 라고 잔소리하지 않아도 되는 거죠.

이게 바로 5번 슬라이드(*"이미 잘 만들어진 한 층을 가져다 쓴다"*) 메시지의 가장 _작은_ 사례입니다. oh-my-claudecode 같은 풀세팅까지 안 가도 — _markdown 한 장_ 으로 행동 규율이 깔려요.

## 2층 #3 — skills/, 슬래시 한 줄로 호출하는 작업 절차

세 번째, **skills/** 입니다. 1강 23번 슬라이드에서 잠깐 언급한 그 개념이에요.

스킬은 — _자주 쓰는 작업 절차에 이름표를 붙여둔_ 마크다운 파일입니다. `.claude/skills/` 폴더에 파일 하나 만들어두면, Claude Code 안에서 _슬래시 한 줄_ 로 그 절차를 호출할 수 있어요.

이번 강의 레포의 스킬 두 개가 좋은 예시입니다.

```text
.claude/skills/
├── harness.md     →  /harness    docs/ 다 읽고 → 사용자와 논의 →
│                                  Phase로 쪼개고 → 순차 실행까지
│                                  *원스톱 풀 자동화*
│
└── review.md      →  /review     docs/rules/ 의 규칙들을 기준으로
                                   방금 작성한 코드를 *규칙 기반 리뷰*
```

**핵심은 — 한 번 잘 만들어두면 _반복해서_ 쓸 수 있다** 는 점입니다. 매번 길게 시키는 대신 _슬래시 한 줄_. `/harness` 가 어떻게 작동하는지, 다음 한 장으로 자세히 보여드릴게요.

## /harness 실행 흐름 — docs/ 채우고, `/harness` 치면 끝

지금까지 본 _모든 게_ 이 한 줄로 모입니다.

> **docs/ 채우고, `/harness` 치면 끝.**

방금 본 `/harness` 스킬이 _실제로 작동할 때_ 이렇게 진행됩니다.

```text
1️⃣  [자동]   docs/ 문서를 전부 읽는다
            └─ PRD · ARCHITECTURE · ADR · UI_GUIDE

           ↓

2️⃣  [같이]   사용자와 논의
            └─ 구체화할 게 있으면 물어본다

           ↓

3️⃣  [자동]   구현 계획을 Phase 로 쪼갠다
            └─ 사용자 피드백 반영

           ↓

4️⃣  [자동]   Phase 파일 생성
            └─ phases/{task-name}/ 폴더에 저장

           ↓

5️⃣  [자동]   execute.py 실행
            └─ Phase 순차 자동 실행 + 상태 관리 + 자동 커밋
```

다섯 단계 중 _네 단계는 자동_ 입니다. 사람이 끼어드는 건 2번 한 번뿐 — 그것도 _"AI가 이해를 못해서"_ 가 아니라 _"방향을 같이 정하기 위해서"_. **2층을 한 번 잘 깔아두면 — 매번 같은 말을 다시 시킬 필요가 없는, 진짜 하니스가 만들어집니다.**

## execute.py 실행 결과 — Phase별 상태 관리 + 자동 커밋

추상적이지 않냐고요? _실제로 돌렸을 때 무엇이 나오는지_ 보여드리겠습니다. _MVP 만들기_ 작업에서 `/harness` 가 5단계를 순차 실행한 직후의 터미널 출력입니다.

```text
$ python3 scripts/execute.py mvp

============================================
 Harness Executor
 Task: mvp  |  Phases: 5  |  Pending: 5
============================================

 ✓ Phase 1: 프로젝트-초기화        [180s]
 ✓ Phase 2: 타입-+-유틸리티        [300s]
 ✓ Phase 3: api-라우트             [240s]
 ✓ Phase 4: ui-컴포넌트            [300s]
 ✓ Phase 5: 메인-페이지-통합       [150s]

============================================
 Task 'mvp' completed!
============================================
```

성과 요약 —

```text
⏱  17분 — 총 소요 시간
✅  5/5 — Phase 완료
🧪  54개 — 테스트 통과
```

여러분이 입력한 명령은 _단 한 줄_ — `/harness` 였습니다. 그 한 줄이 docs/ 를 읽고, Phase로 쪼개고, 다섯 단계를 순차 실행하고, 각 단계마다 자동 커밋하고, 테스트를 돌려서 통과시키는 _모든 일_ 을 처리한 결과예요. **이게 잘 깔린 2층의 위력입니다.**

근데 _어떻게_ 사람 없이 다섯 단계가 흘러갔을까요? 잠깐 매커니즘을 까봅시다.

## claude -p 헤드리스 모드 — 자동화에 특화된 모드

`execute.py` 가 어떻게 그 5단계를 _사람 없이_ 진행했는지, 핵심은 **`claude -p`** 라는 _헤드리스(headless) 모드_ 입니다. _스크립트에서 프로그래밍적으로 Claude를 실행하는_ 모드예요.

```text
┌──────────────────────────────┬──────────────────────────────┐
│  💬  대화형 모드             │  🤖  헤드리스 모드            │
│      claude                  │      claude -p               │
├──────────────────────────────┼──────────────────────────────┤
│  사람이 *채팅하듯이* 주고받기 │  프롬프트를 넣으면 *알아서 실행* │
│  우리가 평소에 쓰는 방식      │  사람이 앞에 앉아 있을 필요 없음 │
└──────────────────────────────┴──────────────────────────────┘
```

`execute.py` 가 Phase마다 _새로운 `claude -p`_ 를 호출합니다. 그림으로 그리면 —

```text
 Phase 1    →   Phase 2    →   Phase 3    →   Phase 4    →   Phase 5
 claude -p     claude -p      claude -p      claude -p      claude -p
   #1            #2             #3             #4             #5
              ✗ 기억 없음    ✗ 기억 없음    ✗ 기억 없음    ✗ 기억 없음
```

여기 _재미있는 점_ — 각 Phase 는 _완전히 새로운 Claude_ 입니다. **이전 Phase 기억이 없어요.** 그래서 docs/ 에 _전부_ 적어두는 게 중요한 거예요. Phase 5 가 Phase 1 의 결정을 알 수 있는 _유일한 길_ 은, _문서로 남아 있는 것_ 뿐입니다.

**하니스의 진짜 역할은 — AI가 매번 잊어버리는 만큼, 우리가 _매번 다시 알려줄 수 있는 기록 시스템_ 을 만들어주는 것.** 1강에서 본 _"하니스의 모든 요소는 모델이 혼자서는 못 하는 일에 대한 가정을 코드화한 것"_ — 그 정의의 가장 진한 사례가 _바로 이 매커니즘_ 입니다.

## 2층 #4 — hooks/, caveman-kor 으로 output 토큰 최적화

마지막 네 번째, **hooks/** 입니다.

훅(hook)은 — _어떤 사건이 일어날 때마다 자동으로 실행되는 행동_ 이에요. CLAUDE.md / docs / skills 가 _"미리 알려주거나, 호출하면 실행되는 정보·절차"_ 라면, hooks는 _"여러분이 시키지 않아도 알아서 일어나는 일"_ 입니다.

이번 강의 레포에서 가장 _비용 절감 효과_ 가 큰 훅 한 가지 — **`caveman-kor`** 을 자세히 보여드릴게요.

### 문제 — 한국어 응답은 _길어지기 쉽습니다_

한국어로 질문하면 Claude가 친절하게 길~게 답해요. _같은 작업_ 인데도 영어 대비 출력 토큰이 _1.5~2배_ 더 들어갑니다. 반복 작업이 많아지면 비용이 무섭게 쌓이죠.

### 해결 — `caveman-kor` 훅

응답이 끝날 때마다 자동으로 _"군더더기 빼고, 핵심만 남겨라"_ 라는 후처리가 들어갑니다. 출력 톤이 _원시인 화자_ 처럼 짧아져요 — _"이렇게. 됐다."_ 식.

```text
일반 응답 (BEFORE)
  "네, 알겠습니다! 말씀하신 부분에 대해서 자세히 설명드리면,
   먼저 첫 번째로 …(이하 200자) … 이런 흐름으로 진행하시면 됩니다."

caveman-kor 훅 적용 후 (AFTER)
  "Phase 1 완료. 다음. Phase 2 시작."
```

### 효과 — 출력 토큰 _30~50% 절감_

같은 작업이 _같은 품질로 끝나지만_, 출력 토큰만 반토막. 그만큼 비용도 절감됩니다.

설정은 `.claude/settings.json` 한 줄에 — `Stop` 이벤트에 `caveman-kor` 훅을 등록하면 끝. **매번 같은 지시를 다시 시키지 않아도 되는** 자동화의 가장 작은 단위. _이게 hooks의 본질_ 입니다.

## 프레임워크 완성 — 4개 레이어 요약

오늘 본 _모든 걸 한 장_ 으로 정리하겠습니다. 2층의 네 레이어가 이렇게 협력합니다.

```text
1️⃣  docs/ — 프로젝트의 뇌
    마크다운 4개로 프로젝트 전체를 설명
    [PRD.md]  [ARCHITECTURE.md]  [ADR.md]  [UI_GUIDE.md]

2️⃣  CLAUDE.md — 프로젝트의 헌법
    CRITICAL 규칙 + TDD
    [CRITICAL 규칙]  [TDD]

3️⃣  /harness + execute.py — 실행 엔진
    Phase 분리 + 순차 실행 + 상태 관리
    [/harness]  [execute.py]  [/review]

4️⃣  Hooks — 토큰 최적화
    응답 끝날 때마다 자동으로 군더더기 제거 → 출력 토큰 30~50% 절감
    [caveman-kor]
```

```text
┌──────────────────────────────────────────────────────┐
│  🎯  GitHub에서 클론 받아서 docs/ 마크다운만 채우세요  │
│      그리고  /harness  치면 끝입니다                  │
└──────────────────────────────────────────────────────┘
```

이게 오늘 보여드린 모든 것입니다. _4개 레이어_ 만 잘 깔아두면 — _어떤 프로젝트든_ 같은 방식으로 자동화됩니다.

## 프로젝트 만들어보기 — 실제 제작 과정

오늘은 **세팅** 까지 했습니다. 4강에서는 — 오늘 깐 두 층의 하니스 위에서, _실제 프로젝트 하나를 처음부터 끝까지 완주_ 합니다.

### 무엇을 만드냐면 — **하나의 실제 프로젝트**

기획부터 코드까지, 실제로 쓸 수 있는 프로젝트 하나를 — 오늘 배운 4개 레이어(`CLAUDE.md` · `docs/` · `skills/` · `hooks/`)를 직접 깔아가며 처음부터 끝까지 만듭니다. _어떤 프로젝트를 만들지는 4강에서 공개_ 합니다.

### 5단계 워크플로우

```text
🦴 뼈대  →  📋 기획  →  🔧 하니스 설계  →  🤖 실행  →  🔍 리뷰
 [같이]      [같이]        [같이]           [AI 혼자]     [같이]
```

| 단계           | 누가        | 무엇을                             |
| -------------- | ----------- | ---------------------------------- |
| 1. 뼈대        | 같이        | 폴더 구조 + 기본 파일              |
| 2. 기획        | 같이        | PRD·ARCHITECTURE·ADR 작성          |
| 3. 하니스 설계 | 같이        | CLAUDE.md + skills + hooks         |
| 4. 실행        | **AI 혼자** | `/harness` 한 줄 → Phase 자동 실행 |
| 5. 리뷰        | 같이        | `/review` + 사람의 마지막 점검     |

### 이게 하니스 기반 워크플로우입니다

> **기획은 같이, 실행은 AI 혼자, 리뷰는 다시 같이.**

비개발자가 _자기 손으로_ — 실제로 쓸 수 있는 무게의 프로젝트를 만들어내는 흐름을, 한 시간짜리 라이브로 직접 보여드리겠습니다.

### Before — 디자인 가이드 없이

```text

엣지 케이스 미기술            →  API 파싱 에러
```

### After — UI_GUIDE + 엣지 케이스 추가 후

```text
+ UI_GUIDE.md                →  안티 슬롭, 깔끔한 다크 UI
+ ADR-009                    →  JSON 파싱 에러 해결
```

## 오늘 한 걸 정리하면

### 1. 하네스는 이미 쓰고 있어요

Claude Code든 Codex든, AI 코딩 도구 자체가 하네스입니다. 거창한 게 아닙니다.

### 2. 나만의 하네스를 한 층 올려보자

CLAUDE.md에 규칙, docs/에 기획서, Hook으로 자동 검증. 이 세 가지면 충분합니다.

### 3. 프레임워크 가져다 쓰세요

GitHub에 올려놨습니다. 클론 받아서 마크다운만 채우면 바로 사용 가능. 비개발자도요.

```text
📦  하네스 프레임워크 GitHub
    github.com/jha0313/harness_framework
    clone → docs/ 채우기 → /harness
```

**하네스를 한 층 올리면, 바이브 코딩의 품질이 확 올라갑니다.**
