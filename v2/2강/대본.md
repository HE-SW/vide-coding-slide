## 04 — `04-claude-md-hierarchy.html` · CLAUDE.md — 5단계 계층

먼저 **CLAUDE.md** 부터. 1강에서 *팀 룰북* 이라고 봤죠. 그게 *한 장* 이 아니라 — **5단계 계층** 입니다.

**Enterprise** 는 조직 정책, 회사 사훈 같은 거예요. 사용자가 끌 수 없습니다. **User**, 글로벌 — `~/.claude/CLAUDE.md` 에 두면 *내 모든 프로젝트*에 공통으로 적용됩니다. **Project**, 프로젝트 루트의 `CLAUDE.md` — 팀 위키처럼 *레포에 커밋* 합니다. **Local**, `CLAUDE.local.md` — 내 책상 메모, `.gitignore` 에 들어가요. **Subdirectory** — 폴더별로 두면 *그 폴더 작업 시에만* 자동 로드됩니다.

위에서 아래로 갈수록 *더 구체적이고*, **더 구체적인 쪽이 우선되도록 설계** 됐습니다. 단, *강제 설정이 아니라 컨텍스트* 라서 100% 일관되게 따르진 않을 수 있어요. 그래서 *어겨선 안 되는 규칙* 은 다음 슬라이드의 **CRITICAL 패턴** 으로 강도를 키워야 합니다.

## 05 — `05-claude-md-import.html` · @import — 분리, 그러나 lazy 아님

200줄 제한 때문에 — *모든 내용을 직접 쓰면* 컨텍스트를 잡아먹어요. 해법은 **참조만 적고 본문은 별도 파일로 분리** 하는 겁니다.

나쁜 예 — CLAUDE.md 안에 API 엔드포인트 50개를 다 적어버리면 200줄 훌쩍 넘어가요. 좋은 예 — 한 줄짜리 참조만 두는 겁니다. *"API 스펙: @docs/api-spec.md"* 같은 식으로요. `@import` 는 *시작 시* 최대 5단계 재귀로 *모두* 컨텍스트에 로드됩니다.

여기서 한 가지 짚고 갈게요. **@import 는 lazy load 가 아닙니다.** 시작할 때 *모두 불러옵니다.* 그래서 *200줄 룰이 더 중요* 한 거예요. 분리는 *작성·관리 편의* 용이지, 토큰 절약은 아닙니다. 진짜 *lazy load* — 필요할 때만 읽기 — 는 따로 있어요. *Subdirectory CLAUDE.md* (해당 폴더 작업 시에만 자동 로드)와 *Skills* (호출 시에만 본문 로드). 이 두 가지는 뒤에서 봅니다.

비유로 풀면 — *팀 위키 한 페이지에 500개 링크를 다 적어둔 것* 과 *한 페이지엔 목차만, 본문은 별도 페이지로 분리한 것* 의 차이입니다. 후자가 *읽기·고치기·검토*가 다 쉬워요.

## 07 — `07-critical-pattern.html` · 200줄 룰 + CRITICAL 패턴

CLAUDE.md 작성 모범 사례, 두 가지만 짚습니다.

**첫째, 200줄 이하** 로 유지하세요. 길어지면 컨텍스트 소모가 커지고, Claude가 규칙을 *일관되게 못 따릅니다*.

**둘째, CRITICAL 접두어**. 정말 어겨선 안 되는 규칙 앞에 `CRITICAL:` 을 붙이는 패턴이에요. 예를 들어 *"CRITICAL: 모든 API 로직은 app/api/ 라우트 핸들러에서만"*, *"CRITICAL: secrets는 .env에만, 코드 하드코딩 절대 금지"*.

왜 통하느냐. 모델은 *영어 대문자* 를 *강조 신호* 로 가중치 더 둡니다. Anthropic 공식 가이드도 `CRITICAL`, `IMPORTANT`, `NEVER`, `ALWAYS` 를 *지시문 강도 표시* 로 권장합니다.

팁 하나 — bullet 의 *맨 앞* 에 둬야 *전체 항목 라벨* 로 읽힙니다. 중간에 들어가면 *조건절* 로 약하게 해석돼요.

## 08 — `08-plan-mode.html` · Plan Mode — 도면 먼저

CLAUDE.md로 *프로젝트 룰북* 은 박았어요. 이제 그 룰북 위에서 *실제로 일을 시작하는 모드*, **Plan Mode** 차례입니다.

비유 — *집 짓기* 예요. 망치를 들고 *바로 못부터* 박는 사람과, *도면을 그리고 시작* 하는 사람. AI한테 일 시킬 때도 똑같습니다.

비교해드릴게요. *"사용자 인증 기능 만들어줘"* 라고 즉시 시키면 — AI가 갑자기 8개 파일 만지기 시작하고, 3번째 파일에서 방향이 어긋나고, 결국 처음부터 다시 시켜야 합니다. 토큰과 시간 둘 다 낭비예요.

같은 일을 `/plan` 으로 시작하면 — Claude가 *계획만 제시* 합니다. *"1. DB 스키마 변경, 2. API 엔드포인트 3개, 3. 프론트 페이지 1개, 4. 각 단계마다 테스트"*. 청중이 *읽고 수정* 한 다음, 만족하면 *Auto-Accept 로 전환해서 실행* 합니다.

Anthropic 공식 4단계는 — **Explore → Plan → Code → Verify** 예요. Plan Mode는 이 흐름의 2번 칸을 *공식 모드* 로 박아둔 겁니다.

지금 한 번 직접 봐봅시다. `Shift + Tab` 두 번 누르면 Plan Mode 진입해요.

## 09 — `09-four-modes.html` · 모드 4종 — 언제 어느 모드

Plan Mode를 봤으니, 나머지 모드들도 한 장에 정리합니다.

`Shift + Tab` 을 누르면 모드가 순환합니다. 네 가지가 있어요. **Normal** — 실제 설정 이름은 `default`. 매 작업 전 묻습니다. **Auto-Accept** — `acceptEdits`. 파일 편집과 *일부 안전한 작업* 을 자동 승인해요. **Plan** — `plan`. 방금 본 그거, 읽기 전용에 계획만 세웁니다. **Auto** — `auto`. AI 분류기가 매 호출 안전성을 판단해 자동 승인/차단합니다.

**Auto Mode** 는 한 가지 주의할 점이 있어요. 모두에게 제공되지 않습니다. 플랜이 **Max, Team, Enterprise, API** 여야 하고 — *Pro 플랜에선 못 씁니다.* 모델도 Sonnet 4.6, Opus 4.6, Opus 4.7만 지원하고, Anthropic API provider 전용입니다. 화면에 Auto 모드가 안 보이는 분이 있으면 — *내 플랜에선 못 쓰는 거구나* 라고 안내해드리세요.

한 사례 보여드릴게요. 이 강의 레포 작업 중에 `git push origin main` 시도했더니 — Auto Mode 분류기가 *"기본 브랜치 보호"* 사유로 차단했어요. 사용자가 따로 명시한 후에야 통과했습니다. AI가 *내 명령을 거절할 수도 있다* 는 걸 처음 보는 순간이죠.

언제 어느 모드냐 — 새 작업은 항상 *Plan Mode*, 검토 후 *Auto-Accept* 로 전환, 위험한 작업은 *Normal로 회귀*. 이 패턴 기억해두세요.

## 12 — `12-one-session-one-feature.html` · 한 세션 = 한 피처

## 16 — `16-tdd-loop.html` · TDD 루프 — 한 입씩 자주

비유 하나. **요리** 예요.

좋은 셰프는 — 재료 손질하고, *한 입 맛보고*, 간 조절하고, 또 *한 입 맛보고*, 간 조절합니다. 마지막에 한 번에 맛보는 셰프와 — 한 입씩 자주 맛보는 셰프. *누가 더 안정적일지* 는 명확하죠.

코딩의 *TDD 루프* 가 정확히 같은 원리입니다. 작은 변경 → 테스트 돌리기 → 통과 확인 → 다음 변경 → 테스트 → 통과 → 다 됐으면 커밋. 문제가 생겨도 **마지막 커밋**으로 돌아가면 되니까 디버깅이 훨씬 쉬워져요.

## 17 — `17-hooks-teaser.html` · Hooks 맛보기 — 매번 입력 vs 한 번 박아두기

방금 본 *테스트·린트·커밋* 루프, 이걸 매번 입력하지 않고 *자동화* 해주는 게 **Hooks** 입니다.

*Before* — *"코드 수정 → 테스트 → 린트 → 커밋"* 네 줄을 매번 반복. *After* — 한 번 박아두면 파일 편집할 때마다 자동 실행. 0줄.

살아있는 사례 하나 보여드릴게요. 이 강의 레포에 깔린 `caveman-kor` hook 이에요. 모든 응답에 *"짧은 한국어로 답해라"* 를 자동 주입합니다. 결과 — **출력 토큰이 30~50% 절감** 됩니다. *"짧게 답해"* 를 매번 시키지 않고 — *hook 으로 박아둔 거죠.*

본격적인 4 이벤트와 JSON 구조는 3부에서 다시 봅니다. 지금은 *"매번 시킬 일을 한 번 박아둘 수 있다"* 까지만.

## 18 — `18-error-logs.html` · 에러 로그 통째로 + Escape 타이밍

디버깅의 철칙 하나 — 에러를 *해석해서 요약하지 마세요.* 통째로 던지세요.

나쁜 예 — *"Python 코드가 NameError 나는데 변수가 안 잡히는 것 같아."* 이렇게 사람이 해석을 넣으면 AI도 *그 해석에 끌려갑니다*. 좋은 예 — *Traceback 전체* 를 그대로 붙여넣는 거예요. AI는 스택 트레이스 분석에 능합니다.

`!` 접두어로 명령을 직접 실행할 수도 있어요. *`!npm test 2>&1 | tail -100`* 처럼 치면 — 명령 출력을 그대로 받아서 AI에게 분석시킬 수 있습니다.

또 한 가지 — **Escape 타이밍** 이에요. Claude가 답하기 전에 보여주는 *thinking 로그* 에서 *잘못된 가정* 이 보이면 — **즉시 Escape**. 잘못된 가정 위에 쌓이는 코드는 전부 쓸모없습니다. *초반에 잡는 게 핵심* 이에요.

## 19 — `19-docs-folder.html` · docs 폴더 운영 — 이 강의 레포가 살아있는 예시

CLAUDE.md 한 장은 200줄 제한이에요. 그래서 *진짜 깊은 정보* 는 — **별도 `docs/` 폴더** 에 둡니다.

이 강의 레포가 살아있는 예시예요. `docs/` 안에 `concepts/`, `techniques/`, `phenomena/`, `tools/`, `people/`, `methodologies/` 폴더가 있고요. CLAUDE.md엔 `@docs/tools/Claude-Code.md` 같은 *참조만* 두고, 필요할 때만 *lazy load* — 진짜 lazy load 두 가지 중 하나 — 되도록 만듭니다.

비유로 풀면 — 회사의 *팀 위키*와 *팀 라이브러리* 관계예요. 위키는 *목차*, 라이브러리는 *전체 문서*. 매번 라이브러리를 통째로 외울 필요 없어요.

이게 *문서 주도 개발* 의 가장 작은 사례입니다. 코드 짜기 전에 *문서 먼저*. 결정과 설계가 *영속 가능한 형태*로 남고, 새 세션이나 새 동료가 *같은 컨텍스트*에서 시작할 수 있게 만듭니다.

## 23 — `23-skills-structure.html` · Skills — 업무 매뉴얼

1강에서 *템플릿 라이브러리* 라고 봤죠. 이제 *구조* 를 풀어봅니다.

위치는 `.claude/skills/` 폴더 안. 스킬 이름의 폴더를 만들고 그 안에 `SKILL.md` 한 장을 둡니다. 필요하면 `template.md`, `examples/` 같은 보조 파일도요.

**2단계 lazy loading** 이 핵심이에요. 평소에는 *이름과 description* 만 컨텍스트에 로드돼요. 50~100바이트. *호출 시* 에만 본문과 참고 파일이 전부 로드됩니다. 그래서 평소엔 가볍게, 쓸 때만 전체. CLAUDE.md나 MCP가 *항상 로드* 되는 것과 결정적 차이입니다.

`SKILL.md` 의 frontmatter에 *사용자가 쓸 법한 표현 3개 이상* 을 description에 박아두면 — Claude가 *자동으로 트리거* 합니다.

## 24 — `24-skill-live.html` · 사전 작성한 Skill 보여주기

라이브 생성은 *너무 많은 변수* 가 있어요. 플러그인 설치, 대화형 응답, 파일 시스템, 권한. 비개발자 첫 진입에 위험합니다. 그래서 *진행자가 사전 작성한 작은 skill 한 장* 을 화면으로 보여드릴게요.

이 강의 레포에 깔린 `presentation_slides` skill 입니다. 폴더 안에 `SKILL.md` 한 장이 있고요 — frontmatter, 디자인 시스템 안내, HTML 보일러플레이트, 절차가 들어가 있어요. 호출은 `/presentation_slides` 한 줄. *모든 슬라이드 디자인 결정과 절차* 가 한 장에 박혀있어서 매번 처음부터 안내할 필요가 없어요.

**함께 입력 세 번째** — 선택입니다. `/skills` 한 번 쳐서 자기 PC의 skill 목록 확인해보세요. *내 PC엔 아직 없구나, 만들면 되겠네* 까지만 — 라이브 *생성* 은 다음 강의에서 직접 만들어봅니다.

## 25 — `25-sub-agents.html` · Sub-Agent — 회사의 부서

1강에서 *회사의 부서* 라고 본 그것. 메인 Claude가 PM이고, 서브에이전트들이 디자인팀·개발팀·QA팀입니다.

Claude Code에 내장된 5종 중에서 — 가장 많이 쓰는 세 가지만 짚어요. **Explore** — *코드 어디에 있나* 탐색 전담, Haiku 모델로 빠릅니다. **Plan** — *어떻게 짤지 먼저 보자*, 계획만 세우는 전담. **General-purpose** — *탐색하고 고치고 테스트까지* 다단계 작업.

각자 *자기 컨텍스트 창, 시스템 프롬프트, 도구 권한* 을 갖습니다. 메인 Claude는 *결과 요약만* 받아요. 그래서 **메인 컨텍스트 오염 방지** 가 핵심 가치입니다.

두 개 더 있어요 — **Bash** (별도 컨텍스트에서 명령 실행)와 **Claude Code Guide** (Claude Code 기능 Q&A). 디테일은 `docs/tools/서브에이전트.md` 보세요.

언제 쓰느냐 — *대량 출력*이 메인을 오염시킬 작업, 예를 들면 전체 테스트 로그나 47개 파일 검색. *직접 하기 vs 심부름 시키기* 판단입니다.

## 26 — `26-hooks.html` · Hooks — 자동화 엔진 본격

비유 하나. **집안의 센서들** 같아요. 문 열면 자동으로 불 켜지죠 — 모션 센서. 차 시동 걸면 안전벨트 경고가 뜨고요 — 시트 센서. **이벤트 → 자동 액션** 입니다. Hooks는 Claude Code 안에서 같은 일을 합니다.

18번에서 *맛만* 봤죠. 이제 본격으로 — 이벤트 종류에 따라 자동화 성격이 달라집니다.

자주 쓰는 네 가지부터. **PreToolUse** — 도구 호출 직전, *"이거 진짜 실행해도 돼?"* **PostToolUse** — 도구 실행 직후, *"끝났네, 자동으로 lint·포맷"*. **Notification** — 응답 대기 시 알림. **Stop** — 턴 종료 시 정리.

그 외에도 *UserPromptSubmit, SessionStart, SessionEnd, PreCompact, SubagentStop* 등 다양한 이벤트가 있습니다. 한 가지 주의 — Hook 실행 중에는 Claude가 *멈춰서 기다립니다.* timeout 꼭 설정하고, 무거운 작업은 *백그라운드* 로 보내세요.

## 27 — `27-hooks-before-after.html` · Hooks 살아있는 사례 — caveman-kor

이 강의 레포에 실제로 깔린 hook 하나를 보여드릴게요. 이름이 **caveman-kor** 입니다.

`UserPromptSubmit` 이벤트에 박힌 hook이에요. 모든 prompt 직후 자동으로 *"짧고 한국어로, 군더더기 빼고 답해라"* 한 줄을 주입합니다. 위치는 `~/.claude/settings.json` 입니다.

지난 6개월 실측 결과 — 출력 토큰이 **30~50% 감소**, 응답 속도 체감 30% 빠름. 설치 비용은 `settings.json` 한 블록, 5에서 10줄 정도면 끝입니다.

비유로 풀면 — *회사의 기본 안내문* 같은 거예요. 신입 사원에게 매번 *"보고서는 한 장으로, 결론 먼저, 출처 명시"* 직접 말할 필요 없이 — *팀 가이드*에 한 번 박아두면 자동으로 따라오죠. Hook은 그 *팀 가이드*를 Claude Code 안에서 자동으로 들이미는 장치입니다.

1강에서 본 *하니스 = 모델 주변 환경 전체* 의 **가장 작은 단위 사례** 예요. 매일 신경 쓸 일을 *한 줄로 박는다* — 그게 핵심입니다.

## 28 — `28-mcp-decision.html` · MCP + 결정표 — 언제 무엇을 쓰나

마지막 부품 — **MCP**, *Model Context Protocol* 입니다.

비유는 **AI 통합의 USB-C** 예요. Anthropic이 2024년 11월에 공개했고, 2025년 12월에 Linux Foundation 산하 *Agentic AI Foundation* 에 기증해서 — *Kubernetes·PyTorch와 같은 중립 인프라* 로 격상됐습니다.

옛날 휴대폰 충전기가 회사마다 달랐죠. 노키아 핀, 삼성 24핀, 애플 30핀, 마이크로 USB, 미니 USB. USB-C 하나로 정리된 것처럼 — 같은 일이 LLM-도구 사이에서 일어난 거예요. GitHub, Figma, Slack, Google Drive를 *모델마다 다른 코드*가 아니라 — **한 가지 형식**으로 LLM에 연결합니다.

토큰 절약 팁 하나 — Notion·Linear·Atlassian 같은 MCP는 *도구 설명만으로* 수천 토큰을 먹습니다. 안 쓰는 건 `/mcp` 에서 비활성화하세요.

**결정표** 하나 보여드릴게요. *언제 무엇을 쓰나*. 프로젝트 *전체*에 항상 적용할 규칙은 **CLAUDE.md**. 이 폴더만 적용은 **Subdirectory CLAUDE.md**. 자주 반복하는 절차는 **Skills**. 이벤트마다 자동 실행은 **Hooks**. 외부 서비스 연결은 **MCP**. 대량 출력 격리는 **Sub-Agent**. 이 표 한 장이 *어디에 박을지* 판단 기준입니다.
