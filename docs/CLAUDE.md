# /docs 작성 가이드

이 파일은 `docs/` 안에서 문서를 추가·수정할 때 따라야 할 규율을 모아둔다. 루트 `CLAUDE.md`(레포 전체)에서 위임받은 *문서 작성 디테일*을 담는 자리다. 새 문서 시드를 짜기 전에 반드시 읽는다.

---

## 1. 카테고리 결정 트리

새 개념이 나왔을 때 어디에 넣을지 헷갈리면 아래 순서로 판단한다.

```
한 시대를 가르는 패러다임/모델 종류        →  concepts/
   (LLM, Transformer, Tokenization, Tool-Use, Agentic-Engineering …)

실무에서 적용하는 *방법론*                  →  techniques/
   (Prompt/Context/Harness Engineering, RAG, 문서 주도 개발, Orchestration)

관찰되는 *문제·한계*                         →  phenomena/
   (환각, Sycophancy, Lost-in-the-Middle, 비결정성, Knowledge Cutoff)

특정 *인물*                                  →  people/
   (Andrej Karpathy …)

사용 가능한 *도구·제품*                      →  tools/
   (Claude Code, MCP, CLAUDE.md, Skills, Hooks, oh-my-claudecode …)

레포 자체에 적용되는 *행동 규율* 파일         →  rules/
   (coding-guidelines.md — 강의 시연용 실파일)

전·후 *개발 방법론*                          →  methodologies/
   (워터폴, 애자일, 하이퍼-워터폴)
```

위 7개에 안 맞으면 **새 폴더를 만든다.** 단 — 새 폴더를 만들기 전에 두 가지 점검:

1. 기존 카테고리 정의를 *살짝 넓히는* 것으로 해결 가능한가? (가능하면 새 폴더 안 만든다)
2. 같은 폴더에 들어갈 후보가 *2개 이상* 있는가? (1개뿐이면 기존 카테고리에 둔다)

후보 카테고리 예시: `models/` (GPT/BERT/Claude 같은 구체 모델), `companies/` (Anthropic, OpenAI), `events/` (시스템 프롬프트 유출 같은 사건).

## 2. 3-조건 서사 (절대 위반 금지)

모든 *개념·기법·도구* 문서는 다음 3-조건을 *순서대로* 따른다. 이걸 어긴 문서는 강의 청중(비개발자)에게 *그림을 그려주지 못한다*.

1. **왜 "{이름}"이라고 부르나** — 인물·연도·어원
2. **그전에는 어떤 문제가 있었나** — 등장 직전의 한계 + *비개발자용 일상 비유 1개* 필수
3. **어떻게 해결했나** — 변화 + 강의에서 어느 흐름과 연결되는지 (회차·슬라이드 번호는 적지 말 것 — §6 참조)

`phenomena/` 같은 *현상* 문서는 1·2가 살짝 다를 수 있다 (예: "왜 이 이름이 붙었나" → "이 현상이 처음 보고된 사건"). 큰 흐름은 동일하게 유지.

---

## 3. 예시 유형 — 최소 2개 이상

**왜 이게 강제 규칙인가**: 1회 발표 결과, 비개발자 청중이 *"정의는 알겠는데 그림이 안 그려진다"* 는 반응이 다수였다. 정의·역사만으로는 부족하고, **여러 각도의 예시**가 있어야 청중이 자기 일상에 매핑한다.

| 유형 | 설명 | 필수 여부 |
|---|---|---|
| **일상 비유** | 비전공자가 즉시 떠올릴 수 있는 일상 상황 (요리, 외국어 대화, 길찾기 등) | **필수 ≥ 1개** |
| **Before/After** | 잘못한 예 vs 잘한 예 — 입력·코드·결과를 나란히 | 적용 가능 시 권장 |
| **실제 사례·사건** | 뉴스, 논문 실험, 회사 케이스, 출시 일자 등 검증된 실사례 (반드시 WebSearch로 검증) | 적용 가능 시 권장 |
| **라이브 시연 가능한 예시** | 강의 중 즉석 재현 가능한 데모 (예: strawberry-R 카운팅, 같은 질문 3회 반복) | 가능한 경우 우선 |

**모범 템플릿** (새 문서 작성 시 반드시 참고):
- `concepts/Tokenization.md` — 라이브 시연 + 일상 비유 + Before/After 모두 포함
- `phenomena/Lost-in-the-Middle.md` — 실제 논문 수치 실험
- `tools/Claude-Code-Hooks.md` — `caveman-kor` 같은 실제 적용 사례

## 4. 데이터 신선도 체크

수치·연도·인물·모델 버전·제품명 같은 **시점 의존 사실**은 작성 시점 기준으로 검증한다.

- 새 문서를 쓰거나 기존 문서를 수정할 때, **현재 날짜(today's date) 기준 1년 이상 오래된 데이터**가 본문에 있다면 WebSearch로 재확인하고 가능한 한 최신 사실로 갱신한다.
- *"최근", "현재", "올해", "요즘"* 같은 시점 의존 단어는 **절대 날짜로 치환**한다.
  - ❌ "최근 출시" → ✅ "2026년 4월 출시"
  - ❌ "현재 표준" → ✅ "2026년 5월 기준 표준"
- 보존 가치 있는 과거 사실은 컨텍스트를 잃지 않게 표기:
  - "(2025년 시점) GPT-4가 표준 → (2026년 5월 기준) GPT-5 / Claude 4.7 라인업"

특히 *2026년 1월 이후* 이벤트는 강의 직접 인용 가능성이 높으니 **반드시 WebSearch 1회 이상**으로 확인 후 인용.

## 5. 새 문서 추가 시 체크리스트

문서 한 개를 새로 쓸 때 *반드시 같이* 손대야 할 곳이 있다. 누락하면 인덱스가 깨진다.

- [ ] 본문 — 3-조건 서사 + 예시 ≥ 2개 + 데이터 신선도
- [ ] 본문 하단 **꼬리에 꼬리 (관련 개념)** 섹션 — 다른 docs로 양방향 링크
- [ ] 본문 하단 **출처** 섹션 — WebSearch로 검증한 1차/2차 자료 URL
- [ ] `docs/README.md` 인덱스에 한 줄 추가 (해당 카테고리 아래)
- [ ] *연결 문서들*의 "꼬리에 꼬리" 섹션에도 새 문서로 가는 링크 추가
- [ ] 강의 흐름의 어느 *개념적 위치*에 들어가는지 본문에 명시 (예: *"하니스 정의 섹션의 가장 진한 사례"*) — **회차 번호·슬라이드 번호는 쓰지 않는다** (§6)

## 6. 강의 스크립트와 docs 매핑

각 강의 폴더의 `script.md` 와 docs는 분리돼 있지만 — 강의자가 청중 질문을 받았을 때 즉시 펼쳐볼 수 있어야 한다. *스크립트 한 섹션 = docs 한 문서 이상* 매핑을 유지한다.

매핑 작업 시:

1. 스크립트에서 *처음 등장하는 용어*에 해당하는 docs 문서가 있는지 확인
2. 새로 등장한 용어가 docs에 없으면 *카테고리 결정 트리* (§1) 따라 새 문서 시드
3. 스크립트가 수정될 때 — docs와 어긋난 부분 있는지 점검 (특히 *연도·수치·이름*)

**중요 — 슬라이드 번호·강의 회차 의존 금지**: docs는 강의 스크립트가 *재작성될 때마다 같이 안 깨지도록* 자족적으로 작성한다. *"N강 #M"* 같은 회차·번호 참조는 docs에 넣지 않는다. 개념 자체로 읽혔을 때 완결돼야 한다.

**스크립트 파일은 사용자 영역**: 각 강의 폴더의 `script.md` 는 사용자가 직접 다듬는 파일. AI가 함부로 고치지 않는다. docs는 자유롭게 수정·추가 가능.

---

## 7. 비개발자 톤 — 풀어 설명

docs는 강의자가 읽지만, *강의자가 그대로 청중에게 옮길 수 있어야* 한다.

- 전문 용어는 **한 번 풀어서 설명한 뒤** 사용. 예: *"하니스(harness, 영어로 마구 — 말의 힘을 마차에 묶어주는 도구)"*
- 영어 용어는 *처음 등장*할 때만 한국어와 병기, 이후엔 한국어 위주.
- 코드 블록은 *주석으로 한국어 설명* 추가. 비개발자가 코드 자체를 못 읽어도 *주석으로 흐름은 따라올 수 있게*.
- 문장이 길어지면 — 끊는다. 강의장에서 *말로 옮길 수 있는 길이*가 기준.

---

## 8. 작업 시 원칙 (루트 CLAUDE.md에서 가져옴)

- **출처 없이 사실 단정 금지**: 연도·인물·논문 같은 디테일은 반드시 WebSearch 검증 후 기재.
- **상호 링크 유지**: 한 문서를 추가/수정하면 README 인덱스와 관련 문서의 *꼬리에 꼬리* 섹션을 함께 갱신.
- **HTML 슬라이드는 별도 작업**: docs 작성과 슬라이드 작성은 분리. 사용자가 명시적으로 슬라이드 작업을 요청할 때만 슬라이드 진입.
- **루트 `CLAUDE.md`가 우선**: 충돌 시 루트의 지시가 이긴다. 이 파일은 *디테일 위임*을 담는다.
