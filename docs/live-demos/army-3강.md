# army/3강 — 단계별 prompt 묶음

> 강의자 운영 메모: 3강은 *5단계 워크플로우 라이브 빌드*. 각 단계 prompt + 예상 시간 + 위험관리.
> 빌드 주제는 강의자가 정함. 본 문서는 *주제 무관 골격* — 빌드 주제 정한 뒤 `[주제]` 자리만 채우면 그대로 사용 가능.

---

## 사전 준비

- 빈 폴더 1개
- 강의자 환경: Claude Code 정상 작동 + 필요한 MCP 연결 확인
- 청중 환경: 같은 빈 폴더 1개 준비 권장 (*따라하기* 모드)

---

## 단계 1 — 뼈대 (10분)

```text
[빈 폴더에서]
$ claude

> 나는 [빌드 주제]를 만들고 싶어.
  핵심 기능 3개:
  1. [A]
  2. [B]
  3. [C]

  사용자는 [대상 사용자]야.
  먼저 폴더 구조부터 같이 잡자. 뭐부터 만들까?
```

**예상 흐름**:
- Claude가 *기본 폴더 구조* 제안 (예: `src/`, `docs/`, `tests/`, `package.json`)
- 강의자 *"이렇게 갈까?"* 합의
- 폴더 골격 생성

**위험관리**:
- 청중도 따라하다 *각자 다른 주제*로 가도 됨 (가계부, 회의록 요약기, 명함 OCR 등)
- 강의자는 *자기 주제 진행 상황*을 화면에 띄우고 청중은 *각자 진행*

---

## 단계 2 — 4종 기획 문서 (25분)

각 문서당 ~6분.

### PRD
```text
> docs/PRD.md 초안 만들어줘.
  - 대상 사용자
  - 핵심 기능 3개 (단계 1에서 합의한 것)
  - 성공 기준 3개 (어떻게 작동하면 성공인가)
  - 비기능 요구 (속도·접근성·언어)
```

### ARCHITECTURE
```text
> docs/ARCHITECTURE.md 만들어줘.
  PRD를 바탕으로:
  - 폴더 구조
  - 모듈 분할
  - 데이터 흐름 (입력 → 처리 → 출력)
```

### ADR
```text
> docs/ADR/0001-기술스택-선정.md 만들어줘.
  결정 기록 형식 (Context, Decision, Consequences)으로:
  - 왜 [언어/프레임워크]를 골랐는가
  - 다른 후보 2개 + 탈락 이유
```

### UI_GUIDE
```text
> docs/UI_GUIDE.md 만들어줘.
  - 다크 테마 / 미니멀 / 한국어 우선
  - 색상 팔레트 5개
  - 폰트 (Noto Sans KR + JetBrains Mono)
  - 간격 단위 (4px / 8px / 16px ...)
```

**강의자 체크포인트**:
- 각 문서 만들어진 직후 *"이거 틀린 부분 있나?"* 1번씩 던지기 → 시코펀시 회피
- *"이 PRD에서 빠진 게 있을까? 거꾸로 검토해봐"*

---

## 단계 3 — 하니스 설계 (30분)

### CLAUDE.md
```text
> /init

(자동 생성된 CLAUDE.md를 강의자가 직접 편집)
```

**추가할 핵심 항목** (강의자가 시범):

```markdown
# [프로젝트 이름]

## CRITICAL — 반드시 지킬 것
- TDD: 모든 새 기능은 테스트 먼저 작성
- 응답은 한국어로
- 환각 방지: 본 적 없는 라이브러리·함수는 *반드시 검색·확인*

## Karpathy 4원칙
1. Think — 코드 짜기 전 의도부터 적기
2. Simple — 추상화 금지, 직선적으로
3. Surgical — 요청한 것만, 옆 코드 건드리지 말기
4. Goal-Driven — PRD 기준으로 항상 자기 검열

## 코드 스타일
- 함수 한 개 = 한 가지 일
- 주석은 *왜*만 적기
```

### 하니스 자동 설치
```text
> /harness

(강의 자료에 포함된 커스텀 skill — 사전 배포 필요)
```

또는 **수동 단계**:

```text
> .claude/hooks/ 폴더 만들어서 다음 hook 3개 설치해줘:

  1. PostToolUse — 코드 수정 후 자동 테스트 실행
  2. UserPromptSubmit — 한국어 출력 가이드 주입 (caveman-kor 스타일)
  3. PreCommit — ADR 갱신 알림

  각 hook은 .sh 파일로, .claude/settings.json에 등록까지.
```

**강의자 코멘트**:
- 1·2강에서 본 단어 *환각·비결정성·시코펀시* 가 *결정론적 hook*으로 *자동 차단*되는 모먼트
- *"이거 한 번 깔아두면 매번 빠뜨리지 않습니다"*

## 단계 4 — 자동 실행 (30분, 강의자만 시연)

```bash
[강의자 시연]
$ claude -p "프로젝트 PRD를 읽고 Phase 1~5를 순서대로 진행해.
  각 Phase 종료 시 진행 docs를 반드시 갱신하고
  Phase 시작 시 이전 Phase docs를 다시 읽어 컨텍스트 복원해."
```

**Phase 표준 흐름**:
1. **Phase 1 — 뼈대 코드** (5분): 기본 파일 골격, import 정리
2. **Phase 2 — 핵심 모듈** (10분): 주요 기능 1개씩 구현
3. **Phase 3 — 테스트 작성** (5분): 각 모듈 단위 테스트
4. **Phase 4 — UI 적용** (5분): UI_GUIDE 따라 화면 구성
5. **Phase 5 — 통합 검증** (5분): 전체 빌드 + 테스트 통과 확인

**강의자 라이브 코멘트** (헤드리스 진행 중 *기다림*을 채우는 톤):

| 시점 | 짚어줄 단어 |
|---|---|
| Phase 1 시작 시 | *지식 컷오프* — 새 Claude는 우리 프로젝트를 모름. 그래서 docs를 읽음 |
| Phase 2 진행 중 | *컨텍스트 엔지니어링* — 새 Claude가 ADR을 읽고 결정 근거를 따라감 |
| Phase 3 (테스트) | *비결정성* — 결과를 *결정론적으로* 검증하기 위한 테스트 |
| Phase 4 (UI) | *프롬프트 엔지니어링* — UI_GUIDE의 Few-shot 효과 |
| Phase 5 끝 | *서브에이전트* — 통합 검증을 별도 에이전트로 띄움 |

**위험관리**:
- 헤드리스 진행이 *예상보다 오래 걸림* → 강의자는 1·2강 단어 복습으로 *흥미 유지*
- *완전 실패* 시 → 강의 자료로 미리 *완성된 결과물*을 백업 폴더에 두고 *"이렇게 나옵니다"* 로 전환

## 단계 5 — 리뷰 (15분)

```text
> /review

(자동 검토 모드)
```

**리뷰 체크리스트** (1강 단어로 정리):

| 1강 단어 | 확인 prompt |
|---|---|
| 환각 | *"이 코드에 import한 라이브러리가 실제로 존재하는지 npm/pypi에서 확인해줘"* |
| 시코펀시 | *"이 코드에서 *작동한다고 자신만만하게 적힌 부분*이 진짜 작동하는지 거꾸로 검증해줘"* |
| 비결정성 | *"같은 입력에 같은 출력이 나오는지 테스트 3회 실행"* |
| 지식 컷오프 | *"이 코드의 라이브러리 버전이 2026-05 기준 최신인지 확인"* |

**문제 발견 시 — 부분 재실행**:
```text
> [문제 부분 코드 인용]
  여기 환각 의심돼. 실제 함수가 존재하는지 검색해서 다시 확인하고
  필요하면 *이 부분만* 다시 작성해줘.
```

**중요**: *전체 재실행 금지*. 잘 작동하는 다른 부분까지 깨질 수 있음.

## 마무리 — AI Adoption Spectrum 자기 진단

청중에게 *4단계 표*를 보여주고 *지금 어디인지* 손들기 / 자기 평가 시켜보기.

```text
| 단계 | 한 줄 |
|---|---|
| L1 — AI-aware | 가끔 ChatGPT |
| L2 — AI-enabled | 업무에 얹어서 쓰는 중 |
| L3 — AI-maximalist | "AI 어디에나" 신념 단계 |
| L4 — AI-native | AI 중심 균형 설계 |
```

**메시지**: 오늘 *5단계 워크플로우*를 본 게 정확히 L4의 *훈련*. *AI에 다 맡기자*가 아니라 *언제 사람이 판단하고 언제 위임할지 설계*하는 게 native.
