# 서브에이전트 (Subagent)

> 큰 작업을 *특정 분야의 작은 AI 동료들*에게 나눠 맡기는 시스템. 코드 리뷰는 리뷰 전문 에이전트, 테스트는 테스트 전문 에이전트. **각 서브에이전트는 자기만의 컨텍스트 창·시스템 프롬프트·도구 권한을 가진 분리된 Claude 인스턴스**.

## 1. 왜 "서브에이전트"라고 부르나

- **Sub-agent**: *"하위 에이전트"*. 메인 에이전트(=Claude Code 본체)가 **자기 일을 일부 위임**할 수 있는 작은 에이전트들.
- **언제 등장했나**: Claude Code의 `/agents` 명령으로 정식 노출된 시점은 2025년 후반. Anthropic 공식 *"Subagents in the SDK"* 문서가 2025-12 시점에 갱신됨.
- 비유: **회사의 부서**. 메인 에이전트는 *프로젝트 매니저*, 서브에이전트들은 *디자인팀·개발팀·QA팀*. PM은 *"이건 디자인팀에 보내자"* 처럼 위임한다.

## 2. 그전에는 어떤 문제가 있었나

LLM 단일 호출은 **단일 컨텍스트 창**에서 모든 작업을 처리해야 했다.

- 큰 코드베이스 탐색 + 보안 리뷰 + 리팩토링 + 테스트 작성을 *한 세션*에 시키면 → 컨텍스트 창이 빠르게 차고, *Lost in the Middle* 현상으로 정확도 하락
- 한 가지 *큰 작업*이 끝나면 메인 대화도 그 결과로 오염됨 (탐색 부산물·임시 가설 같은 *노이즈*가 메인 컨텍스트에 누적)
- 작업마다 *다른 도구 권한*이 필요한데 — 메인 에이전트는 *모든 권한*을 가지고 있어야 함 → 사고 위험

→ *"한 명이 모든 일을 해야 하는 직장"*과 같은 비효율.

**일상 비유**: 팀장 한 명이 영업·디자인·법무·QA를 *동시에* 처리하려는 회사. 머리가 너무 많은 일에 분산돼서 *어느 것도 잘 못한다*. 부서를 나누고 *전문 인력*에게 위임하는 게 정상이다.

## 3. 어떻게 해결했나

**각 서브에이전트는 독립된 Claude 인스턴스**로 동작:
- *자기만의 컨텍스트 창* — 메인 대화와 분리, 부산물이 메인을 오염시키지 않음
- *자기만의 시스템 프롬프트* — 특정 분야에 맞춘 행동 지침
- *자기만의 도구 화이트리스트* — 필요한 도구만 부여 (=권한 최소화)
- *자기만의 모델 선택* — 빠른 작업은 Haiku, 무거운 작업은 Opus

```text
[메인 Claude]
  ├── /agents code-reviewer     → 코드 리뷰만
  │     · 시스템 프롬프트: 보안·성능·가독성 점검
  │     · 도구: Read, Grep
  │     · 모델: Sonnet
  │
  ├── /agents debugger          → 버그 추적
  │     · 시스템 프롬프트: 재현 → 원인 → 수정
  │     · 도구: Read, Bash, Edit
  │     · 모델: Opus (깊은 추론)
  │
  └── /agents general-purpose   → 잡일·탐색
        · 빠른 검색·다단계 작업
        · 모델: Haiku (저비용)
```

### Claude Code 빌트인 서브에이전트 (2026 기준)

- **Explore**: 코드 탐색 전문 — *"이 함수 어디서 호출돼?"*, *"이 패턴이 다른 파일에도 있나?"*
- **Plan**: 구현 계획 설계 — 인터랙티브하게 사용자와 논의하며 step-by-step 작성
- **general-purpose**: 다목적 — 짧은 조사·여러 단계 작업

### 커뮤니티 확장

- VoltAgent 의 **awesome-claude-code-subagents** GitHub 레포: **100개 이상**의 전문 서브에이전트 (보안 감사, 리팩토링, 디자인 리뷰 등)
- `subagents.app`, `wshobson/agents` 등 마켓플레이스가 2025~2026년에 다수 등장

## Before/After — 서브에이전트 유무

| 작업 | 단일 에이전트 (Before) | 서브에이전트 활용 (After) |
|---|---|---|
| 큰 코드베이스에서 함수 정의 찾기 | 메인 컨텍스트에 결과 모두 누적 → 토큰 폭증 | Explore에 위임 → 요약만 메인으로 반환 |
| PR 리뷰 + 보안 점검 | 같은 프롬프트에서 둘 다 처리 | code-reviewer + security-auditor 분리 호출 |
| 비용 최적화 | 모든 작업이 Opus | 가벼운 작업은 Haiku 서브로 라우팅 |
| 권한 분리 | 메인이 모든 권한 보유 | 위험 작업 서브에만 좁은 권한 |

## 라이브 시연 가능한 예시

### 시연 1 — 메인 컨텍스트 보호
```bash
[메인 Claude Code]
> 이 레포 어디에서 'API_KEY'가 사용돼? Explore 시켜줘

[Claude]
  → Explore 서브에이전트 호출
  → 서브가 grep + 파일 읽기 수행
  → 메인엔 결과 요약만 반환:
    "총 3곳 사용. utils/auth.py:14, config/dev.py:8, ..."
```
청중에게: *"메인 대화는 깨끗하게 유지된다. 탐색 노이즈가 안 들어옴."*

### 시연 2 — 분업 위임
```bash
> 이 코드의 보안 취약점만 봐줘

[Claude]
  → security-auditor 서브에이전트 호출
  → 보안 전용 시스템 프롬프트 + 좁은 도구 권한
  → "SQL Injection 가능성 1건 발견" 리포트
```

## MCP vs 서브에이전트 — 헷갈리는 두 개념

| 구분 | MCP | 서브에이전트 |
|---|---|---|
| 다루는 것 | *외부 도구·서비스* 와의 연결 | *내부 작업의 분업* |
| 비유 | USB-C (외부 기기 연결 표준) | 회사 부서 (내부 분업) |
| 위임 받는 쪽 | 외부 시스템 (GitHub·Figma 등) | 다른 Claude 인스턴스 |
| 1강 위치 | #23 첫 번째 항목 | #23 두 번째 항목 |

둘은 **다른 축**이고 종종 *같이* 쓰인다 (예: 한 서브에이전트가 MCP를 통해 GitHub에 접근).
