# 워터폴 (Waterfall, 폭포수 모델)

> 요구분석 → 설계 → 구현 → 검증 → 배포를 **순서대로 한 번씩** 흘려보내는 전통적 소프트웨어 개발 방법론. AI 시대를 이해하려면 먼저 알아야 할 출발점.

## 1. 왜 "워터폴(폭포)"이라고 부르나

- **Waterfall**: 영어로 "폭포". 위에서 아래로 단계가 한 방향으로만 흘러내린다는 시각적 비유.
- **누가·언제**:
  - **1970년**, **Winston W. Royce**가 논문 *"Managing the Development of Large Software Systems"*에서 도식을 제시. 단계가 위→아래로 떨어지는 도표 모양 때문에 "waterfall"이라는 이름이 붙음.
  - 흥미로운 사실: **Royce 본인은 이 모델을 "위험하고 실패를 부른다(risky and invites failure)"고 비판하기 위해** 그 도표를 그렸다. 하지만 후대 사람들이 그 도표만 떼어내 "이상적 모델"로 받아들이면서 산업 표준으로 굳어졌다.
- **왜 한국어 "폭포수"**: 단계가 아래로 떨어진다는 시각적 비유를 그대로 직역.

## 2. 워터폴 이전에는 어떤 문제가 있었나

- 1960년대 이전 소프트웨어 개발은 **"코딩 먼저, 설계는 나중에"** 식 비공식 작업이 흔했다.
- 프로젝트 규모가 커지면서:
  - 요구사항이 모호한 채 시작 → 중간에 갈팡질팡
  - 설계 없이 코드 → 모듈 간 충돌
  - 테스트가 후순위 → 출시 후 결함 폭주
- **NATO Software Engineering Conference (1968)** 에서 *"software crisis"*라는 단어가 등장 — 소프트웨어가 비용·일정·품질 모두 통제 불가능한 상태였다.

**일상 비유**: 설계도 없이 집을 짓기 시작해서 3층 올린 뒤 "1층 기둥이 부족하네" 하고 깨닫는 상황.

## 3. 워터폴이 어떻게 해결했나

- **단계 분리 + 단계 게이트**: 각 단계가 끝나야 다음 단계로. 이전 단계 결과물(요구명세서, 설계서, 테스트계획서 등)이 다음 단계의 입력.
- **문서 중심**: 매 단계가 문서로 남는다 → 누구나 추적 가능, 책임 명확.
- **계획 우선**: 시작 전에 무엇을 만들지·언제 끝낼지를 정한 뒤 실행 → 예측 가능.
- 1970~1990년대 SI·국방·금융 등 **대규모 프로젝트의 표준**이 됨. 한국 공공 SI는 2010년대까지 워터폴이 사실상 강제.

## 워터폴의 한계 (애자일이 등장한 이유)

- **속도**: 요구분석 2주 + 설계 4주 + 구현 4주… 시장 변화에 못 따라간다. 끝났을 때 이미 트렌드가 지나간 경우가 다반사.
- **변경 비용**: 마지막 단계에서 요구가 바뀌면 모든 단계로 거슬러 올라가야 함.
- **고객과 거리**: 출시 직전에야 고객이 결과물을 본다 → "이거 우리가 원한 게 아닌데요"가 출시일에 터진다.
- **죽은 문서**: 코드와 문서가 어긋나기 시작하면 문서가 거짓말이 된다.

## 1강 강의 연결 포인트

- 워터폴은 **"사람이 모든 걸 사전에 정하고 사람이 모든 걸 만든다"**는 전제를 깐다.
- 바이브코딩이 등장한 직전까지의 두 패러다임(워터폴·애자일)이 모두 이 전제를 공유했다.
- AI가 게임 룰을 바꾸면서 — **하이퍼-워터폴** 같은 새 형태가 나타나는 출발점.
