# 하이퍼-워터폴 (Hyper-Waterfall, RHWP)

> **거시적으로는 워터폴, 미시적으로는 애자일.** AI 시대를 위해 설계된 1인+AI 개발 방법론. 워터폴의 규율과 애자일의 속도를 **둘 다 가지려는** 시도.

## 1. 왜 "하이퍼-워터폴"이라고 부르나

- **Hyper-(하이퍼-)**: "초(超)·과(過)·극도의" 라는 뜻의 그리스어 어원 접두사. "Hyperdrive(초고속 항행)", "Hyperlink(초연결)"처럼 **기존 개념을 한 단계 끌어올린다**는 뉘앙스.
- **Waterfall**: 워터폴 그대로 — 계획-설계-구현-검증-배포의 단계 분리.
- 합치면: **"극도화된 워터폴"** — 워터폴의 강점(규율·문서·계획)을 더 강하게 적용하되, AI의 기억 한계와 자기 교정 불가능을 보완하기 위한 장치를 추가한 형태.
- **누가·언제**:
  - 한국 개발자 **edwardkim**이 2026년 초에 정립.
  - **rhwp** (Rust 기반 한국어 HWP 파일 뷰어/에디터) 오픈소스 프로젝트의 *실증 기록*으로 발표.
  - 약 50일간 **1인 + Claude Code(AI) + 외부 기여자 9명**으로 100,000+ 라인 Rust 코드, 941+ 테스트, 729개 문서 생산.
  - 2026년 4월 4일 하루에 "기존 방식으로 2~3주 걸릴 작업"을 완료한 생산성 기록을 남김.

## 2. 그전에는 어떤 문제가 있었나

기존 세 방법론이 모두 한 발씩 부족했다:

| 방법론 | 강점 | 약점 |
|---|---|---|
| **워터폴** | 규율·계획·문서 | "**너무 느리다**" — 2주 요분 + 4주 구현으로 시장 대응 불가 |
| **애자일** | 속도·유연성 | "**문서 소실, 방향 불안정, 기술 부채 누적, 지식이 사람 머리에 갇힘**" |
| **바이브코딩** | 진입 장벽 제거 | "**계획·검증 없이 AI 제안 수락 → 이해 못한 코드는 고칠 수 없다**" |

edwardkim의 핵심 진단:
> *"규율(워터폴) 또는 속도(애자일)의 양자택일을 강요받았다. 이 둘 다 가질 수는 없었다 — **AI가 없던 시대에는**."*

즉, **AI가 등장하기 전에는 둘 중 하나를 포기**할 수밖에 없었다. AI가 그 트레이드오프를 깰 수 있게 됐다.

## 3. 어떻게 해결했나

### 거시 + 미시 이중 구조

- **거시 단계 (워터폴 규율)**: 프로젝트 수준에서 계획-설계-구현-검증-배포 순서 유지. 각 단계 게이트에서 문서/리뷰 강제.
- **미시 단계 (애자일 속도)**: 타스크(수 시간 단위) 안에서는 빠른 반복 사이클 — Plan → Generate → Review → Iterate.

### 3대 원칙

1. **구현 가치(목표)가 AI 컨텍스트에 유지되도록**
   → CLAUDE.md에 규칙 명시, 타스크 등록 시 목표 명확화, 컨텍스트가 길어지면 핵심 맥락 재주입.
2. **작업 통제권을 작업지시자(사람)가 유지해야만**
   → 계획은 AI가 제안하되 **사람이 승인**. 무엇을 만들지·순서·종료 시점은 사람이 결정.
3. **주기적으로 AI 컨텍스트 유지를 확인**
   → 대화가 길어지면 초반 맥락이 손실되므로 핵심 결정사항을 별도 문서로 남기고 재주입.

**통합 원리**: *"사람은 절대 생각을 멈추지 않는다."* — AI가 자동화한다고 해서 판단을 위임하지 않는다.

### 5단계 작업 흐름

```
1. 타스크 등록    GitHub Issue로 범위 정의
2. 수행 계획서    AI 작성 → 사람 검토/승인
3. 단계별 구현    AI 코드+테스트 → 단계별 보고서 → 검증
4. 피드백 반영    사람 피드백 작성 → AI 수정
5. 최종 보고      AI 결과 보고서 → 사람 승인
```

### 7폴더 문서 체계

| 폴더 | 역할 |
|---|---|
| `orders/` | 오늘 할 일 기록 |
| `plans/` | 수행/구현 계획서 |
| `working/` | 단계별 완료 보고서 |
| `report/` | 최종 결과 보고서 |
| `feedback/` | 인간의 교정 기록 |
| `tech/` | 발견한 기술 사항 |
| `troubleshootings/` | 함정 기록 (현재 21개) |

→ 핵심: **"AI의 기억 한계를 극복하고, 인간의 판단을 보존하는 시스템."**

### 결과 비교 (edwardkim의 실증치)

| 지표 | 바이브코딩 | 하이퍼-워터폴 |
|---|---|---|
| 기술 부채 | 축적 | **0에 수렴** |
| 재현 가능성 | 불가능 | **전 과정 추적 가능** |
| 문서량 | 거의 없음 | **729개 문서** |
| 의사결정 주체 | AI | **사람 (AI는 가속기)** |

## 1강 강의 연결 포인트

- 1강 메시지 **"User → Orchestrator"**의 가장 구체적 사례. AI를 부려쓰되 **결정권은 사람이 유지**.
- 1강 후반의 **3가지 축**(Prompt / Context / Harness Engineering)이 모두 결합된 실천 사례:
  - Prompt: 단계마다 명확한 지시
  - Context: CLAUDE.md + 7폴더로 컨텍스트 유지
  - Harness: 5단계 검증 루프
- 비개발자 청중에게 던질 메시지: **"AI가 빨라졌다고 계획·검토·문서를 빼는 게 아니라, AI 덕분에 그것까지 다 할 시간이 생긴다."**

## 시나리오 — 동일 프로젝트, 하이퍼-워터폴 식 진행

**프로젝트**: 사내 휴가 신청 웹앱 ([워터폴](워터폴.md#시나리오) / [애자일](애자일.md#시나리오) 시나리오와 비교)

```text
1일차 — 거시 계획 (사람 + AI)
  ├─ PRD: 휴가 신청 + 결재선 + 잔여 계산 한 번에 정의
  ├─ ARCHITECTURE: 폴더 구조·기술 스택·데이터 흐름
  └─ ADR-001: PostgreSQL 채택 이유

2일차 — 1차 미시 사이클 (AI 가속)
  ├─ /harness 실행 → Phase 5개 자동 분할
  ├─ Phase 1~5 무인 자동 구현 (헤드리스 claude -p)
  └─ 각 Phase 결과를 working/ 에 자동 보고

3일차 — 검증·피드백 (사람 결정)
  ├─ /review 로 규칙 위배 자동 점검
  ├─ 사람 피드백 → feedback/ 폴더에 기록
  └─ AI가 피드백 반영 → 재구현

4일차 — 배포 + 문서화
  ├─ 최종 결과 보고서 report/ 에 저장
  └─ troubleshooting/ 에 발견된 함정 기록 → 다음 프로젝트 자산화

[강점] 워터폴의 문서 + 애자일의 속도 + AI의 가속 = 4일에 끝
[조건] 사람의 판단을 매 단계 *유지* — 자동화에 위임 ❌
```

3가지 방법론을 한 표로:

| 방법론 | 같은 프로젝트 소요 | 문서량 | 변경 비용 |
|---|---|---|---|
| 워터폴 | 9주 | 두꺼움 | 후반 폭증 |
| 애자일 | 5주 (반복) | 거의 없음 | 평탄 |
| **하이퍼-워터폴** | **4일** | **풍부** | **평탄 + AI가 흡수** |

## 산업 트렌드 맥락

하이퍼-워터폴만 외로운 시도가 아니다. 2025~2026년에 같은 방향의 움직임이 동시다발로 나타나고 있다:

- **"Agile planning, waterfall execution"** — 짧은 사이클은 유지하되, 사이클마다 **상세 사전 설계**를 늘리는 트렌드.
- **Tony Alicea의 "Cascades"** — *"rapid AI-driven micro-waterfall development cycles"*. 작은 워터폴을 빠르게 흘려보내는 방식.
- **Amazon Kiro** — "Vibe 모드"와 "Spec 모드"를 명시적으로 분리한 AI 코딩 환경. 후자는 PM처럼 사용자에게 사전 계획을 강제.
- **AI 시대 사전 설계 시간 3배 증가** — PRD가 *"법적 계약처럼 읽힌다"*는 지적까지 등장.

→ 결론: 하이퍼-워터폴은 **개인의 발명**이지만 **산업 전체의 흐름**과 정확히 맞물려 있다.
