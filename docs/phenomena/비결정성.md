# 비결정성 (Non-Determinism / Stochasticity)

> 같은 질문을 두 번 해도 **매번 답이 조금씩 다르다**. *"AI는 한 번 잘 나온 답을 두 번째도 똑같이 줄 거라고 믿으면 안 된다"*의 정체.

## 1. 왜 "비결정성"이라고 부르나

- **Deterministic(결정적)** = 같은 입력이면 항상 같은 출력. 계산기가 그렇다. `2+2`는 언제나 `4`.
- **Non-deterministic(비결정적)** = 같은 입력에도 출력이 달라질 수 있음. 주사위, 일기예보, 그리고 **LLM**.
- LLM이 비결정적인 가장 큰 이유는 **샘플링(sampling)** — 다음 단어를 고를 때 가장 확률 높은 단어 하나만 뽑는 게 아니라 **확률 분포에서 무작위로 뽑기** 때문.
- **Temperature(온도)**: 이 무작위성의 강도를 조절하는 손잡이. 높으면 다양하고 창의적, 낮으면 안정적이고 반복적.
- **한국어 번역어**: 통계·확률 분야의 "확률적/추계적(stochastic)" 또는 "비결정적(non-deterministic)"을 그대로 차용. 정착된 단축 표현은 없고, 일반인용으로는 *"같은 질문에 매번 다른 답"* 이라고 풀어 말하는 편이 빠르다.

## 2. 그전에는 어떤 문제가 있었나

소프트웨어는 본래 **결정적**이었다. 같은 입력에 같은 출력이 나오기 때문에:
- 버그를 재현해서 고칠 수 있다
- 테스트를 한 번 통과하면 다시 통과한다
- 한 사용자에게 잘 동작한 게 다른 사용자에게도 잘 동작한다

LLM 등장 후 이 가정이 깨졌다:
- 같은 프롬프트 두 번 → 답 두 개가 다름
- 어제 잘 됐던 프롬프트가 오늘 잘 안 됨
- 데모 때 한 번 잘 나온 답을 다시 보여주려는데 안 나옴
- *"이거 진짜 맞아?"* 한 번 의심하면 답이 흔들림 (이건 [Sycophancy](Sycophancy.md) — 별개 메커니즘)

**일상 비유**: 즉흥 연주하는 재즈 연주자. 같은 곡을 부탁해도 매번 다른 솔로가 나온다. 멜로디는 비슷하지만 디테일은 그때그때 달라진다.

## 3. 왜 이런 일이 일어나나 (메커니즘)

### (1) 샘플링 자체의 확률성
- 모델은 다음 단어 후보별로 확률을 계산: `the (40%), a (25%), this (15%), ...`
- **temperature > 0** 이면 확률에 비례해 무작위 추출 → 매번 다른 토큰
- 한 토큰만 달라져도 그 뒤 문장이 줄줄이 다른 방향으로 가버림

### (2) Temperature = 0 이어도 완전 결정적이지 않다
업계의 오해: *"temperature=0이면 가장 확률 높은 단어만 뽑으니 항상 같은 답이 나온다."* 실제로는 그렇지 않다.

**Horace He 외 (Thinking Machines Lab, 2025년 9월)** 논문 *"Defeating Nondeterminism in LLM Inference"*가 정리한 진짜 원인:

- **부동소수점 오차**: GPU의 행렬 연산은 부동소수점 반올림이 발생. 같은 계산도 실행 순서에 따라 미세하게 다른 값.
- **GPU 병렬 처리 순서**: 수천 개 코어가 동시에 더할 때 누가 먼저 끝나느냐는 매번 달라짐.
- **동적 배칭(dynamic batching)** — *가장 큰 원인*: 프로덕션 LLM 서버는 여러 사용자 요청을 묶어서 한 번에 처리. **내 프롬프트가 어떤 다른 요청들과 묶이느냐**가 매번 다름 → 행렬 연산 형태가 달라지고 → 결과 토큰이 달라짐.

### (3) 그래서 API 사용자는?
- OpenAI/Anthropic의 `temperature=0` 옵션도 "거의 같다" 수준이지 "완전 같다"는 보장 안 됨
- `seed` 파라미터를 추가로 줘도 위 (2)의 효과 때문에 미세 차이 발생 가능

## 환각·시코펀시·비결정성 비교 (1강 강의용 정리)

세 현상을 청중이 자주 헷갈리므로 한 표로:

| 현상 | 한 줄 정의 | 예시 |
|---|---|---|
| **환각** | 모르는 걸 그럴듯하게 지어냄 | 존재하지 않는 논문·인용·통계를 자신만만하게 답함 |
| **Sycophancy** | 알면서도 사용자 압박에 답을 뒤집음 | "이거 틀리지 않아?" → "맞아 너가 정확했어!" |
| **비결정성** | 같은 질문에 매번 다른 답 | 같은 프롬프트를 두 번 보내면 답 두 개가 다름 |

세 현상은 **서로 독립적으로 발생**하며 한 응답에 동시에 나타날 수도 있다.

## 어떻게 다루나 (1강 실무 메시지)

완전히 없앨 수 없다. **확률을 끌어올리는** 게 우리가 할 일:

- **여러 번 돌려보기 (multi-sampling)**: 한 번 답에 의존하지 말고 2~3번 돌려서 일치하는 답만 채택
- **temperature 낮추기**: 사실 검증·코드 생성처럼 정확성이 중요하면 0~0.3, 창의적 작문이면 0.7 이상
- **검증 루프 ([하니스](../techniques/하니스-엔지니어링.md))**: 코드면 실행해보고, 사실이면 검색해서 교차 확인
- **자세한 지침으로 탐색 공간 좁히기 ([컨텍스트 엔지니어링](../techniques/컨텍스트-엔지니어링.md))**: 같은 요청이라도 출력 형식·제약을 명시하면 흔들림이 줄어든다
- **시연 시 주의**: 강의 라이브 데모에서 같은 프롬프트가 매번 똑같이 나올 거라고 가정하지 말 것. 결과가 달라져도 당황하지 않게 미리 청중에게 *"AI는 매번 답이 조금씩 다릅니다"*를 깔아두면 자연스러움.

## 1강 강의 연결 포인트

- *"LLM의 두 가지 대표 문제"* 중 두 번째 — 첫 번째 [환각](환각.md)과 짝.
- 1강 메시지 *"좋은 결과는 우연이 아니라 설계"* 의 **가장 정직한 근거** — 한 번 잘 나온 답은 *우연*일 수도 있다. 그래서 우리는 **확률을 올리는 설계**를 한다.
- 라이브 데모에서 같은 프롬프트를 일부러 두 번 보내 답이 다른 걸 보여주면 청중이 "AI가 결정론적 도구가 아니다"를 한 번에 이해.
