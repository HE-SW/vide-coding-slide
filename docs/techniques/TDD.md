# TDD (Test-Driven Development)

> *코드를 짜기 전에 테스트부터 짠다*. 1990년대 말 Kent Beck이 익스트림 프로그래밍(XP)에서 정착시킨 개발 규율. AI 시대에 와서는 — *AI가 자기 코드를 자기가 검증하게 만드는 가장 단순한 게이트* 가 됐다. `CLAUDE.md` 템플릿에 *CRITICAL* 항목으로 박아두는 그 규칙이다.

## 1. 왜 "TDD"라고 부르나

- **이름의 유래**: *Test-Driven Development*. 한국어로 *테스트 주도 개발*. *테스트가 코드를 끌고 간다(drive)* 는 뜻. 코드를 먼저 짜고 나중에 테스트 붙이는 게 아니라, **테스트 → 코드 → 테스트** 순서가 뒤집힌다.
- **인물**: **Kent Beck** — 미국 소프트웨어 엔지니어. 1997년경 *익스트림 프로그래밍(XP, Extreme Programming)* 이라는 방법론을 정리하면서 *test-first* 라는 실천 항목을 핵심에 박아 넣었다. 2002~2003년 책 *"Test-Driven Development: By Example"* (Addison-Wesley) 출간으로 산업계 표준 용어로 정착.
- **본인의 표현**: Beck은 자기가 *발명*했다고 하지 않고 *"옛날 어느 책에서 본 걸 다시 끌어왔다(rediscovered)"* 고 표현했다. 1960~70년대 메인프레임 시절부터 일부 팀이 쓰던 방식을 — *프로그래밍 문화 전체에 다시 보급한* 사람으로 보는 게 정확하다.

## 2. 그전에는 어떤 문제가 있었나

테스트를 *코드 다 짜고 나서* 붙이던 시절의 한계:

- **테스트가 자주 빠진다** — 일정에 쫓기면 *"일단 동작하니까 나중에 짜자"* 가 되고, 그 *"나중"* 이 안 온다.
- **회귀 버그 폭발** — 새 기능을 추가하다 *예전 기능이 망가지는 줄 모르고* 배포. 사용자 신고가 들어와야 발견.
- **설계가 후행적** — 코드가 다 짜진 다음 테스트를 붙이려고 하면 *"이거 어떻게 테스트하지?"* 가 자주 막힌다. 테스트하기 어려운 구조로 짜놓은 거다.
- **무엇이 *완료*인지 모호** — *"버그 다 잡았어요?"* *"아마요?"*. 끝났다는 객관적 신호가 없다.

**일상 비유**: *식당 오픈 전에 메뉴판부터 만든다*. 메뉴판이 먼저 있어야 — *"이걸 만들 수 있는가? 재료가 있는가? 맛이 일정한가?"* 가 *주방을 짓기 전에* 검증된다. 메뉴판 없이 일단 주방부터 차렸다가 *"손님 받고 보니 안 되는 메뉴가 절반"* 이 되는 게 *후행 테스트*. 메뉴판 = 테스트, 주방 = 구현 코드.

## 3. 어떻게 해결했나 — Red / Green / Refactor 사이클

TDD의 핵심은 *세 단계 짧은 루프* 다.

```text
🔴  Red       1) 실패하는 테스트를 먼저 짠다
              → 아직 코드가 없으니 *반드시* 실패해야 함
              ↓
🟢  Green     2) 테스트가 통과할 *최소한의 코드* 만 짠다
              → 깔끔하지 않아도 OK, 일단 통과
              ↓
🔵  Refactor  3) 통과 상태를 유지하며 코드를 다듬는다
              → 중복 제거, 이름 정리, 구조 개선
              ↓
              (다음 기능으로 다시 🔴 부터)
```

이 사이클이 가져다주는 효과:

- **테스트가 *사양 그 자체*** — *"이 함수는 이런 입력에 이런 출력을 내야 한다"* 가 코드로 박힌다. 사람 말로 적힌 요구사항보다 정확.
- **회귀 즉시 감지** — 다음 기능을 추가하다 옛 테스트가 빨개지면 *방금 그 변경이 망쳤다* 는 게 즉시 보인다.
- ***완료* 가 객관적** — *"테스트가 다 초록이면 끝"*. 사람의 직감 의존 ↓.
- **테스트 가능한 설계로 자연스럽게 흐름** — 테스트를 먼저 쓰려면 — *작고 독립적인 단위로 코드를 쪼갤 수밖에* 없다.

## 4. Before / After — AI 코딩 시대 버전

### BEFORE — 테스트 후행

```text
USER:  "할인 쿠폰 적용 함수 짜줘"
AI:    [코드 한 덩이]
USER:  [실행 → 동작은 함]
USER:  "어, 근데 0원 쿠폰 들어오면 어떻게 돼?"
AI:    [예외 처리 추가 → 어딘가 망가짐]
USER:  "음 30% 쿠폰은 잘 되는데 50%는 왜 안 되지?"
AI:    [디버깅 시도 → 다시 망가짐]
        ... 이하 무한 반복
```

→ 결국 *어디까지 동작하는지* 사람이 매번 머리로 추적.

### AFTER — TDD 적용

```text
USER:  "할인 쿠폰 적용 함수 짜줘. 다음 케이스 테스트로 명시:
        - 0원 쿠폰 → 원가 그대로
        - 30% 쿠폰 → 70% 가격
        - 100% 초과 쿠폰 → 에러
        - 쿠폰 만료일 지남 → 에러"

AI:    1) test_apply_coupon.py 작성 (4개 케이스, 다 빨강)
       2) apply_coupon.py 작성 → 4개 다 초록
       3) 리팩터 → 여전히 4개 초록
       4) "완료. 4/4 통과"
```

→ *완료 여부가 명확*. 다음에 새 케이스가 발견되면 — 5번째 테스트만 추가하면 된다.

## 5. 라이브 시연 가능한 예시

강의 중 즉석 시연 가능한 30초짜리 데모:

```bash
> 피보나치 함수 짜주는데, 먼저 테스트부터 짜줘
  - fib(0) = 0
  - fib(1) = 1
  - fib(10) = 55
  - fib(-1) → ValueError

[Claude Code 동작]
1. test_fib.py 작성 → pytest → 4개 다 빨강 (코드가 없으니까)
2. fib.py 작성 → pytest → 3개 초록, 1개 빨강 (음수 처리 누락)
3. fib.py 수정 → pytest → 4개 초록
4. "완료. 4/4 통과, fib(10)=55 확인"
```

청중에게: *"코드를 보지 않아도 — *4개 통과* 라는 한 줄로 *완료* 를 신뢰할 수 있다."*

## 6. AI 시대의 TDD — 왜 *지금* 더 중요한가

TDD는 1990년대부터 있었지만, AI 코딩 시대에 와서 *오히려 더* 중요해졌다. 세 가지 이유:

1. **AI는 *그럴듯하게 잘못 짠다*** — 환각([Hallucination](../phenomena/환각.md))이 코드로도 일어난다. 테스트 없이는 *맞아 보이는데 틀린 코드*를 알아채기 어렵다.
2. **AI에게 *완료 기준*을 코드로 줘야 한다** — Karpathy의 4번째 원칙(*Goal-Driven Execution*)의 가장 실용적 구현. *"버그 고쳐줘"* 대신 *"이 테스트를 통과시켜라"* 가 자동 루프를 가능하게 만든다.
3. **헤드리스 모드의 *무기억* 사이클** — `claude -p`는 매 호출 새 Claude. *완료 신호*가 코드(테스트 결과)로 남아 있어야 다음 호출이 *멈출 줄 안다*. 테스트가 없으면 무한 루프.

## 7. 한계와 주의

- **테스트 짜기에도 시간이 든다** — 짧은 스크립트엔 과함. 의료·금융 같은 *틀리면 큰일* 영역에서 효과가 가장 크다.
- **테스트가 *틀리게* 짜질 수 있다** — 잘못된 테스트는 잘못된 코드를 *통과시킨다*. AI에게 시킬 때는 *테스트 먼저 사람이 검토* 하는 단계가 필요.
- **UI·시각 영역엔 직접 적용 어려움** — 화면 디자인은 테스트로 다 안 잡힌다. 비주얼 회귀 테스트(스크린샷 비교) 같은 보조 도구 필요.
- **테스트 커버리지 함정** — *100% 커버리지가 곧 버그 0* 은 아니다. *어떤 케이스를 테스트했는가* 가 더 중요.

## 강의 연결 포인트

- **`CLAUDE.md` 템플릿의 `CRITICAL:` 항목** — *"새 기능 구현 시 반드시 테스트를 먼저 작성하고, 테스트가 통과하는 구현을 작성할 것"* 한 줄. AI에게 *"매 작업마다 TDD"* 를 영구 박는 가장 단순한 방법.
- **`/harness` Phase 게이트** — 각 Phase 통과 기준이 *테스트 결과*. *"54개 테스트 통과"* 한 줄로 자동 실행 결과를 *신뢰*할 수 있는 이유.
- **의료 영역 적용 (예: DICOM Viewer)** — *틀리면 진단에 영향* 가는 영역. TDD가 가장 강하게 작동해야 하는 도메인.
