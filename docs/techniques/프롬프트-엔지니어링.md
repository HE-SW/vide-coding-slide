# 프롬프트 엔지니어링 (Prompt Engineering)

> AI에게 **무엇을 어떻게 요청할 것인가**를 설계하는 기술. "질문을 잘 던지는 능력".

## 1. 왜 "프롬프트 엔지니어링"이라고 부르나

- **Prompt(프롬프트)**: 영어로 "재촉하다·자극하다", 컴퓨팅에서는 사용자가 입력하는 명령줄(`>`)을 가리키던 말. AI 시대에는 **모델에게 주는 입력 텍스트** 전체를 의미하게 됐다.
- **Engineering(엔지니어링)**: 그냥 "물어본다"가 아니라, **반복 가능한 방식으로 결과 품질을 끌어올리는 설계 행위**라는 의미를 강조하기 위해 붙은 단어.
- **언제 등장했나**: 2020년 GPT-3 출시 직후. GPT-3는 "어시스턴트"로 만들어진 모델이 아니었기 때문에, 사용자가 **입력을 어떻게 구성하느냐**에 따라 결과 품질이 극단적으로 달라졌다. 이걸 "트릭처럼 다루는 기술"이라 인식하면서 "prompt engineering"이라는 표현이 자리 잡았다.
- **2022년 ChatGPT 출시 이후** 일반 사용자에게도 알려졌고, "Prompt Engineer"라는 직무까지 등장했다.

## 2. 그전에는 어떤 문제가 있었나

- **GPT-2 이전 (2019년 이전)**: 모델이 충분히 똑똑하지 않아서, 입력을 어떻게 줘도 결과 차이가 크지 않았다. 입력 설계가 의미가 없었다.
- **GPT-3 직후 (2020년)**: 모델은 똑똑해졌지만 **사용 설명서가 없었다**. 같은 질문을 다르게 표현하면 답변 품질이 천차만별이었다. 사람들은 우연히 좋은 답을 받으면 그 표현을 외워서 재사용하는 식으로 대응.
- **결과**: AI를 잘 쓰는 사람과 못 쓰는 사람의 차이가 **운**에 가까웠다. 재현이 어렵고 조직 차원에서 노하우가 쌓이지 않았다.

**일상 비유**: 외국어를 잘 못하는 상대에게 길을 묻는 상황. "Where is the station?"이라고 또박또박 물으면 답을 듣지만, 우물우물 빠르게 말하면 못 알아듣는다. 같은 질문이라도 **표현 방식**이 답을 좌우하던 시기.

## 3. 어떻게 해결했나 — 5가지 패턴 군

사용자들이 시행착오 끝에 발견한 핵심 기법들이 **명시적인 방법론**으로 정리됐다. 큰 가지 5개 (각 별도 문서):

| 패턴 군 | 핵심 질문 | 별도 문서 |
|---|---|---|
| **Zero / One / Few-shot** | *예시를 몇 개 줄까* | [zero-few-shot.md](prompt/zero-few-shot.md) |
| **Chain-of-Thought · ToT · GoT** | *단계별로 생각하게 할까* | [chain-of-thought.md](prompt/chain-of-thought.md) |
| **Role prompting** | *어떤 역할로 답하게 할까* | [role-prompting.md](prompt/role-prompting.md) |
| **Self-consistency · Multi-sampling** | *여러 번 돌려 다수결로 검증할까* | [self-consistency.md](prompt/self-consistency.md) |
| **Meta-prompting · Chaining · ReAct** | *프롬프트를 흐름으로 연결할까* | [meta-prompting.md](prompt/meta-prompting.md) |

OpenAI·Anthropic·Google이 공식 가이드를 내놓으며 표준화. **재현 가능한 기술**로 자리 잡았다.

**강의 메시지 연결**: *"좋은 결과는 우연이 아니라 설계"*. 프롬프트 엔지니어링은 그 설계의 *가장 안쪽 핵*.

## Before / After — 같은 모델, 다른 결과

### 1) Zero-shot — 지시문 다듬기

```text
[BEFORE — 모호한 한 줄]
USER: "이 코드 좀 봐줘"
AI:   "어떤 부분을 봐드릴까요? 보안? 성능? 가독성?
       파일을 보여주세요." (질문만 돌아옴)

[AFTER — 명시적 지시]
USER: "다음 Python 코드의 *보안 취약점*만 찾아서
       심각도(상/중/하)와 함께 표로 정리해줘.
       코드는 ```로 감싸있음."
AI:   |취약점 | 위치 | 심각도|
      |SQL Injection | line 14 | 상 |
      |하드코딩된 키 | line 3 | 중 |
       (즉시 결과)
```

### 2) Few-shot — 예시로 패턴 잡기

```text
[BEFORE]
USER: "이 문장의 감성을 분류해줘. '비 오는 날인데 따뜻하다'"
AI:   "긍정적인 감성으로 보입니다." (이유는 모호)

[AFTER — 예시 3개로 양식 강제]
USER: "다음 형식으로 분류해줘.

  '오늘 하늘이 맑다'   → 긍정 / 단어:맑다
  '시험 또 망쳤어'     → 부정 / 단어:망쳤어
  '음식이 평범해'      → 중립 / 단어:평범

  이제 분류: '비 오는 날인데 따뜻하다'"
AI:  "긍정 / 단어:따뜻하다"   (양식 정확히 따름)
```

### 3) Chain-of-Thought — "단계별로"

```text
[BEFORE]
USER: "사과 5개를 3명에게 나눠주면 한 명당 몇 개?"
AI:   "1.67개" (소수점 답으로 직관 왜곡)

[AFTER — "단계별로 생각해서"]
USER: "사과 5개를 3명에게 나눠주면 한 명당 몇 개?
       단계별로 생각해서 답해줘."
AI:   "1) 5 ÷ 3 = 1 나머지 2
       2) 한 사람당 1개 받고, 2개가 남음
       3) 답: 한 명당 1개, 2개는 나머지"
```
복잡한 추론 문제일수록 **정답률 30~50%까지 상승**한다는 연구 결과가 다수.

### 4) Role prompting — 역할 부여

```text
[BEFORE]
USER: "이 계약서 봐줘. 문제 있어?"
AI:   "어떤 종류의 문제를 보면 될까요?" (범위 모호)

[AFTER — 역할 부여]
USER: "너는 시니어 IT 노동법 변호사야. 다음 채용계약서에서
       *직원에게 불리한 조항*을 찾아서 위험 등급과 함께 알려줘."
AI:   "핵심 위험 3건:
       (상) 5조 비밀유지 — 퇴사 후 5년은 과도
       (중) 9조 야근수당 — 포괄임금제 명시 누락
       ..."
```

## 라이브 시연 가능한 예시

가장 강력한 한 장. 같은 LLM에게 *연이어 두 번* 입력해서 답이 어떻게 달라지는지 보여준다.

```bash
[강의 화면 1]
> 한국에서 가장 인구 많은 도시는?

→ Claude: 서울입니다.

[강의 화면 2 — Chain-of-Thought 추가]
> 한국에서 가장 인구 많은 도시 5개를 차례로 나열해줘. 단계별로 생각해.

→ Claude:
  1) 한국 시·도 인구를 비교한다.
  2) 서울 ~960만 / 부산 ~330만 / 인천 ~300만 / 대구 ~240만 / 대전 ~145만
  3) 답: 서울, 부산, 인천, 대구, 대전
```
청중이 *"한 줄 추가만으로 답의 두께가 달라진다"*를 직관적으로 느낀다.

## 한계와 진화

프롬프트 엔지니어링은 **단발성 질문 품질**에는 강하지만, 실무 LLM 앱(자료 검색·도구 사용·반복 검증)에서는 한계가 있었다. 그래서 2025년부터 **컨텍스트 엔지니어링**이라는 더 큰 개념으로 진화 중 (2026년 현재 표준화 단계). 두 단어는 *대체 관계가 아니라 포함 관계* — 컨텍스트 엔지니어링 안에 프롬프트 엔지니어링이 *가장 작은 코어*로 남아 있다.
