# 하니스 엔지니어링 (Harness Engineering)

> AI 모델 **주변에 두르는 작업 환경 전체**를 설계하는 기술. 모델이 혼자 못 하는 일을, 도구·반복·검증 루프로 가능하게 만든다.

## 1. 왜 "하니스"라고 부르나

- **Harness(하니스)**: 영어로 "마구(馬具)" — 말의 힘을 마차나 쟁기에 연결해 **유용한 일에 쓰도록** 매다는 도구. 등산용 안전 벨트나 낙하산 멜빵도 하니스라고 부른다.
- **소프트웨어 분야의 기존 의미**: "test harness"는 오래전부터 **테스트 환경 전체**(설정 + 실행기 + 검증기)를 가리키던 단어. 코드를 둘러싸고 동작시키는 틀이라는 뜻.
- **AI 에이전트 분야로 확장**: Anthropic이 2024~2025년에 걸쳐 "agent harness"라는 표현을 공식 사용하며 정착. 모델 자체는 그대로 둔 채, **그 주변에 두르는 모든 코드·설정·실행 로직**(파일 도구, 권한 시스템, 컨텍스트 압축, 서브에이전트, 평가 루프 등)을 통칭한다.
- **Anthropic의 정의**: *"A harness is every piece of code, configuration, and execution logic that isn't the model itself."* — 모델이 아닌 **나머지 전부**가 하니스다.

## 2. 그전에는 어떤 문제가 있었나

- LLM 단독 호출은 **한 번 묻고 한 번 답하는 구조**였다. 한계:
  - 코드를 짜줘도 **실행해서 검증**할 방법이 없다
  - 잘못된 답을 해도 **자기 답을 다시 확인**할 수단이 없다
  - 컨텍스트 창이 끝나면 **이전 작업을 잊는다**
  - 작업이 1시간만 넘어가도 **흐름이 끊긴다**
- 결과: 시연(demo)에서는 그럴듯해 보여도, 실제 업무에 투입하면 1~2단계 만에 무너졌다.

**일상 비유**: 천재 인턴이 한 명 있다고 하자. 똑똑하지만 책상도, 컴퓨터도, 회의록도, 검토자도 없이 혼자 일한다면? 결과를 신뢰하기 어렵다. 하니스 엔지니어링은 그 인턴 주위에 **책상·도구·체크리스트·동료**를 셋업하는 일이다.

## 3. 어떻게 해결했나

- Anthropic의 **Claude Code** 같은 에이전트 시스템이 보여준 구성 요소:
  - **도구(tools)**: bash, read/write/edit, grep, browser 등을 모델이 호출할 수 있게 함
  - **컨텍스트 압축**: 긴 작업 중 이전 대화를 자동 요약
  - **서브에이전트(subagents)**: 큰 작업을 작은 에이전트들에게 위임
  - **태스크 시스템**: 작업을 의존 그래프로 관리
  - **평가 루프 (generator-evaluator)**: 한 에이전트가 만들고, 다른 에이전트가 검증 — GAN(생성적 적대 신경망)에서 영감
  - **권한 거버넌스**: 어떤 도구를 언제 쓸 수 있는지 통제
  - **워크트리(worktree) 격리**: 병렬 작업이 서로를 망치지 않게 분리
- Anthropic의 핵심 명제: *"Every component in a harness encodes an assumption about what the model can't do on its own."* — 하니스의 모든 요소는 **모델이 혼자서는 못 하는 일에 대한 가정**을 코드화한 것이다.
- **1강 연결 포인트**: 결과 품질을 높이는 3가지 축의 마지막 — **"어떻게 반복적으로 검증할 것인가"**.
