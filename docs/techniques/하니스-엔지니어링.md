# 하니스 엔지니어링 (Harness Engineering)

> AI 모델 **주변에 두르는 작업 환경 전체**를 설계하는 기술. 모델이 혼자 못 하는 일을, 도구·반복·검증 루프로 가능하게 만든다.

## 1. 왜 "하니스"라고 부르나

- **Harness(하니스)**: 영어로 "마구(馬具)" — 말의 힘을 마차나 쟁기에 연결해 **유용한 일에 쓰도록** 매다는 도구. 등산용 안전 벨트나 낙하산 멜빵도 하니스라고 부른다.
- **소프트웨어 분야의 기존 의미**: "test harness"는 오래전부터 **테스트 환경 전체**(설정 + 실행기 + 검증기)를 가리키던 단어. 코드를 둘러싸고 동작시키는 틀이라는 뜻.
- **AI 에이전트 분야로 확장**: Anthropic이 2024~2025년에 걸쳐 "agent harness"라는 표현을 공식 사용하며 정착. 모델 자체는 그대로 둔 채, **그 주변에 두르는 모든 코드·설정·실행 로직**(파일 도구, 권한 시스템, 컨텍스트 압축, 서브에이전트, 평가 루프 등)을 통칭한다.
- **Anthropic의 정의**: *"A harness is every piece of code, configuration, and execution logic that isn't the model itself."* — 모델이 아닌 **나머지 전부**가 하니스다.

## 2. 그전에는 어떤 문제가 있었나

- LLM 단독 호출은 **한 번 묻고 한 번 답하는 구조**였다. 한계:
  - 코드를 짜줘도 **실행해서 검증**할 방법이 없다
  - 잘못된 답을 해도 **자기 답을 다시 확인**할 수단이 없다
  - 컨텍스트 창이 끝나면 **이전 작업을 잊는다**
  - 작업이 1시간만 넘어가도 **흐름이 끊긴다**
- 결과: 시연(demo)에서는 그럴듯해 보여도, 실제 업무에 투입하면 1~2단계 만에 무너졌다.

**일상 비유**: 천재 인턴이 한 명 있다고 하자. 똑똑하지만 책상도, 컴퓨터도, 회의록도, 검토자도 없이 혼자 일한다면? 결과를 신뢰하기 어렵다. 하니스 엔지니어링은 그 인턴 주위에 **책상·도구·체크리스트·동료**를 셋업하는 일이다.

## 3. 어떻게 해결했나 — 5가지 구성 요소

하니스를 *5가지 부품*으로 분해할 수 있다. 각 부품은 별도 문서로 deep dive:

| 구성 요소 | 한 줄 정의 | 별도 문서 |
|---|---|---|
| **Tools (도구)** | 모델이 *세계와 닿는* 인터페이스 (bash·read·grep·...) | [tools.md](harness/tools.md) |
| **Subagents & Delegation (분업)** | 큰 일을 작은 일꾼들에게 위임. 컨텍스트 보호 + 비용 라우팅 | [subagents-and-delegation.md](harness/subagents-and-delegation.md) |
| **Evaluation Loop (검증 루프)** | 별도 검증기가 *모델 출력을 채점*하고 재시도 (Anthropic 3-agent) | [evaluation-loop.md](harness/evaluation-loop.md) |
| **Control & State (제어·상태)** | Phase 분할 + docs 영속화 + 워크트리 격리 | [control-and-state.md](harness/control-and-state.md) |
| **Permissions & Hooks (안전장치)** | 1+2층 4단 방어선. 모델 *바깥*의 결정론적 가드 | [permissions-and-hooks.md](harness/permissions-and-hooks.md) |

```text
┌────────────────────────────────────────────────────┐
│           하니스 = 모델 *주변*의 모든 것              │
│                                                    │
│   Permissions & Hooks  ←──  결정론적 가드           │
│   ┌──────────────────────────────────────────┐    │
│   │  Control & State                          │    │
│   │  ┌─────────────────────────────────────┐ │    │
│   │  │  Subagents & Delegation             │ │    │
│   │  │  ┌────────────────────────────────┐ │ │    │
│   │  │  │  Evaluation Loop               │ │ │    │
│   │  │  │  ┌─────────────────────────┐  │ │ │    │
│   │  │  │  │  Tools (모델의 손)       │  │ │ │    │
│   │  │  │  │  ┌──────────┐           │  │ │ │    │
│   │  │  │  │  │  Model   │           │  │ │ │    │
│   │  │  │  │  └──────────┘           │  │ │ │    │
│   │  │  │  └─────────────────────────┘  │ │ │    │
│   │  │  └────────────────────────────────┘ │ │    │
│   │  └─────────────────────────────────────┘ │    │
│   └──────────────────────────────────────────┘    │
└────────────────────────────────────────────────────┘
```

Anthropic의 핵심 명제: *"Every component in a harness encodes an assumption about what the model can't do on its own."* — 하니스의 모든 요소는 **모델이 혼자서는 못 하는 일에 대한 가정**을 코드화한 것이다.

**강의 연결 포인트**: 결과 품질을 높이는 3가지 축의 마지막 — **"어떻게 반복적으로 검증할 것인가"**.

## Before / After — 단발 호출 vs 하니스 적용

```text
[BEFORE — 단발 LLM 호출]
USER: "사용자 인증 기능 만들어줘"
AI:    [코드 한 덩이 출력]
       → 컴파일 에러 / 보안 결함 / 테스트 없음
       → 사람이 처음부터 검증·재요청 반복

[AFTER — 하니스 적용 (Claude Code + /harness)]
USER: "/harness"   ← 한 줄
AI:    1) docs/ 자동 로드
       2) Phase 5개로 작업 쪼갬
       3) Phase 1 코드 → 자동 테스트 → 통과 → 자동 커밋
       4) Phase 2 → … → Phase 5
       5) 17분 후 "✓ 5/5 완료, 54개 테스트 통과"
```
같은 모델·같은 작업이 **하니스 유무**로 결과 신뢰도 차이가 극단적.

## 라이브 시연 가능한 예시

### 시연 — 검증 루프의 위력
강의 중에 작은 함수(예: 피보나치)를 두 가지 방식으로 시킨다.

```bash
[방법 1 — 단발 호출 / 검증 없음]
> Python 피보나치 짜줘
→ AI: def fib(n): ...   (실행은 사람이 직접 해야)

[방법 2 — Claude Code (하니스)]
> Python 피보나치 짜고 테스트도 같이
→ AI: 1. fib.py 작성
      2. test_fib.py 작성
      3. pytest 실행 → 1개 실패
      4. fib.py 수정
      5. pytest 재실행 → 통과
      "완료. fib(10) = 55"
```
*"AI가 자기가 짠 코드를 자기가 돌리고 자기가 고친다"* — 청중이 처음 보는 광경.

## 범용 하니스 vs 커스텀 하니스 — 한 층을 *어떻게* 올리나

같은 *2층 하니스* 라도 — *남이 미리 만들어둔 풀세팅* 을 가져오느냐, *내 프로젝트에 맞게 직접 짜느냐* 두 갈래가 있다. **둘 중 어느 쪽도 정답이 아니다 — 상황별 선택**.

| 축 | 범용 하니스 (오픈소스 풀세팅) | 커스텀 하니스 (직접 구성) |
|---|---|---|
| **대표 사례** | [oh-my-claudecode](../tools/oh-my-claudecode.md) — 32개 에이전트 + 5단계 파이프라인 풀패키지 | 이 강의 레포 — `CLAUDE.md` + `docs/` + `skills/` + `hooks/` 4종만 손대기 |
| **장점** | 한 방에 *프로 수준의 하니스* 도착, 멀티 모델 협업·자동 라우팅 같은 고급 기능까지 | 가볍게 시작·자유롭게 수정·내 프로젝트 도메인 규칙 반영 가능 |
| **단점** | 초기 학습 곡선·내 프로젝트 *고유 사정* 모름·수정이 어렵다 | 풀세팅까지 가려면 시간 投入 필요 |
| **추천 대상** | *대규모 다인 프로젝트, 일반화 가능한 작업이 많은 팀* | *1인~소규모, 도메인 특수성 강한 프로젝트, 학습 단계 비개발자* |

**중요한 건 한 층을 *올린다는 것*. 방법은 자유.** 풀세팅을 통째로 끌어 쓰든, 4종 markdown만 손보든 — 둘 다 2층 위로 올라간 거다. 이번 강의는 *비개발자가 직접 짜기 쉬운* 커스텀 쪽에 무게를 둔다.

비유: *"집을 산다 vs 짓는다"* 의 선택 — 둘 다 *집에 들어가 산다* 는 결과는 같다.

## 실제 사례 (2025-2026)

- **Claude Code의 `/harness` 패턴** (이번 강의 레포 자체) — Phase 분할 + headless 자동 실행 + 자동 커밋
- **oh-my-claudecode** (한국 개발자 Yeachan-Heo, 2025-2026) — *team-plan → team-prd → team-exec → team-verify → team-fix* 5단계 자동 파이프라인을 한 솔루션에 묶은 사례
- **GitHub Actions + `claude -p`** — CI에 헤드리스 Claude를 끼워 자동 PR 리뷰
- **SWE-bench 점수 도약**: 2024년 33% → 2026년 70%+ 모델들이 등장 — *모델 자체*보다 *하니스 설계*가 점수의 절반 이상을 차지한다는 분석
