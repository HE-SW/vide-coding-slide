# 문서 주도 개발 — PRD · ARCHITECTURE · ADR · UI_GUIDE

> 2층 하니스의 *뇌* 역할을 하는 4종 마크다운 문서. 각 문서가 *"한 가지 질문"*에 답한다 — **PRD = 뭘 만드는지**, **ARCHITECTURE = 어떻게**, **ADR = 왜**, **UI_GUIDE = 어떻게 보여야**. AI가 *프로젝트 전체*를 이해한 상태에서 답할 수 있게 만드는 4종 세트.

## 1. 왜 "문서 주도"라고 부르나

- 전통적인 *Test-Driven Development(TDD)*가 **테스트가 코드를 끌고 가는** 방식이라면, 문서 주도 개발은 **문서가 AI를 끌고 가는** 방식.
- AI 코딩 시대(2024~)에 들어 *"AI에게 사양을 어떻게 전달할 것인가"* 라는 새 문제가 생겼고, 그 답으로 정착한 패턴.
- 4종 문서 자체는 새 발명이 아니다. **PRD**(Product Requirements Document)는 1990년대부터 IT 업계 표준, **ADR**(Architectural Decision Record)은 2011년 마이클 나이가르트가 정립한 양식, **ARCHITECTURE.md**는 GitHub 오픈소스 컨벤션, **UI 가이드라인**은 Apple HIG·Material Design 같은 디자인 시스템 문화.
- AI 시대에 와서 — 이 4종을 *프로젝트 루트의 docs/ 한 폴더에 모아두는* 패턴이 *baseline 사양*으로 자리 잡았다.

## 2. 그전에는 어떤 문제가 있었나

AI에게 *복잡한 프로젝트*를 시키면 자주 무너지던 이유:

- **컨텍스트 부족**: AI가 *프로젝트 목적*을 모르는 채로 작업 → 옳지만 *"우리 프로젝트엔 안 맞는"* 결과
- **반복 설명**: 매 세션마다 *같은 배경*을 처음부터 다시 입력
- **결정 망각**: *"왜 이 라이브러리를 골랐었지?"* — 1주 전 결정의 이유가 코드에 남지 않음
- **세션 간 단절**: 헤드리스 모드(`claude -p`)는 *무기억* — Phase 1 결정이 Phase 5에 전달될 길이 없음

**일상 비유**: 새 멤버가 입사할 때 *신입 교육 자료*를 안 주는 회사. 매번 *"우리 회사 뭐 하는 곳이에요?"* 부터 새로 설명해야 한다. 4종 문서 = **하루 만에 배경을 다 익힐 수 있는 신입 패키지**.

## 3. 어떻게 해결했나 — 4종 문서

### 📘 PRD.md — *뭘* 만드는지

```markdown
# PRD: {프로젝트명}

## 목표
{한 줄 요약}

## 핵심 기능
1. {기능 1}
2. {기능 2}

## MVP 제외 사항
- {안 만들 것}
```

목표·범위·제외사항을 명시. **"뭘 만들지"** 한 가지 질문에만 답한다. AI가 새 기능을 제안할 때 *MVP 제외 사항*을 보고 *과한 작업*을 스스로 거른다.

### 🏗️ ARCHITECTURE.md — *어떻게* 만드는지

```markdown
# 아키텍처

## 디렉토리 구조
{폴더 트리}

## 패턴
{디자인 패턴}

## 데이터 흐름
{데이터 흐름도}
```

폴더 구조·기술 스택·데이터 흐름. **AI가 새 파일을 만들 때 *어디에 둘지*를 자동으로 판단**할 수 있게 한다.

### 🧭 ADR.md — *왜* 이렇게 만드는지

```markdown
### ADR-001: Next.js App Router 선택
**결정**: App Router 사용
**이유**: SSR + Server Component로 초기 로딩 최적화
**트레이드오프**: 라이브러리 호환성 일부 포기

### ADR-002: ...
```

각 결정과 그 이유, 포기한 것까지. **나중에 *"이거 왜 이렇게 됐어?"*에 답할 수 있는 기록**. 1주 전 결정이 1년 뒤에도 살아있다.

### 🎨 UI_GUIDE.md — *어떻게 보여야* 하는지 (선택)

색상 팔레트·타이포·간격 시스템·컴포넌트 규칙. **디자인 일관성**을 코드로 굳힌다. 디자이너 없이 일하는 비개발자에게 *기본값*을 깔아준다.

## 한 가지 질문, 한 가지 답

```text
PRD          →  뭘 만드는지
ARCHITECTURE →  어떻게 만드는지
ADR          →  왜 이렇게 만드는지
UI_GUIDE     →  어떻게 보여야 하는지
```

네 문서가 합쳐져서 — **AI가 *뭘·어떻게·왜·어떻게 보여*를 동시에 이해**한 상태로 답한다.

## Before/After — 4종 문서 유무

| 상황 | 4종 없음 | 4종 있음 |
|---|---|---|
| 새 세션 시작 | "이 프로젝트 뭐였더라" 매번 설명 | docs/ 자동 로드 → 즉시 작업 |
| AI가 새 라이브러리 제안 | 기존 결정과 충돌해도 모름 | ADR 보고 *충돌 회피* |
| 새 폴더에 파일 추가 | 어디에 둘지 매번 물어봐야 함 | ARCHITECTURE 보고 *자동 판단* |
| 헤드리스 5단계 자동화 | Phase 5가 Phase 1 결정을 모름 | docs로 결정이 *영속화*돼 전달 |

## 라이브 시연 가능한 예시

### 시연 — 같은 프롬프트, 다른 결과
```bash
[docs/ 없는 프로젝트]
> 사용자 인증 기능 추가해줘
[Claude]
  → JWT? OAuth? 매번 물어봄
  → 폴더 위치도 물어봄
  → 30분 후에야 작업 시작

[docs/ 갖춘 프로젝트 (PRD에 "OAuth만 사용" 명시 + ADR-003에 이유 기록)]
> 사용자 인증 기능 추가해줘
[Claude]
  → "ADR-003에 따라 OAuth로 구현합니다.
     ARCHITECTURE에 따라 src/auth/ 폴더에 새 파일 추가."
  → 즉시 시작
```

## 실제 사례

- **이번 강의 레포 자체** — `CLAUDE.md` + `docs/` 구조가 정확히 이 패턴
- **PRD 표준화 운동**: 2024년부터 *"AI 친화적 PRD"* 표준화 시도 다수 (예: GitHub의 *AI-readiness* 커뮤니티)
- **ADR의 부활**: 2025~2026년 AI 코딩 시대에 ADR 채택률이 다시 상승. 사람보다 *AI*가 더 자주 ADR을 읽는 시대.
