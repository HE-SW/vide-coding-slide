# 컨텍스트 엔지니어링 (Context Engineering)

> AI에게 **어떤 정보를 함께 줄 것인가**를 설계하는 기술. 프롬프트 엔지니어링의 확장판.

## 1. 왜 "컨텍스트 엔지니어링"이라고 부르나

- **Context(컨텍스트)**: 모델이 한 번에 볼 수 있는 **입력 전체**(시스템 지시문 + 사용자 질문 + 검색된 문서 + 도구 출력 + 이전 대화 요약 등). LLM은 이 컨텍스트 창(context window) 안에 있는 정보만 보고 답한다.
- **언제 등장했나**: 2025년 6월~7월 사이. 대표적으로:
  - **Tobi Lütke (Shopify CEO)**가 X에서 "context engineering이 더 정확한 표현"이라고 언급
  - **Andrej Karpathy**가 2025년 6월 24일 트윗에서 *"+1 for 'context engineering' over 'prompt engineering'"*이라며 공식적으로 동의 — "산업용 LLM 앱에서는 짧은 작업 지시(prompt)가 아니라 **컨텍스트 창을 정확한 정보로 채우는 섬세한 기예와 과학**이 핵심"이라고 정리
- **왜 "engineering"인가**: 우연히 좋은 문장을 발견하는 것이 아니라, **재사용 가능한 시스템**으로 정보를 조립하는 행위라서.

## 2. 그전에는 어떤 문제가 있었나

- 프롬프트 엔지니어링은 **"마법의 한 문장 찾기"**에 가까웠다. 
- 그러나 실제 프로덕션 LLM 앱(고객 지원 봇, 코딩 에이전트, 문서 분석)에서는:
  - 사용자 질문 한 줄로는 충분한 답이 나오지 않는다
  - 회사 내부 문서, 사용자 과거 기록, 외부 API 결과 등 **여러 출처의 정보**를 모델에게 함께 줘야 한다
  - 컨텍스트 창에 무엇을, 어느 순서로, 얼마만큼 채울지가 결과 품질을 좌우한다
- 이걸 "프롬프트"라고 부르기엔 범위가 너무 컸다.

**일상 비유**: 변호사에게 "이 사건 어떻게 봐?"라고 물어볼 때, 사건 개요·관련 판례·증거 자료를 함께 건네야 제대로 된 답이 나온다. 프롬프트 엔지니어링이 "질문을 잘 던지는 법"이라면, 컨텍스트 엔지니어링은 **"브리핑 자료를 잘 구성하는 법"**.

## 3. 어떻게 해결했나

- 컨텍스트 엔지니어링은 다음을 **시스템적으로 설계**한다:
  - **시스템 지시문**: 모델의 역할·톤·제약
  - **검색 (RAG)**: 데이터베이스/문서에서 관련 정보를 자동 추출해 컨텍스트에 주입
  - **도구 출력**: 코드 실행 결과, API 호출 결과를 다시 컨텍스트로 피드
  - **대화 요약**: 긴 대화를 압축해 컨텍스트 창 한계 극복
  - **순서·우선순위**: 어떤 정보를 앞에, 어떤 정보를 뒤에 배치할지
- Karpathy의 정의: *"the delicate art and science of filling the context window with just the right information for each step"* — 매 단계마다 컨텍스트 창에 정확히 맞는 정보를 채우는 섬세한 예술이자 과학.
- **1강 연결 포인트**: 결과 품질을 높이는 3가지 축 중 두 번째.

## Before / After — 같은 질문, 컨텍스트만 추가

```text
[BEFORE — 단순 질문]
USER: "우리 회사 휴가 정책 알려줘"
AI:   "일반적으로 한국 회사는 연차 15~25일 정도이며…"
       (= 회사를 모르므로 *일반론*으로 환각)

[AFTER — 회사 휴가 규정 PDF + 직원 상태를 컨텍스트에 주입]
USER: [규정 PDF + "사번 12345, 입사 3년차, 미사용 잔여 12일"]
      "내 휴가 정책 알려줘"
AI:   "당신의 입사 3년차 기준 연차는 16일.
       잔여 12일 소진 기한은 12-31. 1주 이상 연속 사용 시
       부서장 사전 결재 필요. (출처: 휴가 규정 §3.2)"
```
같은 모델이 **컨텍스트 추가 한 번**으로 *환각 답변* → *근거 있는 답변* 으로 전환.

## 라이브 시연 가능한 예시

```bash
[강의 화면 — 빈 폴더]
> 이 프로젝트 어떤 톤으로 README 써줘?
→ Claude: "어떤 프로젝트인가요? 정보가 부족합니다."

[CLAUDE.md 한 장 추가 후]
> 이 프로젝트 어떤 톤으로 README 써줘?
→ Claude: "프로젝트 헌법(CLAUDE.md)에 따라 *비개발자
   친화 톤·한국어·예시 포함* 으로 작성하겠습니다."
```
**파일 한 장**으로 컨텍스트 엔지니어링을 작동시키는 *가장 작은* 사례. 이게 2강 #8 의 핵심 메시지.

## 실제 사례

- **GitHub Copilot Chat 컨텍스트 노출**: 2024년 Workspace 출시. 사용자가 *어떤 파일을 컨텍스트에 추가할지* 직접 선택하는 UI 표준화.
- **Claude Code의 `/compact`**: 긴 세션의 대화를 자동 요약해 컨텍스트 창 절약. 대화 요약 컨텍스트 엔지니어링의 표준 구현.
- **2강의 `docs/`**: PRD/ARCHITECTURE/ADR/UI_GUIDE 4종 마크다운으로 컨텍스트를 *영속화* — 매 세션마다 자동 로드.
